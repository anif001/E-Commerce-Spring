1 What is an E-Commerce Backend?

The backend is the server-side system that powers the entire shopping experience — the engine under the hood.

Frontend (React/Angular/Android) ↓ HTTP/JSON Backend (Spring Boot REST APIs) ↓ SQL / Redis Database (PostgreSQL / Redis)

What the Backend Handles:

  • Who you are → Registration, Login, Roles (Customer / Seller / Admin)
  • What you see → Products, Categories, Search, Recommendations
  • What you do → Add to Cart, Place Order, Pay, Track Status
  • Who manages → Admins manage users, Sellers manage inventory

Real-World Example (Amazon Search for "iPhone"):

You type "iPhone" in search bar
  → Browser sends: GET /api/products?search=iphone
  → Backend checks Redis cache first
  → Cache MISS → queries PostgreSQL
  → Runs full-text search (tsvector) + pagination + sorting
  → Returns JSON response with product list
  → Browser renders the results
Everything happens in under 500ms — that's the backend's job.
💡 Interview Tip: When asked "What is backend?", talk about responsibilities — authentication, business logic, data persistence, caching, security, API delivery.

2 How Amazon/Flipkart Backend Works (High Level)

A classic interview question: "Design Amazon's Backend."

Client (Browser/Mobile App) ↓ CDN (CloudFront / Akamai) → Static assets (images, JS, CSS) ↓ Load Balancer (AWS ELB / Nginx) → Distributes traffic ↓ API Gateway (Auth, Rate Limiting, Routing) ↓ Microservices Cluster: ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ │ User Svc │ │ Product Svc │ │ Order Svc │ └──────────────┘ └──────────────┘ └──────────────┘ ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ │ Payment Svc │ │ Inventory Svc│ │ Notification │ └──────────────┘ └──────────────┘ └──────────────┘ ↓ ↓ ↓ ┌──────────────────────────────────────────────────────┐ │ Message Queue (RabbitMQ / Kafka) │ └──────────────────────────────────────────────────────┘ ↓ ↓ ↓ ┌──────────────────────────────────────────────────────┐ │ Database Layer │ │ PostgreSQL (Primary) + Redis (Cache) + Elasticsearch │ └──────────────────────────────────────────────────────┘

Request Flow for "Place Order":

  1. Client sends POST /api/orders with JWT token
  2. API Gateway validates the JWT token
  3. Order Service receives the validated request
  4. Calls Inventory Service → check stock availability
  5. Calls Payment Service → process payment via Razorpay
  6. Calls Notification Service → send confirmation email (async via Kafka)
  7. Returns Order Confirmation to client
🔑 Key Insight: Each service owns its own data. No service directly accesses another service's database.

6 Request Flow — Trace "Place Order"

Step 1: CLIENT → POST /api/orders { Authorization: Bearer JWT }
Step 2: JwtAuthenticationFilter → extracts + validates JWT signature
Step 3: Sets SecurityContextHolder authentication
Step 4: DispatcherServlet → OrderController.placeOrder()
Step 5: @PreAuthorize("hasRole('CUSTOMER')") → authorization check
Step 6: OrderService.placeOrder()
   → Fetch cart → Validate stock → Calculate total
   → Deduct inventory (PESSIMISTIC_WRITE lock)
   → Create Order (PENDING) → Clear cart
   → Call Razorpay → EmailService (async)
Step 7: @Transactional → ALL or NOTHING
Step 8: Response → 201 Created { orderId, amount, status }
🔑 Interview Answer: "Request flows through Security Filter → Controller → Service → Repository → Database. Security and business logic are completely separated."

12 Monolith vs Microservices

Monolith:

[Frontend] ──→ [Single JAR with ALL code: Auth + Products + Orders + Payments]
                    ↓
            ┌───────┼───────┐
            ↓       ↓       ↓
         PostgreSQL Redis   Email

Pros: Simple, fast dev, no network overhead, easy testing.
Cons: Coupled, can't scale parts independently, long builds.

Microservices:

[Frontend] ──→ [API Gateway]
                    ↓
        ┌───────────┼───────────┐
        ↓           ↓           ↓
   [User Svc]  [Product Svc] [Order Svc]
   [User DB]   [Product DB]  [Order DB]

Pros: Independent scaling, teams, deployment, tech flexibility.
Cons: Network latency, distributed transactions, debugging complexity.

Evolution: Startup → Monolith → Modular Monolith → Microservices (Amazon, Netflix).

13 Why We Start with Monolith

  • Distributed transactions need Saga pattern — complex
  • Network overhead: 10 services × 5ms = 50ms per request
  • Need distributed tracing (Jaeger) for debugging
  • 2-person team doesn't need 10 repos
  • Split when 3+ teams or different scaling needs
Approach: Package by feature, no circular deps, interfaces between layers — organized as if it will become microservices.

15 Interview Questions & Answers

🔰 Beginner

Q1: Functional vs Non-Functional requirements with examples?
Functional (WHAT): "User searches products by name." Non-Functional (HOW): "Search loads in under 500ms." Without functional, useless. Without non-functional, unusable.
Q2: Why PostgreSQL over MySQL for e-commerce?
Full ACID always (MySQL needs InnoDB), native JSONB for product attributes, built-in full-text search (tsvector), superior MVCC concurrency, truly open source (MySQL owned by Oracle).
Q3: What problem does Redis solve?
Performance. Product pages get 90% of traffic. Without Redis: every page view = SQL query. With Redis: first request = SQL, next 999 = Redis (5ms vs 200ms). Reduces DB load by 99%.
Q4: Explain JWT structure (three parts).
Header: algorithm + token type. Payload: claims (user ID, roles, expiry). Signature: HMAC of header + payload with secret — verifies integrity. Format: header.payload.signature
Q5: Advantage of stateless JWT over sessions?
Horizontal scaling (any server can verify), mobile-friendly (Authorization header), microservices-ready (each service verifies independently), faster (~1ms vs DB lookup). Disadvantage: hard to revoke — mitigated by short TTL + blacklist.

📊 Intermediate

Q6: Handle 1 million concurrent users viewing same product page?
Redis caching (cache-aside, long TTL), CDN for static assets, rate limiting (100 req/s per IP), read replicas for DB, HikariCP connection pooling. Result: ~10 DB queries for 1M users.
Q7: Monolith vs Microservices — when to choose each?
Monolith: Startup/MVP, small team, simple domain. Microservices: 3+ teams, different scaling needs, polyglot persistence. Best practice: start monolith, extract when a bounded context needs independent scaling.
Q8: Explain full request flow for a product search.
DNS → Load Balancer → Spring Security (public endpoint) → DispatcherServlet → ProductController → ProductService → check Redis cache → MISS → PostgreSQL full-text search → pagination/sorting → cache result → response. ~200ms without cache, ~50ms with.
Q9: Design Order-Product DB schema.
orders(id, user_id, status, total_amount, shipping_address) + order_items(id, order_id, product_id, quantity, unit_price). Store unit_price in order_items because prices change — never JOIN for historical prices.
Q10: What is @Transactional and why critical for e-commerce?
Declares a database transaction. Place Order: deduct inventory + create order + create items + clear cart must ALL succeed or ALL roll back. Without it: partial failure = lost inventory or orphaned orders.

🚀 Advanced

Q11: How to ensure consistency across inventory, payment, and order in microservices?
Saga Pattern (Orchestration): Order Service orchestrates: 1) Reserve Inventory (Inventory Svc), 2) Process Payment (Payment Svc), 3) Send Confirmation (Notification Svc). If payment fails → compensating transaction: Release Inventory. Use idempotency keys + outbox pattern for reliability.
Q12: Flash sale — 10,000 users, 100 items. How?
Best: Redis counter + queue. Before sale: SET stock 100. On click: DECR stock atomically. If result ≥ 0 → add to processing queue. If < 0 → "Sold Out". Background worker processes queue sequentially. Ultra-fast, fair, no race conditions.
Q13: Design product search with full-text, filtering, faceting, personalization.
Layer 1: PostgreSQL tsvector + GIN index. Layer 2: Elasticsearch for fuzzy search, faceted aggregation (by brand, price range, rating), personalization (function_score query boosting based on user history).
Q14: Strangler Fig pattern for monolith to microservices migration?
1) Identify bounded context (e.g., User). 2) Build new microservice. 3) API Gateway routes feature-by-feature to new service. 4) Gradually migrate. 5) Eventually decommission monolith. Amazon took 5+ years for this migration.
Q15: Cache-aside vs Write-through — when to use each?
Cache-Aside: On read, check cache → MISS → query DB → populate cache. Default choice for product pages (read-heavy). Write-Through: On write, update DB + cache synchronously. Use for inventory (cannot serve stale stock). Write-Behind: Write to cache only, async DB persist — risk of data loss.

🔍 Q: What is a "transitive dependency"?

Breaking Down GAV (Group:Artifact:Version):

ElementExamplePurpose

📌 GAV is the unique key. When Maven downloads a dependency, it identifies it by GAV. No two artifacts can have the same GAV. This is how Maven avoids dependency conflicts.
🔍 Q: What does SNAPSHOT mean in version?
A SNAPSHOT version means "under active development." Maven will check for updates daily. A RELEASE version is frozen — Maven downloads it once and caches forever. Best practice: use SNAPSHOT during development, RELEASE for production builds.

12 Interview Questions — Maven & Build

🔰 Beginner

Q1: What is Maven and what problems does it solve?
Maven is a build automation and dependency management tool. It solves: 1) Managing JAR dependencies manually, 2) Building projects consistently across machines, 3) Standardizing project structure, 4) Automating the build lifecycle (compile → test → package → deploy).
Q2: What is pom.xml? Explain GAV coordinates.
pom.xml = Project Object Model. It describes the project — dependencies, build configuration, plugins. GAV = GroupId (reverse domain, e.g., com.ecommerce), ArtifactId (project name, e.g., ecommerce-backend), Version (0.0.1-SNAPSHOT). GAV is the unique identifier for every artifact in Maven.
Q3: What is a Spring Boot Starter?
A curated set of dependencies that work together. For example, spring-boot-starter-web includes Spring MVC, embedded Tomcat, Jackson, and validation — all version-compatible. You add one dependency, Maven downloads 15+ JARs with guaranteed compatibility.
Q4: What is the difference between mvn compile, mvn test, and mvn package?
compile compiles src/main/java. test runs compile + test compilation + executes tests (via Surefire). package runs compile + test + creates JAR/WAR in target/. Each phase includes all previous phases.
Q5: What does @SpringBootApplication do?
It combines three annotations: @Configuration (bean definitions), @EnableAutoConfiguration (auto-configure based on classpath), and @ComponentScan (scan for Spring components). Without it, Spring Boot won't auto-configure or find controllers/services.

📊 Intermediate

Q6: How does Maven resolve dependency conflicts? (Two dependencies bring different versions of the same library)
Nearest Definition Wins (shortest path in dependency tree). If A → C v1.0 and B → D → C v2.0, C v1.0 wins (depth 1 vs depth 2). You can force a version by declaring it directly in your pom.xml. Use mvn dependency:tree to visualize conflicts.
Q7: YAML vs Properties — which is better for Spring Boot configuration?
YAML is better for hierarchical config (nested structures like spring.datasource.url). YAML supports multi-document files for profiles. Properties is flatter and more verbose. Modern Spring Boot projects use YAML. Use properties only for legacy projects.
Q8: What is a "fat JAR" and how does Spring Boot create one?
A fat JAR contains the application classes + all dependencies (JARs inside the JAR). Spring Boot Maven Plugin repackages the JAR with a special loader that can launch the app with java -jar. Without it, you'd need to manage classpath manually.
Q9: Why should you NEVER use ddl-auto: update in production?
Because Hibernate auto-generates DDL from your entities. If you accidentally rename a field, Hibernate drops the old column (with all its data) and creates a new empty one. Use Flyway/Liquibase for production — they version and track schema changes safely.
Q10: What are Maven scopes? Explain compile, runtime, test, provided.
compile (default): available everywhere. runtime: not needed for compilation, needed at runtime (e.g., PostgreSQL driver). test: only for test compilation/execution (e.g., JUnit). provided: provided by the runtime environment (e.g., Tomcat if deploying WAR).

🚀 Advanced

Q11: How does Spring Boot auto-configuration work internally?
@EnableAutoConfiguration imports AutoConfigurationImportSelector, which reads META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports from all JARs on the classpath. Each configuration class has @ConditionalOnClass, @ConditionalOnMissingBean etc. If the condition is met (e.g., H2 driver on classpath), the configuration is applied. This is how adding a dependency "just works."
Q12: Design a multi-module Maven project for e-commerce (common library, API layer, service layer).
parent-pom (pom packaging)
├── common (jar) — DTOs, shared exceptions, utilities
├── api (jar) — Controllers, request/response validation
├── service (jar) — Business logic, repositories
├── persistence (jar) — JPA entities, database config
└── app (jar) — Main class, docker config
Parent POM manages dependencies centrally. Each module has its own pom.xml with parent as the parent POM. Modules are built in dependency order (Maven handles this automatically). This enforces clean separation of concerns.
Q13: What is the Maven Bill of Materials (BOM) pattern?
A BOM is a POM with <dependencyManagement> section but no dependencies. Other projects import it in <dependencyManagement> to get consistent versions. Spring Boot's spring-boot-dependencies is a BOM that manages versions for 1000+ libraries. This is how you get version consistency across all Spring projects.
Q14: How would you secure secrets (DB password, JWT secret, Razorpay key) in production?
1) Environment variables (${DB_PASSWORD}), 2) HashiCorp Vault for dynamic secrets, 3) AWS Secrets Manager / GCP Secret Manager, 4) Spring Cloud Config Server with encrypted values. Never hardcode in application.yml committed to Git. Use spring-boot-starter-vault-config for Vault integration.
Q15: Explain Maven's dependency mediation — what is "nearest definition" and how do you override it?
Nearest Definition: Maven picks the dependency closest to the root in the dependency tree. To override: 1) Declare the dependency directly in your pom.xml with your desired version (shortest path wins). 2) Use <exclusions> to exclude transitive versions. 3) Use <dependencyManagement> to centrally control versions. 1 What is JPA? What is Hibernate?

JPA (Jakarta Persistence API):

JPA is a specification (interface/contract) for Object-Relational Mapping (ORM) in Java. It defines:

  • How Java objects (entities) map to database tables
  • How to query data (JPQL, Criteria API)
  • How to manage transactions and relationships

JPA is NOT an implementation — it's just a set of interfaces (EntityManager, EntityTransaction, annotations like @Entity).

Hibernate:

Hibernate is the most popular implementation of JPA. It's the actual engine that:

  • Generates SQL from your annotations
  • Manages database connections
  • Handles caching (first-level, second-level)
  • Provides lazy loading, dirty checking, change tracking
Analogy: JPA = Interface (like USB specification) Hibernate = Implementation (like a USB cable) Spring Data JPA = Spring's wrapper on top of JPA → Provides JpaRepository interface (extends EntityManager) → Auto-implements CRUD methods (findById, save, delete) → Generates queries from method names (findByName, findByEmail)

🔍 Q: JPA vs Hibernate — what's the difference?
JPA is a specification (interfaces + annotations). Hibernate is an implementation of that specification. You code against JPA interfaces so you can switch implementations (e.g., from Hibernate to EclipseLink) without changing code. In practice, everyone uses Hibernate under the hood via Spring Boot's spring-boot-starter-data-jpa.

2 Entity Mapping — @Entity, @Table, @Id, @GeneratedValue

Every JPA entity maps to a database table. Let's understand each annotation.

@Entity

What it does: Marks a class as a JPA entity — a Java object that maps to a database table.

Without it: Hibernate ignores the class. No table is created, no queries work.

Required: The class must have a no-arg constructor (public or protected) and a field annotated with @Id.

@Table

What it does: Specifies the database table name. If omitted, the table name equals the class name.

Why use it: @Table(name = "users") because User class → users table. Without it, the table would be named User (case-sensitive, may cause issues).

Also supports: schema, uniqueConstraints, indexes.

@Id

What it does: Marks the primary key field. Every entity MUST have exactly one @Id.

Without it: Hibernate throws a mapping exception at startup.

@GeneratedValue

What it does: Tells Hibernate how to generate primary key values.

@GeneratedValue(strategy = GenerationType.IDENTITY)
// Uses database auto-increment (SERIAL in PostgreSQL)
// Most common for simple primary keys

@GeneratedValue(strategy = GenerationType.SEQUENCE)
// Uses a database sequence
// Best for performance (pre-allocates IDs)
// Requires @SequenceGenerator

@GeneratedValue(strategy = GenerationType.TABLE)
// Uses a separate table to track IDs
// Portable but slow — avoid

@GeneratedValue(strategy = GenerationType.AUTO)
// Default — Hibernate chooses (SEQUENCE for PostgreSQL)
📌 We use GenerationType.IDENTITY for simplicity. In production with high insert volume, SEQUENCE with allocationSize performs better because it batches ID generation instead of hitting the DB for every insert.

Actual code from our project:

@Entity
@Table(name = "users")
public class User {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    // ...
}

5 Fetch Types: LAZY vs EAGER

Fetch type determines WHEN related data is loaded from the database.

Fetch TypeWhen Data LoadsSQL GeneratedPerformance
LAZY (default for @OneToMany, @ManyToMany)Only when you access the fieldSeparate SELECT when .getProducts() is called✅ Fast initial load, N+1 risk
EAGER (default for @ManyToOne, @OneToOne)Immediately when entity is loadedJOIN or separate SELECT immediately❌ Slower initial load, no N+1

Why we override to LAZY:

// We EXPLICITLY set LAZY everywhere for consistency:
@ManyToOne(fetch = FetchType.LAZY)
@JoinColumn(name = "category_id", nullable = false)
private Category category;

// Why? If we load 100 products and don't need category info,
// EAGER would fetch 100 categories unnecessarily.
// With LAZY, we only query category when we actually access it.
⚠️ LazyInitializationException: If you try to access a lazy field outside a transaction (@Transactional), Hibernate throws LazyInitializationException because the session is closed. Solution: 1) Keep the transaction open, 2) Use JOIN FETCH in JPQL, 3) Use @EntityGraph.
🔍 Q: What is the N+1 query problem and how do you solve it?
N+1: 1 query to fetch products + N queries to fetch each product's category = N+1 total queries. Solution: Use JOIN FETCH p.category in JPQL or @EntityGraph(attributePaths = "category"). This generates a single SQL query with JOIN instead of N+1 separate queries.

6 Cascade Types & Orphan Removal

Cascade propagates operations (save, delete) from parent to child entities.

Cascade Types:

Cascade TypeMeaning

Where we use Cascade:

// Category → SubCategories (parent owns children)
@OneToMany(mappedBy = "parent", cascade = CascadeType.ALL, orphanRemoval = true)
private List<Category> subCategories = new ArrayList<>();

// When you delete a Category, ALL sub-categories are also deleted.
// orphanRemoval = true: if you remove from the list, DB row is deleted.

Where we DON'T use Cascade:

// Product → Category (ManyToOne)
// We NEVER cascade delete a Category when a Product is deleted!
// Products belong to categories, not the other way around.
// Cascade on ManyToOne is almost always wrong.
⚠️ Common Mistake: Adding cascade = CascadeType.ALL on @ManyToOne. This means deleting a Product would ALSO delete the Category — which would delete ALL products in that category. Only cascade from parent to child, never from child to parent.

7 Lifecycle Callbacks — @PrePersist, @PreUpdate

JPA provides lifecycle hooks — methods that execute automatically at certain points in an entity's lifecycle.

@PostLoad      // After SELECT (populate transient fields)
@PostPersist   // After INSERT

Our implementation:

    createdAt = LocalDateTime.now();
    updatedAt = LocalDateTime.now();  // Initially same as createdAt
}

15 Enum Mapping — @Enumerated(STRING) vs ORDINAL

This is a common interview trap and a frequent production bug.

public enum Role {
    CUSTOMER,   // STRING → "CUSTOMER"  | ORDINAL → 0
    SELLER,     // STRING → "SELLER"    | ORDINAL → 1
    ADMIN       // STRING → "ADMIN"     | ORDINAL → 2
}

// What happens if someone inserts a new role in the MIDDLE?
public enum Role {
    CUSTOMER,        // STRING stays "CUSTOMER"  | ORDINAL stays 0 ✅
    PREMIUM_CUSTOMER,// STRING → "PREMIUM" ✅     | ORDINAL → 1 ❌ (was SELLER!)
    SELLER,          // STRING → "SELLER" ✅      | ORDINAL → 2 ❌ (was ADMIN!)
    ADMIN            // STRING → "ADMIN" ✅       | ORDINAL → 3 ❌
}

// With ORDINAL: ALL EXISTING DATA IS CORRUPTED
// User with role 1 (was SELLER) now becomes PREMIUM_CUSTOMER!
✅ ALWAYS use @Enumerated(EnumType.STRING)
❌ NEVER use @Enumerated(EnumType.ORDINAL)

ORDINAL stores integer positions — inserting, removing, or reordering enum values shifts all existing data. STRING stores the literal name — safe from reordering.

16 Interview Questions — JPA & Hibernate

🔰 Beginner

Q1: What is JPA? How is it different from Hibernate?
JPA is a specification (interface contract) for ORM in Java. Hibernate is an implementation of JPA. JPA defines annotations like @Entity, @Id, @OneToMany. Hibernate provides the actual engine that generates SQL, manages sessions, and handles caching. You code against JPA interfaces to stay implementation-agnostic.
Q2: What annotations are needed to make a class a JPA entity?
1) @Entity — marks the class, 2) @Id — marks the primary key, 3) @GeneratedValue — strategy for ID generation, 4) Optionally @Table(name = "...") for table name. The class must have a no-arg constructor (public or protected).
Q3: What is the difference between @Column(nullable = false) and @Column(unique = true)?
nullable = false adds a NOT NULL constraint — the column cannot contain NULL. unique = true adds a UNIQUE constraint — no duplicate values allowed. They are independent constraints. A column can be unique but nullable (PostgreSQL allows multiple NULLs in a unique column).
Q4: Why should you never use @Data on JPA entities?
1) @EqualsAndHashCode includes the id field, which is null before persisting — two unsaved entities become equal. 2) @ToString includes lazy-loaded fields, triggering LazyInitializationException. 3) @RequiredArgsConstructor conflicts with the no-arg constructor required by JPA. Use explicit @Getter @Setter @NoArgsConstructor @AllArgsConstructor @Builder.
Q5: What is the purpose of @PrePersist and @PreUpdate?
They are JPA lifecycle callbacks. @PrePersist fires before INSERT — used to set createdAt. @PreUpdate fires before UPDATE — used to set updatedAt. This ensures timestamps are always set by the persistence layer, not by application code.

📊 Intermediate

Q6: LAZY vs EAGER fetching — when to use each?
LAZY: Data loaded on demand (when accessed). Default for @OneToMany, @ManyToMany. Reduces initial load. Risk: LazyInitializationException if accessed outside transaction. EAGER: Data loaded immediately. Default for @ManyToOne, @OneToOne. Increases initial load but avoids N+1. Best practice: Always use LAZY explicitly and use JOIN FETCH in queries when you need the data.
Q7: What is the N+1 query problem and how do you solve it?
N+1: 1 query to fetch parent entities + N queries to fetch each parent's children. Example: 100 products (1 query) → 100 category queries (100 queries) = 101 total. Solutions: 1) JOIN FETCH p.category in JPQL, 2) @EntityGraph(attributePaths = "category"), 3) @BatchSize(size = 20) to batch lazy loads, 4) Use Specification with fetch joins.
Q8: What is CascadeType.ALL and when should you NOT use it?
CascadeType.ALL propagates all operations (PERSIST, MERGE, REMOVE, REFRESH, DETACH) from parent to child. Use it on the parent side of a composition relationship (e.g., Category → SubCategories, Cart → CartItems). NEVER use it on @ManyToOne (child → parent) — deleting a product would cascade-delete its category, deleting ALL products in that category. Only cascade from parent to child.
Q9: Why is BigDecimal used for prices instead of double/float?
Floating-point arithmetic has precision errors: 0.1 + 0.2 = 0.30000000000000004. For money calculations, this is unacceptable. BigDecimal provides exact arithmetic with arbitrary precision. The database column uses DECIMAL(10,2) for exact numeric storage.
Q10: How does the Adjacency List model work for hierarchical categories?
Each category row stores a parent_id that references the same table's id. Top-level categories have parent_id IS NULL. To get the full tree: WITH RECURSIVE CTE in PostgreSQL. Pros: simple, unlimited depth. Cons: recursive queries needed for full subtree. Alternative: Materialized Path (store path like "001.002.003") for faster subtree queries.

🚀 Advanced

Q11: What is the difference between CascadeType.REMOVE and orphanRemoval = true?
CascadeType.REMOVE: When parent is deleted, children are also deleted. orphanRemoval = true: When a child is removed from the parent's collection, it's automatically deleted from the database. Example: removing a CartItem from Cart.items list → Hibernate deletes that CartItem row. Without orphanRemoval, the child becomes orphaned (foreign key violation or null).
Q12: How would you design an entity for a product that can have variable attributes (e.g., phone has RAM, clothing has size)?
Two approaches: 1) JSONB column: @Column(columnDefinition = "JSONB") private String attributes; — flexible, queryable with PostgreSQL JSON operators. 2) EAV pattern: ProductAttribute(entity_id, attribute_name, attribute_value) table — fully flexible but complex queries (needs pivot). For most cases, JSONB is the better choice — simple, queryable, indexed.
Q13: What happens if you access a lazy-loaded proxy outside of a transaction?
Hibernate throws LazyInitializationException: "could not initialize proxy — no Session". Because the Hibernate session is closed when the transaction ends. The proxy object exists but can't load data without a session. Solutions: 1) Use @Transactional to keep session open, 2) Use JOIN FETCH to pre-load, 3) Use Spring's OpenEntityManagerInViewFilter (anti-pattern — keeps session open for entire HTTP request).
Q14: What is the difference between GenerationType.IDENTITY and GenerationType.SEQUENCE?
IDENTITY: Uses database auto-increment (SERIAL in PostgreSQL). The DB generates the ID on INSERT. Hibernate must execute the INSERT immediately to get the ID — cannot batch inserts. SEQUENCE: Uses a database sequence. Hibernate pre-allocates IDs (e.g., 50 at a time with allocationSize = 50) and assigns them before INSERT — supports batch inserts. For high-volume writes, SEQUENCE with allocationSize performs significantly better.
Q15: You have an Order with OrderItems. How would you implement "snapshot pricing" to ensure historical accuracy?
Store unitPrice and subtotal directly in the OrderItem table. At the time of order placement, copy the product's current price into unitPrice and compute subtotal = quantity × unitPrice. Never use orderItem.product.price to display historical prices — the product price may have changed. This is how Amazon, Flipkart, and all major e-commerce platforms handle it.

1 What is Spring Data JPA?

Spring Data JPA is a Spring module that provides repository support for JPA. It sits on top of JPA + Hibernate and eliminates boilerplate data access code.

Without Spring Data JPA (raw JPA): // You write ALL of this for EVERY entity: @Repository public class UserDao { @PersistenceContext private EntityManager em; public User findById(Long id) { return em.find(User.class, id); } public User save(User user) { if (user.getId() == null) em.persist(user); else em.merge(user); return user; } public void delete(User user) { em.remove(user); } public List findAll() { return em.createQuery("SELECT u FROM User u", User.class).getResultList(); } } With Spring Data JPA: // ONE LINE — Spring generates the implementation! public interface UserRepository extends JpaRepository<User, Long> {} // Now you have: findAll(), findById(), save(), deleteById(), count(), existsById() // Plus: findAll(Pageable), findAll(Sort), findAllById(Iterable)

🔍 Q: Does @Transactional on a repository method extend to the service layer call?
No! @Transactional is not inherited from method calls within the same class. If you call repo.methodA() from service.doSomething(), and methodA() has @Transactional, it only applies to methodA(). For a transaction spanning multiple repo calls, put @Transactional on the service method.

8 @Modifying & @Query for UPDATE/DELETE

For UPDATE and DELETE operations that don't go through the entity lifecycle, use @Modifying + @Query.

@Modifying
@Query("UPDATE Product p SET p.stock = p.stock - :quantity WHERE p.id = :productId AND p.stock >= :quantity")
int reduceStock(@Param("productId") Long productId, @Param("quantity") int quantity);

// Returns: number of rows affected (0 if insufficient stock)

@Modifying
@Query("DELETE FROM CartItem ci WHERE ci.cart.id = :cartId")
void clearCart(@Param("cartId") Long cartId);

Important Rules for @Modifying:

  • Must be used with @Query — cannot be used with derived methods
  • Must be in a transaction — the method needs @Transactional (or the calling service method needs it)
  • Clears persistence context — after execution, Hibernate's first-level cache is cleared. Any managed entities must be re-fetched
  • Add clearAutomatically = true, flushAutomatically = true for safety
// Best practice:
@Modifying(clearAutomatically = true, flushAutomatically = true)
@Query("UPDATE Product p SET p.stock = p.stock - :qty WHERE p.id = :id AND p.stock >= :qty")
int reduceStock(@Param("id") Long id, @Param("qty") int qty);
⚠️ Why clearAutomatically? After a bulk UPDATE, Hibernate's first-level cache still has the old values. If you then read the same entity, you'll get stale data. Setting clearAutomatically = true clears the cache, forcing the next read to go to the database.

public interface CartRepository extends JpaRepository<Cart, Long> {

Why JOIN FETCH matters here:

// Without JOIN FETCH (just findByUserId):
// 1. SELECT * FROM carts WHERE user_id = 1     (1 query)
// 2. When you call cart.getItems()...
//    SELECT * FROM cart_items WHERE cart_id = 1 (1 more query)
// Total: 2 queries (N+1 with 1 cart)

// 1. SELECT c.*, ci.* FROM carts c

// Total: 1 query — all data loaded at once!

14 JOIN FETCH & Entity Graphs for N+1

The N+1 query problem is the most common performance issue in JPA. Let's see how to solve it.

Method 1: JOIN FETCH in @Query

// Without JOIN FETCH: N+1 queries
@Query("SELECT o FROM Order o WHERE o.user.id = :userId")
// For 20 orders → 1 (orders) + 20 (orderItems) = 21 queries

// With JOIN FETCH: 1 query
@Query("SELECT o FROM Order o JOIN FETCH o.orderItems WHERE o.user.id = :userId")
// 1 query with JOIN — all data loaded at once

Method 2: @EntityGraph (declarative, no JPQL needed)

// Define a named entity graph on the entity:
@Entity
@NamedEntityGraph(name = "Order.withItems",
                  attributeNodes = @NamedAttributeNode("orderItems"))
public class Order { ... }

// Use it in the repository:
public interface OrderRepository extends JpaRepository<Order, Long> {

    @EntityGraph("Order.withItems")
    Page<Order> findByUserId(Long userId, Pageable pageable);
    // Automatically generates JOIN FETCH without writing @Query
}

Method 3: @BatchSize (lazy load in batches)

@Entity
public class Product {
    // ...

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "category_id")
    @BatchSize(size = 20)   // Load 20 categories at once
    private Category category;
}

// Without @BatchSize: 100 queries for 100 products
// With @BatchSize(size=20): 100/20 = 5 queries
MethodProsCons
JOIN FETCHExplicit control, works with @QueryMust write JPQL, can't use derived methods
@EntityGraphWorks with derived methods, declarativeNeed to define graph on entity
@BatchSizeTransparent, no query changesStill N/batches queries (not 1)
💡 Interview Answer: "The N+1 problem occurs when you load N parent entities and each triggers an additional query to load its children. I solve it using JOIN FETCH in @Query for specific use cases, @EntityGraph for derived methods, and @BatchSize as a safety net for unpredictable access patterns."

15 Locking — @Lock(PESSIMISTIC_WRITE) for Inventory

During checkout, multiple users might try to buy the last item simultaneously. Without locking, we get race conditions — two users both see stock=1 and both place orders, resulting in stock=-1.

Pessimistic Locking:

public interface ProductRepository extends JpaRepository<Product, Long> {

    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("SELECT p FROM Product p WHERE p.id = :id")
    Optional<Product> findByIdWithLock(@Param("id") Long id);
}

// When called:
// SELECT p.* FROM products p WHERE p.id = 1 FOR UPDATE
// The FOR UPDATE locks the row — other transactions must WAIT

Optimistic Locking:

// Add a version field to the Product entity:
@Version
private Integer version;

// When updating:
// UPDATE products SET stock = stock - 1, version = 2
// WHERE id = 1 AND version = 1
// If version changed (another transaction updated it first) → OptimisticLockException
// You catch the exception and retry
Locking StrategyHow It WorksPerformanceBest For
PessimisticSELECT ... FOR UPDATE — locks the row❌ Slower (other transactions wait)High contention, low latency requirement (checkout)
Optimistic@Version — checks on commit✅ Faster (no locks, retry on conflict)Low contention, read-heavy (most scenarios)
No LockingLast commit wins✅ Fastest❌ Data corruption risk in e-commerce
📌 For inventory deduction during checkout, we use PESSIMISTIC_WRITE — the checkout transaction is short (milliseconds) and the cost of overselling (negative stock, cancelled orders) is much higher than the cost of waiting. Optimistic locking would cause retries at the worst possible time (user is trying to pay).

16 Specification & Criteria API (Advanced)

When queries need dynamic filters (user selects "Category = Electronics, Price 500-2000, Sort by Rating"), building a query string or having 50 derived methods is impractical. Use Specifications.

// Step 1: Make your repository extend JpaSpecificationExecutor
public interface ProductRepository extends JpaRepository<Product, Long>,
                                           JpaSpecificationExecutor<Product> {}

// Step 2: Create specification builder
public class ProductSpecification {

    public static Specification<Product> hasCategory(Long categoryId) {
        return (root, query, cb) ->
            categoryId == null ? null : cb.equal(root.get("category").get("id"), categoryId);
    }

    public static Specification<Product> priceBetween(BigDecimal min, BigDecimal max) {
        return (root, query, cb) -> {
            if (min == null && max == null) return null;
            if (min == null) return cb.lessThanOrEqualTo(root.get("price"), max);
            if (max == null) return cb.greaterThanOrEqualTo(root.get("price"), min);
            return cb.between(root.get("price"), min, max);
        };
    }

    public static Specification<Product> nameContains(String keyword) {
        return (root, query, cb) ->
            keyword == null ? null :
                cb.like(cb.lower(root.get("name")), "%" + keyword.toLowerCase() + "%");
    }
}

// Step 3: Use in service
Specification<Product> spec = Specification
    .where(ProductSpecification.hasCategory(categoryId))
    .and(ProductSpecification.priceBetween(minPrice, maxPrice))
    .and(ProductSpecification.nameContains(keyword));

Page<Product> products = productRepository.findAll(spec, pageable);
🔑 Why Specifications? Without them, you'd need 10+ derived methods: findByCategoryIdAndPriceBetween, findByCategoryIdAndNameContaining, findByNameContainingAndPriceBetween, etc. Specifications compose dynamically — each filter is optional and combinable.

17 Interview Questions — Spring Data JPA

🔰 Beginner

Q1: What is Spring Data JPA? How does it differ from JPA and Hibernate?
JPA is a specification for ORM. Hibernate is an implementation of JPA. Spring Data JPA is a Spring module that provides repository support — it automatically generates implementations for interface methods based on naming conventions. You just declare an interface extending JpaRepository, Spring provides the implementation.
Q2: What is the difference between CrudRepository, PagingAndSortingRepository, and JpaRepository?
CrudRepository: Basic CRUD (save, findById, deleteById, count). PagingAndSortingRepository: extends CrudRepository + adds findAll(Pageable) and findAll(Sort). JpaRepository: extends PagingAndSortingRepository + adds JPA-specific methods (flush, saveAndFlush, deleteInBatch). Always use JpaRepository.
Q3: How does Spring Data JPA know what query to execute from a method name?
It parses the method name using a strategy pattern. For example, findByEmailAndRole(String email, Role role) is parsed as: findBy (action) → Email (field) → And (conjunction) → Role (field). It generates: SELECT u FROM User u WHERE u.email = ?1 AND u.role = ?2. Method name parsing happens at startup, so naming errors are caught immediately.
Q4: What is the difference between findByEmail and existsByEmail?
findByEmail loads the full entity (SELECT all columns). existsByEmail generates a COUNT query (SELECT COUNT(1)) — it only returns true/false. Always use existsBy when you only need to check existence. It's faster and uses less memory.
Q5: What does Pageable allow you to do?
Pageable encapsulates pagination and sorting parameters: page number, page size, and sort criteria. When passed to a repository method, Spring Data JPA automatically generates LIMIT/OFFSET and ORDER BY clauses. It returns a Page object containing the data plus total count, total pages, and navigation metadata.

📊 Intermediate

Q6: When would you use @Query instead of a derived method name?
1) Complex queries with OR conditions across different fields (like search across name AND description with same keyword). 2) Queries using database-specific features (full-text search, JSONB queries). 3) JOIN FETCH to solve N+1. 4) UPDATE/DELETE operations (must use @Modifying). 5) When the method name would be too long (over 30 characters).
Q7: What is the N+1 problem and how do you solve it in Spring Data JPA?
Problem: 1 query to fetch orders + N queries to fetch each order's items = N+1. Solutions: 1) JOIN FETCH in @Query, 2) @EntityGraph for derived methods, 3) @BatchSize to batch lazy loads. Example: @Query("SELECT o FROM Order o JOIN FETCH o.orderItems") loads everything in one query.
Q8: What is the difference between @Query with and without nativeQuery=true?
JPQL (nativeQuery=false, default): Queries Java entities by class name (SELECT p FROM Product p). Database-independent. Hibernate translates to SQL. Native (nativeQuery=true): Queries database tables directly (SELECT * FROM products). Database-specific. Use native for DB-specific features (PostgreSQL's tsvector, MySQL's MATCH AGAINST) or complex SQL that JPQL can't express.
Q9: What is @Modifying and when is it used?
@Modifying is used with @Query for UPDATE and DELETE operations. It tells Spring Data JPA that the query modifies data, so the persistence context should be cleared after execution. Set clearAutomatically = true and flushAutomatically = true for safety. Example: bulk inventory updates, cart clearing.
Q10: What is the difference between Page and Slice?
Page executes TWO queries: one for data (LIMIT/OFFSET) and one for COUNT. Provides total pages, total elements. Slice executes ONE query: fetches data + one extra row to check if next page exists. No count query. Use Page when you need pagination controls (total pages). Use Slice for infinite scroll (no total count needed).

🚀 Advanced

Q11: What is PESSIMISTIC_WRITE locking and how does it prevent overselling?
@Lock(PESSIMISTIC_WRITE) generates SELECT ... FOR UPDATE, which locks the selected rows until the transaction completes. When User A is checking out and has the inventory row locked, User B's query waits. This prevents two users from buying the last item simultaneously. Use for high-contention scenarios like last-item checkout.
Q12: How would you implement a dynamic filtering system where users can filter products by category, price range, brand, and rating — all optional?
Use Specifications (JpaSpecificationExecutor). Define a Specification for each filter: hasCategory(), priceBetween(), hasBrand(), minRating(). Each returns null if the filter is not provided. Compose: Specification.where(hasCategory(catId)).and(priceBetween(min, max)). Pass to findAll(spec, pageable). This is scalable, readable, and testable.
Q13: You're getting a LazyInitializationException when accessing order.getOrderItems() outside a transaction. How do you fix it?
Three approaches: 1) JOIN FETCH in the query — @Query("SELECT o FROM Order o JOIN FETCH o.orderItems"). 2) @EntityGraph — @EntityGraph(attributePaths = "orderItems") on the repository method. 3) Open Session in View (OSIV) — Spring Boot's default keeps the session open for the entire HTTP request (convenient but an anti-pattern for production — can cause connection exhaustion).
Q14: How do you handle pagination for a query that uses JOIN FETCH on a collection?
You can't use Pageable with JOIN FETCH on a to-many collection — Hibernate warns: "firstResult/maxResults specified with collection fetch; applying in memory!" This means pagination happens in Java memory (all rows loaded), defeating the purpose. Solution: Use @BatchSize instead of JOIN FETCH, or write a two-step query: 1) Page of IDs with Pageable, 2) Fetch entities by those IDs with JOIN FETCH.

1 What is the Service Layer?

The Service Layer is the heart of the application — it contains all business logic. It sits between controllers (HTTP layer) and repositories (data access layer).

Controller Layer (HTTP) ↓ receives DTOs, calls services Service Layer (Business Logic) ← @Service ↓ validates, orchestrates, transforms Repository Layer (Data Access) ← @Repository ↓ queries database Database (PostgreSQL)

What the Service Layer Does:

ResponsibilityExample
Business Rules"Cannot cancel order after shipping", "Stock cannot go negative"
OrchestrationPlace Order: check stock → deduct inventory → create order → clear cart → send email
Validation"Email already exists", "Product belongs to seller"
DTO ConversionEntity ←→ Response DTO (never expose entities to controllers)
Transaction Management@Transactional — multiple DB operations succeed or fail together
Authorization"Only the seller can update this product"
🔑 Why a separate Service Layer? Controllers should only handle HTTP concerns (status codes, headers). If business logic leaks into controllers, you can't reuse it (e.g., same order logic for web + mobile + API). Services are framework-agnostic and testable in isolation.

3 AuthService — Registration Flow

@Transactional
public AuthResponse register(RegisterRequest request) {

    // 1. Business validation — check duplicate email
    if (userRepository.existsByEmail(request.getEmail())) {
        throw new BadRequestException("Email already registered");
    }

    // 2. Role assignment — prevent self-assign ADMIN
    Role role = Role.CUSTOMER;  // Default role
    if (request.getRole() != null) {
        try {
            role = Role.valueOf(request.getRole().toUpperCase());
            if (role == Role.ADMIN) {
                throw new BadRequestException("Admin registration not allowed");
                // Only existing admins can create other admins
            }
        } catch (IllegalArgumentException e) {
            throw new BadRequestException("Invalid role: " + request.getRole());
        }
    }

    // 3. Entity creation with Builder pattern
    User user = User.builder()
            .name(request.getName())
            .email(request.getEmail())
            .password(passwordEncoder.encode(request.getPassword()))  // BCrypt hash!
            .role(role)
            .build();

    // 4. Persist to database
    user = userRepository.save(user);

    // 5. Generate JWT tokens immediately (user is logged in after register)
    String accessToken = jwtTokenProvider.generateAccessToken(
            user.getId(), user.getEmail(), user.getRole().name());
    String refreshToken = jwtTokenProvider.generateRefreshToken(user.getId());

    // 6. Build response
    return buildAuthResponse(user, accessToken, refreshToken);
}

Security considerations:

  • Password is NEVER stored as plain text — passwordEncoder.encode() uses BCrypt (slow, salted hash)
  • Role is validated — user can't register as ADMIN
  • Email uniqueness — checked at application level (DB has UNIQUE constraint as backup)
  • @Transactional — if anything fails, user creation is rolled back

public AuthResponse login(LoginRequest request) {

    // 1. Delegate authentication to Spring Security's AuthenticationManager
    //    This triggers CustomUserDetailsService.loadUserByUsername()
    //    which fetches user from DB and verifies password
    try {
        authenticationManager.authenticate(
                new UsernamePasswordAuthenticationToken(
                        request.getEmail(), request.getPassword()));
    } catch (Exception e) {
        throw new UnauthorizedException("Invalid email or password");
        // We don't say WHICH one is wrong — security by obscurity
    }

    // 2. Fetch the authenticated user
    User user = userRepository.findByEmail(request.getEmail())
            .orElseThrow(() -> new UnauthorizedException("User not found"));

    // 3. Generate JWT tokens
    String accessToken = jwtTokenProvider.generateAccessToken(
            user.getId(), user.getEmail(), user.getRole().name());
    String refreshToken = jwtTokenProvider.generateRefreshToken(user.getId());

    // 4. Build and return response
    return buildAuthResponse(user, accessToken, refreshToken);
}

JWT Token Structure (from JwtTokenProvider):

Access Token (15 min expiry):
  Header:  { "alg": "HS256" }
  Payload: { "sub": "123", "email": "user@ecom.com",
             "role": "CUSTOMER", "iat": 1700000000, "exp": 1700000900 }
  Signature: HMAC-SHA256(base64(header) + "." + base64(payload), secret)

Refresh Token (7 days expiry):
  Payload: { "sub": "123", "type": "refresh", "iat": 1700000000, "exp": 1700604800 }

5 AuthService — Refresh Token

public AuthResponse refreshToken(String refreshToken) {

    // 1. Validate the refresh token (check signature + expiry)
    if (!jwtTokenProvider.validateToken(refreshToken)) {
        throw new UnauthorizedException("Invalid or expired refresh token");
    }

    // 2. Extract user ID from the token (no DB call needed yet)
    Long userId = jwtTokenProvider.getUserIdFromToken(refreshToken);

    // 3. Verify user still exists (they might have been deleted)
    User user = userRepository.findById(userId)
            .orElseThrow(() -> new UnauthorizedException("User not found"));

    // 4. Generate new tokens (old refresh token is discarded by client)
    String newAccessToken = jwtTokenProvider.generateAccessToken(
            user.getId(), user.getEmail(), user.getRole().name());
    String newRefreshToken = jwtTokenProvider.generateRefreshToken(user.getId());

    return buildAuthResponse(user, newAccessToken, newRefreshToken);
}

Token Rotation Security:

  • Short-lived access tokens (15 min) — if stolen, limited damage window
  • Refresh token is a long-lived credential (7 days)
  • In production: store refresh tokens in Redis with user mapping → invalidate on logout
  • Token rotation: each refresh invalidates the old refresh token (prevents replay)

8 CartService — Add, Update, Remove Items

@Transactional
public CartResponse addItemToCart(Long userId, CartItemRequest request) {
    // 1. Get existing cart or create new one
    Cart cart = getOrCreateCart(userId);

    // 2. Verify product exists and has stock
    Product product = productRepository.findById(request.getProductId())
            .orElseThrow(() -> new ResourceNotFoundException("Product not found"));
    if (product.getStock() < request.getQuantity()) {
        throw new BadRequestException("Insufficient stock. Available: " + product.getStock());
    }

    // 3. Check if product ALREADY in cart → increment quantity
    //    Otherwise → add new item
    cart.getItems().stream()
            .filter(item -> item.getProduct().getId().equals(request.getProductId()))
            .findFirst()
            .ifPresentOrElse(
                    existing -> existing.setQuantity(
                            existing.getQuantity() + request.getQuantity()),
                    () -> {
                        CartItem newItem = CartItem.builder()
                                .cart(cart)
                                .product(product)
                                .quantity(request.getQuantity())
                                .build();
                        cart.getItems().add(newItem);
                    });

    // 4. Cascade.ALL + orphanRemoval = true handles saving items automatically
    cart = cartRepository.save(cart);
    return toCartResponse(cart);
}
            

            
Why ifPresentOrElse instead of if-else?

            
ifPresentOrElse is a functional programming pattern. It handles both cases in a single stream operation: if product exists in cart → increment quantity; if not → add new item. This is cleaner than checking with a separate findFirst() and then looping again.

9 OrderService — Place Order with Inventory Deduction

This is the most critical business operation — it spans multiple entities and must be atomic.

@Transactional  // ← ALL or NOTHING
public OrderResponse placeOrder(Long userId, String shippingAddress) {

    // 1. Get user's cart (must have items)
    Cart cart = cartService.getOrCreateCart(userId);
    if (cart.getItems().isEmpty()) {
        throw new BadRequestException("Cart is empty");
    }

    // 2. For each cart item: validate stock + deduct inventory
    List<OrderItem> orderItems = cart.getItems().stream()
            .map(cartItem -> {
                Product product = cartItem.getProduct();

                // Stock validation
                if (product.getStock() < cartItem.getQuantity()) {
                    throw new BadRequestException(
                            "Insufficient stock for: " + product.getName());
                }

                // Inventory deduction
                product.setStock(product.getStock() - cartItem.getQuantity());
                productRepository.save(product);

                // Snapshot pricing (copy price at time of order)
                BigDecimal unitPrice = product.getPrice();
                BigDecimal subtotal = unitPrice.multiply(
                        BigDecimal.valueOf(cartItem.getQuantity()));

                return OrderItem.builder()
                        .product(product)
                        .quantity(cartItem.getQuantity())
                        .unitPrice(unitPrice)    // ← SNAPSHOT
                        .subtotal(subtotal)      // ← PRE-COMPUTED
                        .build();
            })
            .collect(Collectors.toList());

    // 3. Calculate total
    BigDecimal totalAmount = orderItems.stream()
            .map(OrderItem::getSubtotal)
            .reduce(BigDecimal.ZERO, BigDecimal::add);

    // 4. Create order
    Order order = Order.builder()
            .user(cart.getUser())
            .orderStatus(PENDING)
            .paymentStatus(UNPAID)
            .totalAmount(totalAmount)
            .shippingAddress(shippingAddress)
            .build();

    orderItems.forEach(item -> item.setOrder(order));  // Set bidirectional
    order.setOrderItems(orderItems);

    // 5. Save order (cascade saves orderItems)
    order = orderRepository.save(order);

    // 6. Clear the cart
    cart.getItems().clear();
    cartRepository.save(cart);

    return toOrderResponse(order);
}
🔑 @Transactional is CRITICAL here: If inventory deduction succeeds but order creation fails, the stock is already deducted — product is "sold" but no order exists. @Transactional rolls back ALL operations if ANY step fails. In production with microservices, you'd need the Saga pattern; with a monolith, @Transactional is sufficient.

10 OrderService — Cancel Order with Inventory Restore

@Transactional
public OrderResponse cancelOrder(Long orderId, Long userId) {
    Order order = orderRepository.findById(orderId)...;

    // Authorization check
    if (!order.getUser().getId().equals(userId)) {
        throw new BadRequestException("Not authorized to cancel this order");
    }

    // Business rule: can't cancel shipped/delivered orders
    if (order.getOrderStatus() == SHIPPED ||
        order.getOrderStatus() == DELIVERED) {
        throw new BadRequestException("Cannot cancel order in " +
                order.getOrderStatus() + " status");
    }

    // COMPENSATING ACTION: Restore inventory
    order.getOrderItems().forEach(item -> {
        Product product = item.getProduct();
        product.setStock(product.getStock() + item.getQuantity());
        productRepository.save(product);
    });

    order.setOrderStatus(CANCELLED);
    order = orderRepository.save(order);

    return toOrderResponse(order);
}

The Compensating Transaction Pattern:

When we cancel an order, we must undo the inventory deduction that happened during placeOrder(). This is a manual compensating transaction — the same concept used in Saga Pattern for microservices. Every operation that changes state should have a corresponding "undo" operation.

11 PaymentService — Razorpay Integration

@Service
public class PaymentService {

    private RazorpayClient razorpayClient;

    @PostConstruct  // Runs after constructor — initializes the Razorpay client
    public void init() throws RazorpayException {
        this.razorpayClient = new RazorpayClient(razorpayKeyId, razorpayKeySecret);
    }

    @Transactional
    public String createRazorpayOrder(Long orderId) {
        Order order = orderRepository.findById(orderId)...;

        // Create Razorpay order (amount in paise: ₹100 = 10000 paise)
        JSONObject orderRequest = new JSONObject();
        orderRequest.put("amount", order.getTotalAmount()
                .multiply(BigDecimal.valueOf(100)).intValue());  // ₹ → paise
        orderRequest.put("currency", "INR");
        orderRequest.put("receipt", "order_" + orderId);
        orderRequest.put("payment_capture", 1);  // Auto-capture payment

        RazorpayOrder razorpayOrder = razorpayClient.orders.create(orderRequest);

        // Store Razorpay order ID in our order
        order.setRazorpayOrderId(razorpayOrder.get("id"));
        orderRepository.save(order);

        return razorpayOrder.get("id");  // Return to frontend
    }

    public void verifyPayment(String razorpayOrderId,
                              String razorpayPaymentId,
                              String signature) {
        // Razorpay sends: signature = HMAC-SHA256(orderId + "|" + paymentId, secret)
        String expectedSignature = HmacSHA256(
                razorpayOrderId + "|" + razorpayPaymentId,
                razorpayKeySecret);

        if (!expectedSignature.equals(signature)) {
            throw new RuntimeException("Invalid payment signature — possible tampering!");
        }

        // Update order status
        Order order = findByRazorpayOrderId(razorpayOrderId);
        order.setRazorpayPaymentId(razorpayPaymentId);
        order.setPaymentStatus(PAID);
        orderRepository.save(order);
    }
}

Payment Flow:

1. Backend creates Razorpay Order → returns razorpay_order_id
2. Frontend opens Razorpay checkout with this ID
3. User completes payment on Razorpay's page
4. Razorpay redirects to backend with: payment_id + signature
5. Backend verifies signature (HMAC-SHA256)
6. If valid → update order as PAID
7. If invalid → reject (possible fraud)

12 EmailService — Async Notifications

@Service
@Slf4j
public class EmailService {

    private final JavaMailSender mailSender;

    @Async  // ← Runs in a separate thread! Does NOT block the main flow
    public void sendOrderConfirmation(String to, String orderId, String status) {

            message.setSubject("Order Confirmation - Order #" + orderId);

  • In production: Use a message queue (RabbitMQ/Kafka) instead of @Async. The queue guarantees delivery even if the email service restarts.

13 DTO Pattern — Why Separate Request/Response Objects?

DTO = Data Transfer Object. A separate object for API communication, distinct from JPA entities.

Never expose entities directly:

// ❌ WRONG — returning entity directly:
@GetMapping("/products/{id}")
public Product getProduct(@PathVariable Long id) {
    return productRepository.findById(id).get();
    // Problems: exposes ALL fields (including internal ones),
    // triggers lazy loading in serialization, circular references
}

// ✅ RIGHT — return DTO:
@GetMapping("/products/{id}")
public ProductResponse getProduct(@PathVariable Long id) {
    return productService.getProductById(id);
    // Only exposes the fields the API consumer needs
}

ReasonExplanation

Services throw business exceptions. The GlobalExceptionHandler catches them and returns proper HTTP responses.

ResourceNotFoundException  → 404 NOT_FOUND

// GlobalExceptionHandler pattern:
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(ResourceNotFoundException.class)
    public ResponseEntity<Map<String, Object>> handleNotFound(ResourceNotFoundException ex) {
        return buildResponse(HttpStatus.NOT_FOUND, ex.getMessage());
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<Map<String, Object>> handleValidation(
            MethodArgumentNotValidException ex) {
        // Returns field-level errors:
        // { "name": "Name is required", "email": "Invalid email format" }
        Map<String, String> errors = new HashMap<>();
        ex.getBindingResult().getAllErrors().forEach(error -> {
            String field = ((FieldError) error).getField();
            String message = error.getDefaultMessage();
            errors.put(field, message);
        });
        return ResponseEntity.badRequest().body(errors);
    }

    @ExceptionHandler(Exception.class)
    public ResponseEntity<Map<String, Object>> handleGeneral(Exception ex) {
        // Fallback — never expose stack traces to client!
        return buildResponse(HttpStatus.INTERNAL_SERVER_ERROR,
                "An unexpected error occurred");
    }
}
⚠️ NEVER expose stack traces to clients! In the general Exception handler, we return "An unexpected error occurred" — not the exception message or stack trace. Stack traces contain internal details (package names, DB structure) that hackers can exploit. Log the full error server-side, but return a generic message to the client.

17 Interview Questions — Service Layer & Business Logic

🔰 Beginner

Q1: What is the Service Layer and why is it important?
The Service Layer contains all business logic. It sits between controllers (HTTP) and repositories (data). It handles validation, orchestration, DTO conversion, transactions, and authorization. Without it, business logic leaks into controllers (not reusable, hard to test) or into repositories (ORM concerns mixed with business rules).
Q2: What is the difference between @Service, @Component, and @Repository?
All three register classes as Spring beans. @Service indicates business logic layer. @Repository indicates data access layer (Spring adds exception translation for JDBC exceptions). @Component is a generic bean for any other layer. They're functionally identical for bean scanning but semantically important for layering.
Q3: Why is constructor injection preferred over @Autowired field injection?
1) Immutability (final fields). 2) Testability (can pass mocks without Spring). 3) Explicit dependencies (constructor grows if class has too many responsibilities). 4) Null safety (Spring ensures all dependencies are set). Field injection hides dependencies and makes testing harder.
Q4: What is the purpose of DTOs? Why not return entities directly?
DTOs decouple the API contract from the DB schema. They prevent: exposing sensitive fields (password), lazy loading exceptions during serialization, circular references (Order→User→Order), and sending unnecessary data over the wire. They also allow you to change the DB schema without breaking API clients.
Q5: What does @Transactional do? When should you use it?
@Transactional ensures that multiple database operations execute in a single database transaction — all succeed or all roll back. Use it on service methods that perform multiple writes (e.g., placeOrder: deduct inventory + create order + clear cart). Use @Transactional(readOnly = true) on read-only methods to optimize performance.

📊 Intermediate

Q6: Explain the "Place Order" flow — what happens from request to response?
1) Controller receives POST /api/orders with shipping address. 2) JWT filter validates token → sets authentication. 3) @PreAuthorize checks role = CUSTOMER. 4) OrderService.placeOrder(): fetch cart → validate stock → deduct inventory → create order with order items (snapshot pricing) → clear cart. 5) @Transactional ensures atomicity. 6) Return OrderResponse DTO.
Q7: How does the refresh token flow work?
1) Client sends refresh token to POST /api/auth/refresh. 2) AuthService validates the refresh token (signature + expiry). 3) Extracts userId from token. 4) Verifies user still exists. 5) Generates NEW access token + NEW refresh token. 6) Client discards old tokens. This provides continuous authentication without asking the user to log in every 15 minutes.
Q8: How does JwtAuthenticationFilter work? Walk through the entire filter chain.
OncePerRequestFilter runs on every request. 1) Extracts "Bearer token" from Authorization header. 2) Validates JWT signature + expiry using JwtTokenProvider. 3) Parses userId/role/email from claims. 4) Loads UserDetails from CustomUserDetailsService. 5) Creates UsernamePasswordAuthenticationToken. 6) Sets SecurityContextHolder — now @PreAuthorize works in controllers.
Q9: How would you handle the scenario where email server is down during order placement?
Email should be sent asynchronously using @Async. The order placement completes and returns immediately. The email is sent in a background thread. If the email fails, we log the error but the order is already confirmed. In production, use a message queue (RabbitMQ) — the queue persists the message and retries delivery, so the email eventually succeeds.
Q10: What is the difference between authentication and authorization?
Authentication = Who are you? (Login with email + password → JWT token). Authorization = What can you do? (Role check: @PreAuthorize("hasRole('ADMIN')")). Authentication happens once per request (JWT filter). Authorization happens per-endpoint (@PreAuthorize). You must be authenticated before authorization can be checked.

🚀 Advanced

Q11: How do you prevent race conditions when two users try to buy the same last item simultaneously?
Use pessimistic locking. In OrderService.placeOrder(), when deducting inventory: @Lock(PESSIMISTIC_WRITE) on the product repository query. This generates SELECT ... FOR UPDATE, which locks the product row until the transaction commits. User B's query waits. Without this, both users would see stock=1 and both would deduct, resulting in stock=-1.
Q12: Design a service method that handles bulk order status update (e.g., ship 100 orders at once).
@Transactional
public int bulkUpdateStatus(List<Long> orderIds, OrderStatus newStatus) {
    List<Order> orders = orderRepository.findAllById(orderIds);
    int updated = 0;
    for (Order order : orders) {
        if (canTransition(order.getOrderStatus(), newStatus)) {
            order.setOrderStatus(newStatus);
            updated++;
        }
    }
    return updated;  // Number of orders actually updated
}

@Modifying
@Query("UPDATE Order o SET o.orderStatus = :status WHERE o.id IN :ids")

For 100+ orders, use the @Modifying approach (single SQL statement). For smaller batches where you need status transition validation, the entity approach is safer.
Q13: How would you implement role-based access control where SELLER can only manage their own products, but ADMIN can manage all?
Two layers: 1) Method-level with @PreAuthorize: @PreAuthorize("hasAnyRole('SELLER', 'ADMIN')") on ProductController. 2) Service-level with entity owner check: if (!product.getSeller().getId().equals(userId) && !isAdmin) { throw new UnauthorizedException(); }. The first layer ensures only sellers/admins can access the endpoint. The second ensures sellers can only touch their own products.
Q14: Explain the compensating transaction pattern as used in OrderService.cancelOrder().
When a user cancels an order, we must undo the inventory deduction that happened during placeOrder(). The compensating action: product.setStock(product.getStock() + item.getQuantity()). In a monolith, this is straightforward (same transaction). In microservices (Saga pattern), you'd publish an "InventoryRelease" event to the Inventory Service. Every write operation should have a corresponding compensating operation.
Q15: What happens if @Transactional on a service method calls another @Transactional method within the same class?
This is a proxy limitation — Spring's @Transactional works via AOP proxies. When methodA() (with @Transactional) calls methodB() (with @Transactional) within the same class, methodB's @Transactional is IGNORED because the call goes through this (the actual object), not the proxy. This is called self-invocation. Solutions: 1) Move methodB to a separate service class, 2) Inject self-reference via ApplicationContext, 3) Use AspectJ weaving (avoid unless necessary).

2 Controller Annotations Deep Dive

@RestController

Combines @Controller + @ResponseBody. Every method return value is automatically serialized to JSON (via Jackson) and written to the HTTP response body. No need for @ResponseBody on each method.

@RequestMapping

Class-level annotation that defines the base URL path. All endpoint methods in the class are prefixed with this path.

Shortcut Annotations

Spring provides syntactic sugar over @RequestMapping(method = ...):

  • @GetMapping — GET requests
  • @PostMapping — POST requests
  • @PutMapping — PUT requests
  • @PatchMapping — PATCH requests
  • @DeleteMapping — DELETE requests

@PathVariable

Extracts values from URL path placeholders. E.g., /api/products/{id} → @PathVariable Long id.

@RequestParam

Extracts query parameters. Supports required, defaultValue. E.g., @RequestParam(required = false) String q for search.

@RequestBody

Deserializes the HTTP request body (JSON) into a Java object, usually a DTO. Works with @Valid for bean validation.

@Valid

Triggers Jakarta Bean Validation on the annotated parameter. If validation fails, Spring throws MethodArgumentNotValidException, which our GlobalExceptionHandler catches and returns 400 Bad Request.

@AuthenticationPrincipal

Injects the currently authenticated UserDetails (or custom principal) directly into the controller method parameter. We use a custom UserPrincipal that extends User and exposes getUserId() and getUserEmail().

ResponseEntity<T>

Gives full control over the HTTP response: status code, headers, and body. We use it to return appropriate status codes (201 Created, 204 No Content) and structured JSON bodies.

@Validated vs @Valid

  • @Valid — standard Jakarta Bean Validation, triggers cascaded validation on nested objects
  • @Validated — Spring's variant, supports validation groups (e.g., @Validated(Create.class))

12 15 Interview Q&A

Q1: What's the difference between @Controller and @RestController?
@RestController = @Controller + @ResponseBody. @Controller is used for MVC views (returns template names), @RestController is for REST APIs — it serializes return values directly to JSON/XML via HttpMessageConverter.
Q2: How does Spring deserialize JSON into Java objects in @RequestBody?
Spring uses HttpMessageConverter implementations. For JSON, it uses MappingJackson2HttpMessageConverter (Jackson's ObjectMapper). It reads the request body, parses JSON, and maps fields to the target DTO using getters/setters or constructors.
Q3: What is the difference between @RequestParam and @PathVariable?
@PathVariable extracts values from URL path placeholders (e.g., /products/{id}), while @RequestParam extracts query parameters (e.g., ?category=books). Path variables are for mandatory resource identifiers; request params are for optional filters.
Q4: How do you handle validation errors in a REST controller?
Use @Valid on @RequestBody parameters. Spring automatically throws MethodArgumentNotValidException on validation failure. A @ControllerAdvice (GlobalExceptionHandler) catches it and returns a structured 400 Bad Request response with field-level error messages.
Q5: What is the purpose of @AuthenticationPrincipal?
It injects the currently authenticated principal (UserDetails) from SecurityContext into a controller method parameter. We use a custom UserPrincipal that extends User and exposes getUserId() and getUserEmail(), giving controllers access to the authenticated user's ID without manually extracting from SecurityContextHolder.
Q6: What is ResponseEntity and when would you use it vs a plain return type?
ResponseEntity gives full control over the HTTP response — status code, headers, and body. Use it when you need non-standard statuses (201 Created, 204 No Content) or custom headers. For simple 200 OK responses, a plain return type is fine.
Q7: How does @PreAuthorize work internally?
Spring AOP proxies the controller bean. Before method invocation, the MethodSecurityInterceptor checks the SpEL expression against the current SecurityContext. If the user lacks the required authority, it throws AccessDeniedException, which our GlobalExceptionHandler converts to 403 Forbidden.
Q8: What's the difference between hasRole() and hasAuthority()?
hasRole('ADMIN') checks for ROLE_ADMIN (Spring prepends "ROLE_" prefix automatically). hasAuthority('ROLE_ADMIN') checks the exact string. hasRole('ADMIN') is equivalent to hasAuthority('ROLE_ADMIN'). Always use hasRole() for role-based checks.
Q9: How does Swagger UI know about your API without manual documentation?
springdoc-openapi scans all @RestController beans at runtime, reads the request mappings, method parameters, annotations (@GetMapping, @PathVariable, @RequestBody, etc.), and generates an OpenAPI 3.0 JSON spec. Swagger UI then renders this spec as an interactive documentation page.
Q10: What is the difference between @GetMapping and @RequestMapping(method = RequestMethod.GET)?
They are functionally identical. @GetMapping is a composed annotation that includes @RequestMapping(method = RequestMethod.GET) as a meta-annotation. The shortcut annotations are more concise, readable, and reduce boilerplate. Spring recommends using the shortcut annotations.
Q11: How do you return a 201 Created response with a location header?
Use ResponseEntity.created(URI.create("/api/products/" + id)).body(response). The .created() method sets the status to 201 and the Location header to the provided URI. Alternatively, use ResponseEntity.status(HttpStatus.CREATED).body(response) if you don't need the Location header.
Q12: What is Pageable and how does Spring Data Web Support work?
Pageable is an interface that encapsulates pagination (page number, size) and sorting. Spring Data Web Support (enabled by default) automatically resolves Pageable from query parameters (?page=0&size=20&sort=name,asc). @PageableDefault sets defaults. The return type Page includes content plus metadata (total elements, total pages, etc.).
Q13: Why do we separate controllers, services, and repositories into layers?
Separation of concerns: controllers handle HTTP concerns, services handle business logic and transactions, repositories handle data access. This makes the code testable (mock layers independently), maintainable (change one layer without affecting others), and follows the Single Responsibility Principle.
Q14: How would you handle file upload in a REST controller?
Use @RequestParam("file") MultipartFile file. Spring's MultipartResolver handles multipart/form-data parsing. The file can be saved to disk, cloud storage (S3), or converted to bytes. Add @Max(5MB) for size validation. The controller should return a URL or file ID for subsequent access.
Q15: What is the @CrossOrigin annotation and when is it needed?
@CrossOrigin enables CORS (Cross-Origin Resource Sharing) on a controller or endpoint. It's needed when a frontend (e.g., React at localhost:3000) makes requests to a different origin (backend at localhost:8080). Without it, browsers block cross-origin requests. Can be configured globally in SecurityConfig or per-controller.
