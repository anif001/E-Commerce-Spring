Spring Data JPA Repository Layer — Senior-Level Interview Deep Dive

1. Spring Data JPA Overview

Spring Data JPA is a subproject of Spring Data that provides repository support for the Jakarta Persistence API (JPA). Its core promise: you define interfaces with method signatures, and Spring provides the implementation at runtime. No boilerplate DAO classes, no manual EntityManager plumbing for standard operations.

How It Works Without Written Implementations

When the Spring container starts, it scans for interfaces extending repository markers (e.g., JpaRepository) and, for each, dynamically creates a JDK dynamic proxy. The proxy intercepts every method call and routes it through Spring Data's RepositoryFactorySupport, which decides how to handle the invocation:

  • Predefined CRUD methods (e.g., save(), findById(), delete()) are forwarded to the default implementation class: SimpleJpaRepository.
  • Derived query methods (e.g., findByEmail()) are parsed by the PartTree parser, which decomposes the method name into subject and predicate parts, then builds a CriteriaQuery via the QueryExecutorMethodInterceptor.
  • Methods annotated with @Query have their JPQL or SQL string extracted and executed through the EntityManager at invocation time.

Repository Hierarchy

2. Repository by Repository

2.1 UserRepository

UserRepository.java package com.ecommerce.repository; import com.ecommerce.enums.Role; import com.ecommerce.model.User; import org.springframework.data.jpa.repository.JpaRepository; import java.util.List; import java.util.Optional; public interface UserRepository extends JpaRepository<User, Long> { Optional<User> findByEmail(String email); boolean existsByEmail(String email); List<User> findByRole(Role role); }

Optional<User> findByEmail(String email)

Derived query. Spring Data parses findByEmail as: "find entities WHERE email = ?1". The parser splits on the findBy prefix, extracts Email as the property name, and builds SELECT u FROM User u WHERE u.email = ?1. Returns Optional to force the caller to handle the absent case explicitly rather than risking null dereferences.

boolean existsByEmail(String email)

Exists derived query. The existsBy subject keyword tells Spring Data to generate SELECT COUNT(1) FROM User u WHERE u.email = ?1 and return true if the count > 0. This is more efficient than findByEmail because it only fetches a scalar count, not the full entity graph.

List<User> findByRole(Role role)

Derived query with enum parameter. Translates to SELECT u FROM User u WHERE u.role = ?1. Since Role is an @Enumerated(EnumType.STRING), Hibernate binds the enum's name() string value. Multiple results return a List.

2.2 ProductRepository

ProductRepository.java package com.ecommerce.repository; import com.ecommerce.model.Product; import org.springframework.data.domain.Page; import org.springframework.data.domain.Pageable; import org.springframework.data.jpa.repository.JpaRepository; import org.springframework.data.jpa.repository.Query; import org.springframework.data.repository.query.Param; import java.util.List; public interface ProductRepository extends JpaRepository<Product, Long> { Page<Product> findByCategoryId(Long categoryId, Pageable pageable); Page<Product> findBySellerId(Long sellerId, Pageable pageable); Page<Product> findByNameContainingIgnoreCase(String name, Pageable pageable); @Query("SELECT p FROM Product p WHERE " + "LOWER(p.name) LIKE LOWER(CONCAT('%', :keyword, '%')) OR " + "LOWER(p.description) LIKE LOWER(CONCAT('%', :keyword, '%'))") Page<Product> searchByKeyword(@Param("keyword") String keyword, Pageable pageable); @Query("SELECT p FROM Product p WHERE p.category.id IN :categoryIds") Page<Product> findByCategoryIds(@Param("categoryIds") List<Long> categoryIds, Pageable pageable); @Query("SELECT p FROM Product p WHERE p.stock < :threshold ORDER BY p.stock ASC") List<Product> findLowStockProducts(@Param("threshold") int threshold); List<Product> findByStockLessThanEqual(int stock); }

Page<Product> findByCategoryId(Long categoryId, Pageable pageable)

Derived pagination query. Translates to SELECT p FROM Product p WHERE p.category.id = ?1. Because the return type is Page, Spring Data automatically generates a COUNT query: SELECT COUNT(p) FROM Product p WHERE p.category.id = ?1. The Pageable parameter controls offset (page), limit (size), and sort order.

Page<Product> findBySellerId(Long sellerId, Pageable pageable)

Same pattern as above. Product.seller is a @ManyToOne(fetch = FetchType.LAZY) to User. The derived query navigates seller.id via dot notation in the path expression.

Page<Product> findByNameContainingIgnoreCase(String name, Pageable pageable)

Derived query with multiple criteria keywords. Containing translates to LIKE %:name% (wrapping the parameter in %). IgnoreCase applies LOWER() to both the column and the parameter. Resulting JPQL: SELECT p FROM Product p WHERE LOWER(p.name) LIKE LOWER(CONCAT('%', ?1, '%')).

@Query("SELECT p FROM Product p WHERE LOWER(p.name) LIKE LOWER(CONCAT('%', :keyword, '%')) OR LOWER(p.description) LIKE LOWER(CONCAT('%', :keyword, '%'))") Page<Product> searchByKeyword(@Param("keyword") String keyword, Pageable pageable)

Custom JPQL query with @Query. A derived method cannot express OR across two different columns, so we write JPQL explicitly. @Param("keyword") binds the named parameter :keyword. The CONCAT('%', :keyword, '%') builds the LIKE pattern in JPQL. The Pageable parameter is intercepted by Spring Data, which wraps the JPQL in a count query (SELECT COUNT(p) FROM Product p WHERE ...) and applies ORDER BY and OFFSET/LIMIT based on the Pageable values. This is JPQL (database-agnostic, validated at boot time), not native SQL.

@Query("SELECT p FROM Product p WHERE p.category.id IN :categoryIds") Page<Product> findByCategoryIds(@Param("categoryIds") List<Long> categoryIds, Pageable pageable)

Demonstrates the IN clause with a List parameter. Spring Data expands the collection into a comma-separated parameter list. Again, JPQL with pagination. A derived method would need findByCategoryIdIn(List<Long> ids, Pageable pageable) — both are valid; the @Query version makes the SQL explicit for readability.

@Query("SELECT p FROM Product p WHERE p.stock < :threshold ORDER BY p.stock ASC") List<Product> findLowStockProducts(@Param("threshold") int threshold)

Custom JPQL returning List (no pagination). Explicit ORDER BY inside JPQL. The equivalent derived method would be findByStockLessThan(int stock, Sort sort) but the @Query annotation makes the ordering self-documenting and guarantees it regardless of caller.

List<Product> findByStockLessThanEqual(int stock)

Derived query with LessThanEqual. Translates to SELECT p FROM Product p WHERE p.stock <= ?1. The criteria keyword LessThanEqual is mapped to the JPQL <= operator.

2.3 CategoryRepository

CategoryRepository.java package com.ecommerce.repository; import com.ecommerce.model.Category; import org.springframework.data.jpa.repository.JpaRepository; import java.util.List; import java.util.Optional; public interface CategoryRepository extends JpaRepository<Category, Long> { Optional<Category> findByName(String name); List<Category> findByParentIsNull(); List<Category> findByParentId(Long parentId); boolean existsByName(String name); }

Optional<Category> findByName(String name)

Simple derived query: WHERE name = ?1. Optional return because name has unique = true constraint — at most one result.

List<Category> findByParentIsNull()

Derived query with IsNull keyword. Translates to SELECT c FROM Category c WHERE c.parent IS NULL. Returns root-level categories (those without a parent). No parameters needed.

List<Category> findByParentId(Long parentId)

Derived query: WHERE c.parent.id = ?1. Returns immediate child categories of a given parent.

boolean existsByName(String name)

Optimized existence check using COUNT, avoiding full entity loading. Used before creating a category to enforce uniqueness at the application level.

2.4 CartRepository

CartRepository.java package com.ecommerce.repository; import com.ecommerce.model.Cart; import org.springframework.data.jpa.repository.JpaRepository; import org.springframework.data.jpa.repository.Query; import org.springframework.data.repository.query.Param; import java.util.Optional; public interface CartRepository extends JpaRepository<Cart, Long> { Optional<Cart> findByUserId(Long userId); @Query("SELECT c FROM Cart c JOIN FETCH c.items WHERE c.user.id = :userId") Optional<Cart> findWithItemsByUserId(@Param("userId") Long userId); void deleteByUserId(Long userId); }

Optional<Cart> findByUserId(Long userId)

Derived query: WHERE c.user.id = ?1. Returns the user's cart without the items collection loaded (lazy). The Cart.items list is @OneToMany(fetch = FetchType.LAZY), so accessing items outside a transaction would throw LazyInitializationException.

@Query("SELECT c FROM Cart c JOIN FETCH c.items WHERE c.user.id = :userId") Optional<Cart> findWithItemsByUserId(@Param("userId") Long userId)

JOIN FETCH for eager loading in a single query. JOIN FETCH is a JPA/JPQL construct that instructs the persistence provider to eagerly fetch the associated collection in the same SELECT statement using a SQL LEFT OUTER JOIN (or INNER JOIN for non-optional associations). Without JOIN FETCH, accessing cart.getItems() would trigger a separate SQL query for each cart (the N+1 problem). With it, a single SQL query fetches both the cart and all its items: SELECT c.*, ci.* FROM carts c LEFT JOIN cart_items ci ON ci.cart_id = c.id WHERE c.user_id = ?. This is the go-to solution for the N+1 problem in read operations.

void deleteByUserId(Long userId)

Derived delete query. The deleteBy prefix triggers a derived delete, which Spring Data translates to a DELETE FROM cart WHERE user_id = ? JPQL bulk delete. By default, derived delete methods are @Transactional and require @Modifying semantics internally. Notice: no @Modifying annotation is present — Spring Data automatically applies it for derived deleteBy methods (it detects the deleteBy prefix and wraps the execution in a modifying query).

2.5 OrderRepository

OrderRepository.java package com.ecommerce.repository; import com.ecommerce.enums.OrderStatus; import com.ecommerce.model.Order; import org.springframework.data.domain.Page; import org.springframework.data.domain.Pageable; import org.springframework.data.jpa.repository.JpaRepository; import org.springframework.data.jpa.repository.Query; import org.springframework.data.repository.query.Param; import java.time.LocalDateTime; import java.util.List; public interface OrderRepository extends JpaRepository<Order, Long> { Page<Order> findByUserId(Long userId, Pageable pageable); Page<Order> findByOrderStatus(OrderStatus status, Pageable pageable); @Query("SELECT o FROM Order o WHERE o.orderStatus = :status AND o.createdAt < :since") List<Order> findStaleOrders(@Param("status") OrderStatus status, @Param("since") LocalDateTime since); @Query("SELECT COUNT(o) FROM Order o WHERE o.user.id = :userId AND o.orderStatus = :status") long countByUserAndStatus(@Param("userId") Long userId, @Param("status") OrderStatus status); @Query("SELECT COALESCE(SUM(o.totalAmount), 0) FROM Order o WHERE " + "o.orderStatus = :status AND o.createdAt BETWEEN :start AND :end") Double getTotalRevenueBetween(@Param("status") OrderStatus status, @Param("start") LocalDateTime start, @Param("end") LocalDateTime end); }

Page<Order> findByUserId(Long userId, Pageable pageable)

Derived pagination for user's order history. Translates to SELECT o FROM Order o WHERE o.user.id = ?1. Spring Data appends ORDER BY and pagination from the Pageable parameter, and generates a count query automatically.

Page<Order> findByOrderStatus(OrderStatus status, Pageable pageable)

Same pagination pattern filtering by OrderStatus enum. Used by admin dashboards to page through orders in a given state (e.g., PENDING, SHIPPED).

@Query("SELECT o FROM Order o WHERE o.orderStatus = :status AND o.createdAt < :since") List<Order> findStaleOrders(@Param("status") OrderStatus status, @Param("since") LocalDateTime since)

Custom JPQL with two named parameters. Finds orders that have been in a given status longer than a threshold. Used by a scheduled job to auto-cancel pending orders after 24 hours. Returns List (no pagination).

@Query("SELECT COUNT(o) FROM Order o WHERE o.user.id = :userId AND o.orderStatus = :status") long countByUserAndStatus(...)

Aggregate query returning a scalar. COUNT(o) returns a Long. The equivalent derived method would be long countByUserIdAndOrderStatus(Long userId, OrderStatus status) — but the @Query version makes the intent explicit and consistent with the other custom queries.

@Query("SELECT COALESCE(SUM(o.totalAmount), 0) FROM Order o WHERE o.orderStatus = :status AND o.createdAt BETWEEN :start AND :end") Double getTotalRevenueBetween(...)

Aggregate with SUM and COALESCE. SUM(o.totalAmount) aggregates a BigDecimal column. COALESCE(..., 0) ensures the query returns 0.0 instead of null when no matching orders exist. BETWEEN :start AND :end is inclusive on both bounds. This query cannot be expressed as a derived method and is a canonical example of when @Query is required.

3. Derived Query Methods Deep Dive

Spring Data JPA parses repository method names into JPQL queries at startup using the PartTree parser (org.springframework.data.repository.query.parser.PartTree). The algorithm:

  1. Strip the subject keyword (e.g., findBy, existsBy, countBy)
  2. Parse the remainder into predicate parts separated by And / Or
  3. Each part is split into a property path and optional criteria keywords
  4. Build a CriteriaQuery using JPA Criteria API internally

How parsing works internally: At application startup, JpaQueryLookupStrategy resolves each method. For derived methods, PartTreeJpaQuery is created, which uses PartTree to decompose the name. Each Part contains a PropertyPath (resolved against the entity's metamodel) and a Type (the operator, e.g., LESS_THAN, CONTAINING, IS_NULL). The tree is then rendered into JPQL via CriteriaQuery.

4. @Query Annotations

The @Query annotation allows you to define the JPQL (or native SQL) directly on the method. It is resolved at startup — invalid JPQL causes an ApplicationContext failure.

JPQL vs Native SQL

// JPQL (database-agnostic, entity-based) @Query("SELECT p FROM Product p WHERE p.price > :minPrice") List<Product> findByMinPrice(@Param("minPrice") BigDecimal minPrice); // Native SQL (database-specific, table-based) @Query(value = "SELECT * FROM products p WHERE p.price > :minPrice", nativeQuery = true) List<Product> findByMinPriceNative(@Param("minPrice") BigDecimal minPrice);

This project uses JPQL exclusively. Reasons: database portability (H2 for tests, PostgreSQL for prod), compile-time validation by Hibernate, and automatic entity graph management (dirty checking, cascading).

Named Parameters with @Param

@Param("name") binds the method parameter to the JPQL named parameter :name. Spring Data also supports positional parameters (?1, ?2), but named parameters are preferred because:

  • Explicit binding — order-independent
  • Self-documenting — the parameter name in JPQL matches the business concept
  • Safe under refactoring — reordering method parameters doesn't break queries

Alternatives to @Query

AlternativeWhen to Use
@NamedQuery (on entity)Reusable named JPQL defined once on the entity class; invoked by entityManager.createNamedQuery(). Less common in modern Spring Data.
Specification + JpaSpecificationExecutorDynamic, type-safe, composable WHERE clauses built programmatically. Best for multi-filter search UIs with optional, combinable criteria.
QueryDSLFluent, compile-time-safe query builder. Requires annotation processing. Powerful but adds build complexity.
@EntityGraphDeclarative fetch strategy to solve N+1 without writing JOIN FETCH in JPQL. Define named entity graphs on the entity, reference them in the repository.

5. Pagination & Sorting

How Pageable Works

org.springframework.data.domain.Pageable is an interface that bundles pagination parameters: page number (0-indexed), page size, and sort. Created via the PageRequest factory:

Pageable pageable = PageRequest.of(0, 20, Sort.by("createdAt").descending()); // page 0 (first page), 20 items per page, newest first

When a repository method accepts Pageable and returns Page, Spring Data:

  1. Applies ORDER BY from Pageable.getSort() to the query
  2. Applies database-level pagination (e.g., LIMIT ? OFFSET ? for PostgreSQL)
  3. Generates and executes a count query to determine total elements
  4. Wraps results in a PageImpl with content, total count, page metadata

What Page<T> Contains

getTotalElements()longTotal matching records across all pages
getTotalPages()intTotal number of pages (ceiling of totalElements / size)
getNumber()intCurrent page number (0-indexed)
getSize()intNumber of items per page

Page vs List

Returning List from a paginated repository method still applies pagination (LIMIT/OFFSET) to the data query, but skips the count query. Use List when you don't need total count (e.g., "fetch 5 most recent orders" with PageRequest.of(0, 5)). Use Page when building UI pagination controls that need total pages.

6. Transaction Management

@Transactional with Repositories

By default, SimpleJpaRepository marks all CRUD methods with @Transactional:

  • findById(), findAll(), count() → @Transactional(readOnly = true)
  • save(), delete(), saveAll() → @Transactional

Custom query methods in your interfaces inherit transactional behavior from the caller (typically the service layer). If no @Transactional is present on the calling service method:

  • Derived find queries: executed within a read-only transaction opened automatically (Spring Data wraps them)
  • @Query with @Modifying: must be invoked within a transaction, otherwise javax.persistence.TransactionRequiredException

Read-Only Transactions

@Transactional(readOnly = true) public List<Product> getLowStockProducts(int threshold) { return productRepository.findLowStockProducts(threshold); }

readOnly = true is an optimization hint to Hibernate:

  • Disables dirty checking (no snapshot comparison for managed entities)
  • Disables auto-flush before query execution (avoids unnecessary writes)
  • Allows the JDBC driver to potentially use read-only connections (if supported)
  • Does not prevent writes programmatically (use a DB user with read-only grants for enforcement)

HikariCP Connection Pooling & Transactions

HikariCP serves as the connection pool between the application and the database. Here is the transaction-connection lifecycle:

  1. When @Transactional starts, Spring's TransactionInterceptor calls DataSourceUtils.getConnection(), which retrieves a connection from HikariCP's pool.
  2. The connection is bound to the current thread (via TransactionSynchronizationManager).
  3. All JPA operations within the transaction reuse the same connection.
  4. On commit/rollback, the connection is returned to the pool.

If the pool is exhausted, threads block on connectionTimeout (20,000 ms in our config). After timeout, SQLTransientConnectionException is thrown.

The N+1 Problem

The problem: When you fetch a list of entities (e.g., 10 products) and then iterate over each to access a lazy association (e.g., product.getCategory().getName()), Hibernate executes: 

    1 query:  SELECT * FROM products
    + 10 queries: SELECT * FROM categories WHERE id = ?  (one per product)
    = 11 total queries (N+1)

The solutions used in this project:

JOIN FETCH c.items		Adds a SQL JOIN that loads the association in the same query. The fetched collection is initialized before the EntityManager closes.

8. Interview Q&A

Q1: How does Spring Data JPA create repository implementations without us writing any implementation class?
A: During context initialization, @EnableJpaRepositories triggers JpaRepositoryFactoryBean to scan for interfaces extending Repository. For each interface, a JDK dynamic proxy is created via ProxyFactory. The proxy's InvocationHandler is QueryExecutorMethodInterceptor, which maintains a map of method → RepositoryQuery. The query can be:
  • A CRUD method delegated to SimpleJpaRepository (the default implementation)
  • A derived query parsed by PartTreeJpaQuery
  • A @Query method resolved by SimpleJpaQuery
All this happens at startup, so there is zero runtime overhead for query resolution.
Q2: What is the difference between JPQL and native SQL in @Query?
A: JPQL (Java Persistence Query Language) operates on the entity model — it uses entity names and field names (e.g., SELECT p FROM Product p WHERE p.price > :min). It is database-agnostic and validated at startup by Hibernate. Native SQL operates on actual table and column names (e.g., SELECT * FROM products WHERE price > :min). It is database-specific, not validated at startup, and results are mapped to entities only if the result columns match entity mappings. Use JPQL by default; reserve native SQL for database-specific features (e.g., PostgreSQL JSONB operators, ON CONFLICT, full-text search).
Q3: How does the @Query annotation work under the hood?
A: At startup, JpaQueryLookupStrategy detects the @Query annotation and wraps it in a SimpleJpaQuery object. This object stores the JPQL string, the EntityManager, and query metadata. At invocation time, it calls entityManager.createQuery(jpqlString), binds method arguments to named/positional parameters via Query.setParameter(), executes the query, and maps the result set to entities. For Page returns, it also creates and executes a count query (wrapping the JPQL in SELECT COUNT(*) or using an explicit countQuery attribute). For @Modifying queries, it calls query.executeUpdate() and clears the persistence context if clearAutomatically = true.
Q4: What is the N+1 problem and how do you solve it in Spring Data JPA?
A: The N+1 problem occurs when you fetch N entities and then access a lazy association on each one, generating N additional SQL queries (total N+1). For example, fetching 100 products and iterating to print product.getCategory().getName() generates 1 query for products + 100 queries for categories. Solutions: (1) JOIN FETCH in JPQL — loads the association in the same query via a SQL JOIN. (2) @EntityGraph — declarative fetch plan, equivalent to JOIN FETCH without writing JPQL. (3) @BatchSize — reduces N+1 to ceil(N/batchSize)+1 queries. (4) QueryDSL / Specs with explicit fetches. The best solution for read-heavy paths is JOIN FETCH or @EntityGraph.
Q5: How does Pageable work internally?
A: Pageable is an interface with getPageNumber(), getPageSize(), and getSort(). PageRequest is the standard implementation. When a repository method receives Pageable:
  1. Spring Data applies Sort from pageable.getSort() to the query's ORDER BY clause
  2. It applies database-specific pagination: PostgreSQL uses LIMIT size OFFSET page * size
  3. For Page return types, it generates a count query by wrapping the original JPQL in SELECT COUNT(entity), preserving WHERE clauses. The count query can be overridden with @Query(countQuery = "...") if the auto-generated one is inefficient
  4. Results are wrapped in PageImpl(content, pageable, totalElements)
Q6: What is the difference between JOIN and JOIN FETCH in JPQL?
A: JOIN creates an inner join in SQL but does not guarantee that the associated entity is loaded into the persistence context. The association remains lazy — accessing it outside a transaction still causes LazyInitializationException. The JOIN only affects filtering. JOIN FETCH does the same SQL join but also initializes the association, loading the related data into the persistence context immediately. The fetched data is available even after the EntityManager closes (for DTOs) or before proxy access. In short: JOIN = filter only; JOIN FETCH = filter + load.
Q7: How does Spring Data derive queries from method names, and what are the limitations?
A: Spring Data uses the PartTree parser to decompose the method name. It first extracts the subject keyword (findBy, existsBy, countBy, deleteBy, etc.) and then parses the predicate split by And/Or. Each predicate part is decomposed into a property path (e.g., CategoryId → category.id) and optional criteria keywords (Containing, LessThan, IgnoreCase, etc.). Limitations: Cannot express OR across different columns in a single derived method (e.g., search by name OR description). Cannot use JOIN FETCH, subqueries, aggregate functions (SUM, AVG), GROUP BY, HAVING, or UNION. For these, use @Query.
Q8: Why does @Modifying require @Transactional, and what does clearAutomatically do?
A: @Modifying indicates that the query is a DML operation (INSERT, UPDATE, DELETE). JPA requires that DML operations (executeUpdate()) are executed within a transaction. Without @Transactional, javax.persistence.TransactionRequiredException is thrown. clearAutomatically = true (default false) clears the persistence context after the query executes, preventing stale entity state from being used later. For example, after a bulk UPDATE that modifies entities already in the session, the cache should be cleared so subsequent reads fetch fresh data from the database.
Q9: What is a JPA Specification and when would you use it over @Query?
A: A Specification is a functional interface from Spring Data JPA that wraps JPA Criteria API predicates. Your repository extends JpaSpecificationExecutor, and you pass Specification objects to query methods like findAll(Specification, Pageable). Use Specifications when: you need dynamic, composable WHERE clauses at runtime — for example, a product search with optional filters (categories, price range, rating, seller). Each filter is a Specification, and they are combined with where(), and(), or(). Use @Query when: the query is static and known at compile time. Specifications avoid string concatenation and are type-safe, but are more verbose for simple queries.
Q10: How does HikariCP connection pooling work, and how do the pool settings affect application behavior?
A: HikariCP maintains a pool of physical JDBC connections to the database. Architecture: It uses a ConcurrentBag for lock-free connection borrowing, a FastList for collection efficiency, and a custom ThreadFactory for pool management. Behavior: When a thread requests a connection: (1) if idle connections exist, one is borrowed from the bag; (2) if not, a new connection is created (up to maximumPoolSize); (3) if all are in use, the thread waits up to connectionTimeout ms. Under load spikes, with maximumPoolSize=10, threads will queue. If the queue exceeds 20s, the request fails. Key insight: Pool size should NOT match application thread count. PostgreSQL is a process-per-connection database — too many connections causes context switching overhead and memory pressure. The formula poolSize = T * (C - 1) + 1 (T = thread count, C = max connections per thread) from the HikariCP wiki suggests most applications are optimal with poolSize = 10 + (activeWorkers * 0.5). Our value of 10 is reasonable for a moderate-traffic e-commerce backend.
