1. Entity Design Principles

What Is a JPA Entity?

A JPA entity is a plain Java class annotated with @Entity that is managed by the persistence context (Hibernate). Each entity instance corresponds to a row in a database table. The entity's fields (or properties) map to table columns.

Why Entities Exist

Entities bridge the object-oriented domain model with the relational database (ORM). Instead of writing raw JDBC and hand-crafting SQL, entities let us work with objects while Hibernate generates the SQL. This gives us:

  • Abstraction — work with Java objects, not ResultSet rows
  • Automatic dirty checking — Hibernate detects which fields changed and issues minimal UPDATE statements
  • Cascading operations — persist/remove a parent and its children in one call
  • Lazy loading — fetch only what you need, when you need it (via proxies)
  • Identity management — the @Id and persistence context guarantee at-most-one-instance-per-row within a session

2.1 User

@Entity
@Table(name = "users")
@Getter @Setter @NoArgsConstructor @AllArgsConstructor @Builder
public class User {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false)
    private String name;

    @Column(nullable = false, unique = true)
    private String email;

    @Column(nullable = false)
    private String password;

    @Enumerated(EnumType.STRING)
    @Column(nullable = false)
    private Role role;

    private String phone;

    private String address;

    @Column(updatable = false)
    private LocalDateTime createdAt;

    private LocalDateTime updatedAt;

    @PrePersist
    protected void onCreate() {
        createdAt = LocalDateTime.now();
        updatedAt = LocalDateTime.now();
    }

    @PreUpdate
    protected void onUpdate() {
        updatedAt = LocalDateTime.now();
    }
}

Annotation Deep-Dive

@Entity — Marks this class as a JPA entity. Hibernate's EntityManagerFactory scans for @Entity classes during bootstrap and registers them in the Metamodel. Without it, the class is just a POJO. Internally, Hibernate creates a EntityPersister per entity class which knows how to generate CRUD SQL and manage the persistence lifecycle. An entity must have a no-arg constructor (public or protected).

@Table(name = "users") — Maps the entity to the users table. Without @Table, Hibernate defaults the table name to the entity class name (User). We customize it because user is a reserved word in PostgreSQL (and SQL-92). The plural users avoids keyword conflicts and follows naming conventions. @Table also accepts schema, catalog, uniqueConstraints, and indexes.

@Id + @GeneratedValue(strategy = GenerationType.IDENTITY) — @Id marks the primary key field. Every entity must have an @Id. @GeneratedValue tells Hibernate how to generate PK values:

StrategyHow It WorksPostgreSQLProsCons
IDENTITYRelies on DB auto-increment (SERIAL / IDENTITY columns)BIGSERIAL / GENERATED BY DEFAULT AS IDENTITYSimple, no sequence call before insertRequires immediate insert (disables JDBC batch inserts, no persist() before PK known)
SEQUENCECalls a DB sequence (e.g., hibernate_sequence) for each new IDCREATE SEQUENCE — native PostgreSQLSupports batch inserts, ALLOCATION_SIZE optimizationExtra round-trip per batch (mitigated by allocationSize)
TABLEEmulates sequence with a DB table (pessimistic)Works but suboptimalDB-portableSlow, pessimistic locking, never use in production
AUTOHibernate picks (often SEQUENCE for PostgreSQL)Delegates to Hibernate's dialectConvenientNot explicit; may surprise you when DB changes

We chose IDENTITY because it maps directly to PostgreSQL's BIGSERIAL / identity columns and is the simplest strategy. The trade-off is that Hibernate cannot use JDBC batch inserts for IDENTITY (it must insert immediately to get the generated key). For an e-commerce backend where writes are moderate, this is acceptable. For high-volume write systems, SEQUENCE with allocationSize=50 is preferred.

@Column(nullable = false, ...) — Configures the DB column:

  • nullable = false — Adds NOT NULL constraint at schema generation. Also triggers JPA validation before flush (if validation is enabled).
  • unique = true — Adds a UNIQUE constraint. Equivalent to @Table(uniqueConstraints = ...) but inline.
  • columnDefinition = "TEXT" — Overrides the default SQL type. By default Hibernate maps String to VARCHAR(255). For long text, we override to TEXT.
  • precision = 10, scale = 2 — For decimal columns. precision = total digits, scale = digits after decimal. Used with BigDecimal. Maps to NUMERIC(10,2).
  • updatable = false — Excludes this column from UPDATE statements. Useful for createdAt.

@Enumerated(EnumType.STRING) — Stores the enum as a string in the DB. EnumType.ORDINAL stores the integer ordinal() (0, 1, 2...). NEVER use ORDINAL in production. If someone reorders enum constants (e.g., adds SUPER_ADMIN before ADMIN), all existing ORDINAL data corrupts silently. STRING stores the literal name (e.g., "CUSTOMER"), which survives reordering. The cost is slightly more storage (a VARCHAR vs an integer) — a trivial trade for data integrity.

@PrePersist / @PreUpdate — JPA lifecycle callback annotations:

  • @PrePersist — Fires before EntityManager.persist() or em.flush() for a new entity. Used here to set createdAt and updatedAt.
  • @PreUpdate — Fires before the UPDATE statement is generated for a managed entity.

Alternatives: @EntityListeners(AuditingEntityListener.class) with Spring Data JPA's @CreatedDate/@LastModifiedDate, which is declarative and reusable. Our @PrePersist/@PreUpdate approach is explicit and self-contained; both are valid. The callback method must return void, take no parameters, and be protected or public.

Lombok @Builder — Generates a builder pattern. Works fine with JPA as long as a no-arg constructor exists (@NoArgsConstructor). Hibernate needs the no-arg constructor to create proxy instances for lazy loading. Note: @Builder sets all fields to their Java defaults (null, 0, false), which is why @Builder.Default is needed for collections and fields with default values.

2.2 Product

@Entity
@Table(name = "products")
@Getter @Setter @NoArgsConstructor @AllArgsConstructor @Builder
public class Product {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false)
    private String name;

    @Column(columnDefinition = "TEXT")
    private String description;

    @Column(nullable = false, precision = 10, scale = 2)
    private BigDecimal price;

    @Column(nullable = false)
    private Integer stock;

    private String imageUrl;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "category_id", nullable = false)
    private Category category;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "seller_id", nullable = false)
    private User seller;

    @Column(updatable = false)
    private LocalDateTime createdAt;

    private LocalDateTime updatedAt;

    @PrePersist
    protected void onCreate() { createdAt = LocalDateTime.now(); updatedAt = LocalDateTime.now(); }

    @PreUpdate
    protected void onUpdate() { updatedAt = LocalDateTime.now(); }
}

Annotation Deep-Dive

@ManyToOne(fetch = FetchType.LAZY) — Defines a many-to-one relationship: many products belong to one category/one seller.

  • Why LAZY? — Lazy loading defers fetching the associated entity until it is accessed. With FetchType.EAGER, Hibernate automatically fetches the association via a JOIN or separate SELECT every time the owning entity is loaded, even if you don't need it. This causes:
    • Performance waste — Loading 100 products would also fetch 100 categories and 100 sellers, even if you only need product names.
    • N+1 problem — (See Q&A) — If you query a list and EAGER uses a separate SELECT per row, you get N+1 queries.
    • Cartesian products — Multiple EAGER collections on the same entity produce massive cross-joins.
  • What LAZY does internally: Hibernate wraps the associated entity in a proxy object (a dynamically generated subclass via javassist or ByteBuddy). The proxy has only the @Id value initialized. On the first method call (e.g., getCategory().getName()), the proxy triggers a SELECT to load the full entity from the database. This only works within an active @Transactional session; outside, you get a LazyInitializationException.
  • Why not EAGER? — Spring/Data JPA expert Vlad Mihalcea and the Hibernate team explicitly recommend always using LAZY. EAGER is an anti-pattern in nearly all cases; it violates the principle of least surprise and forces unnecessary queries on every read.

@JoinColumn(name = "category_id", nullable = false) — Configures the foreign key column:

  • name — The column name in the products table that references the parent table's PK.
  • nullable = false — Every product must have a category and a seller. This is a DB-level constraint.
  • Without @JoinColumn, Hibernate generates a default name (e.g., category_id from category field + _id).

@Column(columnDefinition = "TEXT") — Overrides default VARCHAR(255) for the description field. Since product descriptions can be paragraphs long, TEXT is appropriate. This is database-specific; Hibernate passes the string verbatim in the DDL.

Generated Table

CREATE TABLE products (
    id          BIGSERIAL       PRIMARY KEY,
    name        VARCHAR(255)    NOT NULL,
    description TEXT,
    price       NUMERIC(10,2)   NOT NULL,
    stock       INTEGER         NOT NULL,
    image_url   VARCHAR(255),
    category_id BIGINT          NOT NULL REFERENCES categories(id),
    seller_id   BIGINT          NOT NULL REFERENCES users(id),
    created_at  TIMESTAMP       NOT NULL,
    updated_at  TIMESTAMP       NOT NULL
);

2.3 Category

@Entity
@Table(name = "categories")
@Getter @Setter @NoArgsConstructor @AllArgsConstructor @Builder
public class Category {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false, unique = true)
    private String name;

    private String description;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "parent_id")
    @JsonIgnoreProperties({"hibernateLazyInitializer", "handler"})
    private Category parent;

    @OneToMany(mappedBy = "parent", cascade = CascadeType.ALL, orphanRemoval = true)
    @Builder.Default
    @JsonIgnoreProperties({"parent"})
    private List subCategories = new ArrayList<>();

    public void addSubCategory(Category subCategory) {
        subCategories.add(subCategory);
        subCategory.setParent(this);
    }
}

Annotation Deep-Dive

Self-referencing @ManyToOne / @OneToMany — Category references itself to create a tree hierarchy (adjacency list model). The parent field maps to parent_id in the same table. Each category can have one parent (null = root) and many children. This is a single-table hierarchy.

@JsonIgnoreProperties({"hibernateLazyInitializer", "handler"}) — This is critical for JSON serialization:

  • Hibernate's LAZY proxy objects contain extra fields: hibernateLazyInitializer and handler. Without @JsonIgnoreProperties, Jackson attempts to serialize these Hibernate-internal fields and fails or causes infinite recursion.
  • On the subCategories field, @JsonIgnoreProperties({"parent"}) breaks the circular serialization: when serializing a subCategory, Jackson omits the parent field, preventing infinite recursion back up the tree.

@Builder.Default — Lombok's @Builder sets all fields to default values (null for objects). The @Builder.Default annotation tells Lombok to use the field initializer (new ArrayList<>()) as the builder's default. Without this, the builder would produce a Category with subCategories = null, causing NullPointerException in addSubCategory().

cascade = CascadeType.ALL, orphanRemoval = true — See detailed explanation below. In short: any operation on a Category cascades to its subCategories (persist, merge, remove). Orphan removal deletes any subCategory removed from the list.

addSubCategory() helper — This is the bidirectional relationship management helper. It ensures both sides of the relationship are consistent: it adds to the subCategories collection AND sets the child's parent. Without this, you'd need to manually call subCategory.setParent(this) every time. Forgetting to set both sides is a common JPA pitfall that leads to null foreign keys or in-memory inconsistency.

Generated Table

CREATE TABLE categories (
    id          BIGSERIAL       PRIMARY KEY,
    name        VARCHAR(255)    NOT NULL UNIQUE,
    description VARCHAR(255),
    parent_id   BIGINT          REFERENCES categories(id)
);

2.4 Cart

@Entity
@Table(name = "carts")
@Getter @Setter @NoArgsConstructor @AllArgsConstructor @Builder
public class Cart {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @OneToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "user_id", nullable = false, unique = true)
    private User user;

    @OneToMany(mappedBy = "cart", cascade = CascadeType.ALL, orphanRemoval = true)
    @Builder.Default
    private List items = new ArrayList<>();
}

Annotation Deep-Dive

@OneToOne(fetch = FetchType.LAZY) — One user has exactly one cart. @OneToOne is semantically correct here: the relationship cardinality is 1:1. Internally it behaves similarly to @ManyToOne (a FK column is created), but it also adds a UNIQUE constraint on the FK (unique = true explicit here too). The choice of @OneToOne vs @ManyToOne is semantic: the schema is identical (FK + unique). Use @OneToOne when the domain says "exactly one" to communicate intent.

@OneToMany(mappedBy = "cart", cascade = CascadeType.ALL, orphanRemoval = true) — One cart has many items. mappedBy = "cart" indicates that the CartItem.cart field owns the relationship (the FK column is in the cart_items table). The cart table itself has no items column; the relationship is purely through the foreign key.

  • CascadeType.ALL — See table below. All operations (persist, merge, remove, refresh, detach) on Cart cascade to its CartItems.
  • orphanRemoval = true — If a CartItem is removed from the items list, Hibernate automatically deletes it from the database. Without orphanRemoval, the CartItem would remain in DB with a null cart_id (or would need manual deletion).

Generated Table

CREATE TABLE carts (
    id      BIGSERIAL PRIMARY KEY,
    user_id BIGINT    NOT NULL UNIQUE REFERENCES users(id)
);

2.5 CartItem

@Entity
@Table(name = "cart_items")
@Getter @Setter @NoArgsConstructor @AllArgsConstructor @Builder
public class CartItem {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "cart_id", nullable = false)
    private Cart cart;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "product_id", nullable = false)
    private Product product;

    @Column(nullable = false)
    private Integer quantity;

    public BigDecimal getSubtotal() {
        return product.getPrice().multiply(BigDecimal.valueOf(quantity));
    }
}

Annotation Deep-Dive

@ManyToOne on cart and product — CartItem is the owning side of both relationships. The FK columns (cart_id, product_id) reside in the cart_items table. This is the standard pattern: the "many" side holds the foreign key.

getSubtotal() as a transient method — This is a runtime calculation, not a persisted column. It's simply unitPrice * quantity. Note: for CartItem, the price comes from the current product.getPrice(). For the order (historical record), we persist unitPrice and subtotal in OrderItem to capture the price at the time of purchase.

Generated Table

CREATE TABLE cart_items (
    id         BIGSERIAL    PRIMARY KEY,
    cart_id    BIGINT       NOT NULL REFERENCES carts(id),
    product_id BIGINT       NOT NULL REFERENCES products(id),
    quantity   INTEGER      NOT NULL
);

2.6 Order

@Entity
@Table(name = "orders")
@Getter @Setter @NoArgsConstructor @AllArgsConstructor @Builder
public class Order {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "user_id", nullable = false)
    private User user;

    @Enumerated(EnumType.STRING)
    @Column(nullable = false)
    @Builder.Default
    private OrderStatus orderStatus = OrderStatus.PENDING;

    @Enumerated(EnumType.STRING)
    @Column(nullable = false)
    @Builder.Default
    private PaymentStatus paymentStatus = PaymentStatus.PENDING;

    @Column(nullable = false, precision = 10, scale = 2)
    private BigDecimal totalAmount;

    private String shippingAddress;

    private String razorpayOrderId;

    private String razorpayPaymentId;

    @OneToMany(mappedBy = "order", cascade = CascadeType.ALL, orphanRemoval = true)
    @Builder.Default
    private List orderItems = new ArrayList<>();

    @Column(updatable = false)
    private LocalDateTime createdAt;

    private LocalDateTime updatedAt;

    @PrePersist
    protected void onCreate() { createdAt = LocalDateTime.now(); updatedAt = LocalDateTime.now(); }

    @PreUpdate
    protected void onUpdate() { updatedAt = LocalDateTime.now(); }
}

Annotation Deep-Dive

@Enumerated(EnumType.STRING) x2 — Two independent enum fields: orderStatus (tracks fulfillment lifecycle) and paymentStatus (tracks payment lifecycle). They are intentionally separated because they change independently: an order can be SHIPPED before payment clears (COD), or payment can be PAID but order still PENDING. Storing them as separate enum columns avoids coupling two orthogonal state machines.

@Builder.Default on enum fields — Without @Builder.Default, Lombok's builder would set orderStatus to null, which violates the nullable = false column constraint. @Builder.Default ensures the builder initializes the field with the Java initializer value (OrderStatus.PENDING). Same for paymentStatus and the orderItems list.

Razorpay fields — razorpayOrderId and razorpayPaymentId are stored directly on the Order entity for simplicity. An alternative is a separate Payment entity supporting multiple payment gateways, but for this project's scope, inline fields are pragmatic.

2.7 OrderItem

@Entity
@Table(name = "order_items")
@Getter @Setter @NoArgsConstructor @AllArgsConstructor @Builder
public class OrderItem {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "order_id", nullable = false)
    private Order order;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "product_id", nullable = false)
    private Product product;

    @Column(nullable = false)
    private Integer quantity;

    @Column(nullable = false, precision = 10, scale = 2)
    private BigDecimal unitPrice;

    @Column(nullable = false, precision = 10, scale = 2)
    private BigDecimal subtotal;
}

Annotation Deep-Dive

unitPrice and subtotal are persisted — Unlike CartItem.getSubtotal() which is a runtime calculation, OrderItem stores the actual price and subtotal at the time of the order. This is critical for financial/audit compliance. Product prices change over time; if you recalculated order history from current prices, past orders would show wrong amounts. Storing unitPrice and subtotal freezes the financial record.

Why not join back to Product for price? — Never compute historical financial data from dynamic source data. The product's price today could be different from when the order was placed. Always snapshot the price at order time.

Generated Table

CREATE TABLE order_items (
    id         BIGSERIAL       PRIMARY KEY,
    order_id   BIGINT          NOT NULL REFERENCES orders(id),
    product_id BIGINT          NOT NULL REFERENCES products(id),
    quantity   INTEGER         NOT NULL,
    unit_price NUMERIC(10,2)   NOT NULL,
    subtotal   NUMERIC(10,2)   NOT NULL
);

3. Relationship Mapping Patterns

Uni-directional vs Bi-directional

In this project, we use bi-directional relationships for parent-child compositions (Cart-CartItem, Order-OrderItem, Category-subCategories) but uni-directional where only one side needs navigation (Product-Category, Product-User(seller)).

Rule of thumb: Use bi-directional only when you need to navigate both ways in the domain. Every bi-directional relationship adds complexity: you must keep both sides consistent. For simple lookups (e.g., "which products belong to this category?"), a repository query is cleaner than a bi-directional @OneToMany.

Owning Side vs Inverse Side

  • Owning side — The entity that contains @JoinColumn. The table with the foreign key. Example: CartItem.cart (has cart_id column).
  • Inverse side — The entity that has mappedBy. Does not have a FK column; the mapping is read-only from this side. Example: Cart.items.
  • Hibernate only checks the owning side for relationship changes. If you modify only the inverse side (e.g., cart.getItems().add(item) without setting item.setCart(cart)), Hibernate may not persist the relationship.

Helper Methods (Bidirectional Consistency)

For bi-directional relationships, always provide a helper method that sets both sides:

public void addSubCategory(Category subCategory) {
    subCategories.add(subCategory);   // inverse side
    subCategory.setParent(this);      // owning side
}

This ensures in-memory consistency. Without it, calling parent.addSubCategory(child) followed by child.getParent() would return null, leading to bugs.

Common Pitfall: Forgetting to set the owning side in a bi-directional relationship. If you do cart.getItems().add(item) but never call item.setCart(cart), Hibernate persists the CartItem with a NULL cart_id because it only looks at the owning side (CartItem.cart) for the FK value.

4. Schema Design Decisions

4.1 Why unitPrice is Stored in OrderItem

Price history / financial audit. The unitPrice column captures the product price exactly when the order was placed. Future price changes to the Product entity do not retroactively alter past orders. This is a legal/accounting requirement for any e-commerce system. Never recalculate historical financial data from live entity state.

4.2 Why Category Uses Self-Referencing ManyToOne (Adjacency List)

The adjacency list model (parent_id column) is the simplest hierarchical model in SQL. Alternatives:

  • Nested Sets — Efficient reads, expensive writes, hard to maintain.
  • Closure Table — Flexible, requires a separate table, more complex queries.
  • Materialized Path — Store full path as string, limited depth.

For an e-commerce category tree (typically < 3 levels deep, < 1000 categories), adjacency list is the right call. It maps naturally to JPA's @ManyToOne/@OneToMany and queries are efficient with a known depth.

4.3 Why Composition Pattern (Cart-CartItem, Order-OrderItem) Instead of Single Entity

A cart/order is not simply a list of product IDs. It contains metadata (user, status, timestamps, payment info). Each item has quantity and (for orders) price snapshot. Normalizing into separate entities gives:

  • Single responsibility — Cart manages lifecycle; CartItem manages quantity/price.
  • No denormalized data — No repeating groups, no JSON columns, no array columns.
  • Efficient updates — Update quantity on one item without rewriting the entire list.
  • Indexable FK queries — "Find all orders containing product X" is a simple index seek on order_items.product_id.

4.4 Why Separate orderStatus and paymentStatus Enums

Order lifecycle (PENDING, CONFIRMED, SHIPPED, DELIVERED, CANCELLED, RETURNED) and payment lifecycle (PENDING, PAID, FAILED, REFUNDED) are orthogonal state machines. They change at different times and for different reasons:

  • An order can be SHIPPED before payment is PAID (Cash on Delivery).
  • A payment can be PAID but the order remains PENDING (awaiting fulfillment).
  • A payment can be REFUNDED while the order status stays DELIVERED.

Combining them into one field (e.g., an enum with 36 combinations) would be a design smell. Two independent enums are cleaner, more extensible, and map directly to business workflows.

5. JSON Serialization

How Jackson Serializes Entities

By default, Jackson serializes all public getters (or fields, depending on visibility). When serializing a Product with a Category, Jackson calls product.getCategory(), which triggers Hibernate's LAZY proxy to load the full Category from the DB (if inside a transaction). The Category then has a parent field, which triggers another load, and so on. This creates a circular reference that causes a StackOverflowError or infinite JSON recursion.

@JsonIgnoreProperties to Break Cycles

We use @JsonIgnoreProperties at the class level to suppress specific properties during serialization:

  • @JsonIgnoreProperties({"hibernateLazyInitializer", "handler"}) — Hides Hibernate proxy internals from Jackson. Without this, Jackson may fail to serialize LAZY proxies when the session is closed.
  • @JsonIgnoreProperties({"parent"}) on subCategories — When serializing a Category, skip the parent field on each subCategory to break the circular reference.

Prefer DTOs Over Serializing Entities

The cleanest approach is to not serialize entities at all. Instead, map entities to DTOs (Data Transfer Objects) before serialization:

  • Security — Avoid accidentally exposing sensitive fields (e.g., password, razorpayOrderId). Even with @JsonIgnore, a single missing annotation leaks data.
  • Control — Shape the JSON exactly to the API contract, with no extra fields.
  • Decoupling — Changes to the entity model (field renames, new relationships) don't automatically change API responses.
  • Lazy loading safety — DTOs are plain POJOs, not Hibernate proxies. No risk of LazyInitializationException.
Recommendation: Use MapStruct or manual DTO mappers in the service layer. Reserve entity serialization for internal/debug endpoints only.

@JsonIgnore Use Cases

@JsonIgnore is appropriate for specific fields that must never be serialized:

  • User.password
  • Internal @Version fields (optimistic locking)
  • Internal IDs that shouldn't be exposed

6. Interview Q&A

Q1: What are the differences between IDENTITY, SEQUENCE, TABLE, and AUTO generation strategies? Which one do you use and why?
IDENTITY relies on the database's auto-increment column (e.g., PostgreSQL's BIGSERIAL or GENERATED BY DEFAULT AS IDENTITY). Hibernate inserts the row first, then retrieves the generated ID. This prevents JDBC batch insert optimization because Hibernate must execute the INSERT immediately to get the ID.

SEQUENCE uses a database sequence (e.g., CREATE SEQUENCE in PostgreSQL). Hibernate calls NEXTVAL before inserting, which allows batch inserts (IDs are pre-allocated via allocationSize). This is the most performant strategy for high-volume writes.

TABLE emulates a sequence using a database table with pessimistic locking. Portable but slow — never use in production.

AUTO delegates the choice to Hibernate's dialect. For PostgreSQL, it typically uses SEQUENCE.

We chose IDENTITY in this project. It's the simplest, maps directly to PostgreSQL's identity columns, and for our e-commerce write volume the batch insert limitation is acceptable. For high-throughput systems, prefer SEQUENCE.
Q2: Why do we use FetchType.LAZY instead of FetchType.EAGER? What problem does LAZY loading solve?
LAZY loading defers fetching an association until the code actually accesses it. EAGER forces Hibernate to always fetch the association (via JOIN or separate SELECT) regardless of whether it's needed.

Problems with EAGER:
(1) Unnecessary queries — Loading a list of products for a catalog page also fetches every category and seller, even if you only display names.
(2) N+1 select problem — If Hibernate uses a separate SELECT per entity for an EAGER association, querying 100 products produces 1+100 queries.
(3) Cartesian product — Multiple EAGER collections on the same entity produce cross-joins that multiply result sets.

LAZY loading uses proxy objects — dynamically generated subclasses that contain only the ID. When you access a property on the proxy, Hibernate initializes it with a SELECT. This only works within an active Hibernate session (@Transactional).

The rule: always use LAZY for all associations. If you need data eagerly, use JOIN FETCH in JPQL or @EntityGraph for explicit, query-scoped eager loading.
Q3: What is the N+1 query problem? How do you detect and fix it?
The N+1 problem occurs when a query fetches N entities and then executes N additional queries to load lazy associations for each entity.

Example: List<Product> products = productRepository.findAll(); followed by iterating product.getCategory().getName(). If category is LAZY, Hibernate executes 1 query for the products + N queries for each product's category.

Detection: Enable Hibernate SQL logging (spring.jpa.show-sql=true or logging.level.org.hibernate.SQL=DEBUG). Use a tool like datasource-proxy or p6spy to capture query counts.

Fixes:
(1) JOIN FETCH in JPQL: SELECT p FROM Product p JOIN FETCH p.category
(2) @EntityGraph: @EntityGraph(attributePaths = "category") on the repository method
(3) Criteria API with fetch()
(4) DTO projection: SELECT new ProductDTO(p.id, p.name, c.name) FROM Product p JOIN p.category c

The key insight: LAZY loading is the default (good), but you must explicitly plan for eager fetching when you know the access pattern.
Q4: Why should you use EnumType.STRING instead of EnumType.ORDINAL? What happens if you don't?
EnumType.ORDINAL stores the enum's ordinal value (0, 1, 2, ...) as an integer in the database. This is extremely fragile. If you add, remove, or reorder enum constants, the numeric values shift, and existing database rows silently become corrupted — they reference the wrong enum constant with no error.

Example: A row with ordinal 2 might mean SHIPPED today, but if you insert a new status PROCESSING between CONFIRMED and SHIPPED, that same 2 now incorrectly points to PROCESSING.

EnumType.STRING stores the constant name ("SHIPPED") as a VARCHAR. Renaming a constant is a compile-time change that requires a data migration. Adding/removing constants doesn't affect existing data. The storage difference (int vs varchar) is negligible. Always use STRING.
Q5: How does @Transactional relate to lazy loading? What is a LazyInitializationException and how do you avoid it?
@Transactional demarcates the boundaries of a Hibernate session (persistence context). LAZY proxies can only initialize while the session is open — inside the transaction.

A LazyInitializationException occurs when you access a LAZY association outside the transaction boundary. For example:
- A service method returns an entity without eagerly loading an association
- The controller (outside the transaction) calls entity.getLazyField()
- Hibernate has no open session, so the proxy cannot initialize → exception

Solutions:
(1) Keep the transaction open — Use @Transactional on the controller or use Spring's OpenSessionInViewFilter (anti-pattern, don't use in production — it holds DB connections too long).
(2) Eagerly fetch within the transaction — Use JOIN FETCH or @EntityGraph in the repository/ service method.
(3) Use DTOs — Map to DTOs inside the transaction, then work with plain POJOs outside (best practice).
(4) Spring's @Transactional(readOnly = true) on service methods for read queries ensures the session stays open for lazy loading and DB uses read-optimized connections.
Q6: What is the difference between CascadeType and orphanRemoval? Can you give a scenario where you'd use one without the other?
CascadeType controls which operations on the parent entity propagate to child entities. The types are: PERSIST (save parent = save children), MERGE, REMOVE, REFRESH, DETACH, and ALL (all of the above).

orphanRemoval = true means that when a child entity is removed from the parent's collection, Hibernate automatically deletes that child from the database. Without it, the child's FK is set to NULL (if nullable) or the child row stays orphaned.

Key difference: CascadeType.REMOVE deletes children when the parent is deleted. orphanRemoval deletes children when they are removed from the collection (even if the parent still exists).

Scenario: Use CascadeType.ALL without orphanRemoval — An Author has many Books, but books can exist independently. When you delete an Author, you want to delete their books too (CascadeType.REMOVE). But if you remove a Book from the Author's list, you don't want to delete the Book (orphanRemoval = false).

In our project, we use both together for Cart-CartItem and Order-OrderItem because items are compositional — they have no independent existence. Removing an item from the cart should delete the cart_item row.
Q7: Why is unitPrice stored in the OrderItem table? Why not just join to Product.price when displaying order history?
Because prices change. If a product's price changes after an order is placed, joining to Product.price would show the current price, not the price at the time of the order. This would produce incorrect financial records and break audit compliance.

By storing unitPrice as a snapshot in OrderItem, we freeze the price at order time. The same applies to subtotal (quantity * unitPrice). This creates an immutable financial record that never changes, even if product prices are updated later.

This is the same reason CartItem uses a transient getSubtotal() (runtime calculation from current price) while OrderItem persists the values — the cart is transient state, the order is historical fact.
Q8: How does a self-referencing entity like Category work? What are the alternatives for tree structures in JPA?
How it works: The Category table has a parent_id column that references the same table's id. Each category has at most one parent (null = root category). The @ManyToOne on parent and @OneToMany(mappedBy = "parent") on subCategories model this as a bi-directional self-referencing relationship. Hibernate handles this as a regular FK constraint — it just happens to reference the same table.

Alternatives:
(1) Nested Sets — Uses left and right columns. Efficient for reads (find all descendants in one query), expensive for writes (must renumber siblings). Hard to model in JPA.
(2) Closure Table — Stores ancestor-descendant pairs in a separate table. Flexible, supports multiple hierarchies, but requires extra table and more complex queries.
(3) Materialized Path — Stores full path as a string ("/electronics/computers/laptops"). Simple to query with LIKE but limited depth, no referential integrity.

For a typical e-commerce category tree (shallow depth, moderate size), the adjacency list is the simplest and most JPA-aligned choice.
Q9: What issues can arise when using Lombok's @Builder with JPA entities? How do you prevent them?
Issue 1: No-arg constructor missing. @Builder generates an all-args constructor. JPA/Hibernate requires a no-arg constructor (for proxy creation). Fix: Add @NoArgsConstructor.

Issue 2: Default field values are lost. @Builder initializes all fields to default JVM values (null, 0, false), ignoring field initializers like new ArrayList<>() or OrderStatus.PENDING. Fix: Add @Builder.Default on each field with a non-default initializer.

Issue 3: Circular references with toString() / equals() / hashCode(). Lombok's @EqualsAndHashCode or @ToString on bi-directional relationships causes StackOverflowError. Fix: Exclude relationship fields: @ToString.Exclude, @EqualsAndHashCode.Exclude. Better yet, avoid @EqualsAndHashCode on entities entirely — use business keys or @Data is considered harmful for JPA entities.

Issue 4: @AllArgsConstructor + @Builder can produce entities with null IDs. Calling new User(1L, ...) is fine, but User.builder().id(1L).build() can produce a managed entity with a manually set ID, which Hibernate treats as detached (it will try to UPDATE, not INSERT). Fix: Don't expose id in the builder, or use @Builder(builderMethodName = "createBuilder") with limited fields.
Q10: What is the difference between uni-directional and bi-directional relationships? When would you choose one over the other?
Uni-directional: Only one entity knows about the relationship. Example: Product knows its Category and Seller, but Category doesn't have a List<Product>. The FK is in the source entity's table.

Bi-directional: Both entities know about each other. One side is the owning side (has @JoinColumn), the other is the inverse side (has mappedBy). Example: Cart has List<CartItem> items and CartItem has Cart cart.

When to choose:
(1) Prefer uni-directional unless you need the inverse navigation. If you never query "all products in this category," don't add @OneToMany. Use a repository method instead (productRepository.findByCategoryId(id)).
(2) Use bi-directional when both directions are used in the domain (e.g., Cart-CartItem: "get items in this cart" and "which cart does this item belong to?"). Also use it for compositional patterns where CascadeType.ALL and orphanRemoval are required on the parent side.
(3) Avoid bi-directional for performance-sensitive paths. Each bi-directional association adds proxy overhead and maintenance complexity (keeping both sides consistent).

In this project: Cart-CartItem, Order-OrderItem, and Category-subCategories are bi-directional (compositional). Product-Category and Product-Seller are uni-directional (simple lookup).
