1 pom.xml Deep Dive

The pom.xml (Project Object Model) is the heart of any Maven project. It defines the project structure, dependencies, build configuration, and plugins. Every Spring Boot application starts here.

Parent — spring-boot-starter-parent

<parent>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-parent</artifactId>
    <version>3.4.1</version>
    <relativePath/>
</parent>

What it does: spring-boot-starter-parent is a special parent POM that provides:

  • Dependency Management: Pre-defined versions for hundreds of dependencies (Spring, Jackson, Hibernate, etc.). You no longer need to specify versions for Spring-managed dependencies — they inherit from the parent.
  • Plugin Configuration: Pre-configured Maven plugins (spring-boot-maven-plugin, maven-compiler-plugin, maven-surefire-plugin) with sensible defaults.
  • Java Version: Default Java version set to 17 (matching our project).
  • Resource Filtering: Automatic filtering of application properties using Maven's resource filtering.

Why we chose it: It eliminates version management for all Spring-managed libraries. Without it, you would manually track and align versions of every Spring dependency, which is error-prone and tedious.

Alternatives: You could use spring-boot-dependencies as a BOM import in <dependencyManagement> instead of using <parent>. This is useful when you already have a corporate parent POM. However, spring-boot-starter-parent is the recommended approach for most projects because it also provides plugin defaults.

Properties

<properties>
    <java.version>17</java.version>
    <jjwt.version>0.12.6</jjwt.version>
    <razorpay.version>1.4.4</razorpay.version>
    <springdoc.version>2.7.0</springdoc.version>
    <testcontainers.version>1.20.4</testcontainers.version>
</properties>

These properties centralize version numbers so they can be reused across multiple dependencies. Key choices:

  • java.version: 17 — Java 17 is the current LTS release with long-term support. It gives us pattern matching, records, sealed classes, and better performance than Java 11.
  • jjwt.version: 0.12.6 — io.jsonwebtoken (jjwt) is the most popular JWT library for Java. Version 0.12.x supports HMAC, RSA, and ECDSA algorithms. We chose this over Auth0's java-jwt because jjwt has a cleaner API for building/parsing JWTs and better Spring integration.
  • razorpay.version: 1.4.4 — Official Razorpay Java SDK. We use the latest stable version for payment gateway integration.
  • springdoc.version: 2.7.0 — SpringDoc OpenAPI generates Swagger UI automatically. We chose this over Springfox because Springfox is no longer actively maintained and has compatibility issues with Spring Boot 3.x.
  • testcontainers.version: 1.20.4 — TestContainers version for integration testing with real databases.

Dependencies by Group

Web Layer — spring-boot-starter-web

Dependency: org.springframework.boot:spring-boot-starter-web

What it does: Brings in everything needed to build REST APIs — Spring MVC (DispatcherServlet), embedded Tomcat server, Jackson for JSON serialization/deserialization, and validation support.

Why we chose Tomcat: It is the default embedded servlet container in Spring Boot. Tomcat is mature, well-tested, and performs excellently for most use cases. Alternatives include Undertow (better throughput for high-concurrency) and Jetty (lighter weight, used in embedded environments). We stick with Tomcat because it is the most widely deployed and has the best community support. For an e-commerce backend, Tomcat handles our expected load well.

Alternatives not chosen: spring-boot-starter-webflux (reactive stack with Netty) — we don't need reactive programming for this project. The standard servlet stack (Tomcat + Spring MVC) is simpler, more familiar, and sufficient for our CRUD-heavy workloads.

Data Access — JPA + PostgreSQL

Dependencies: spring-boot-starter-data-jpa, org.postgresql:postgresql (runtime scope)

What it does: Spring Data JPA provides the Hibernate ORM integration — entity mapping, repository interfaces (CRUD, pagination, custom queries), and transaction management. The PostgreSQL driver enables JDBC communication with our database.

Why Hibernate: It is the most mature JPA implementation (15+ years). It supports lazy loading, caching (first-level and second-level), HQL/JPQL queries, criteria API, and automatic schema generation. Alternatives include EclipseLink (more standards-compliant but less popular) and MyBatis (SQL-centric, not JPA). For an e-commerce app with complex entity relationships (Product -> Category, Order -> OrderItem), Hibernate's ORM capabilities are ideal.

Connection Pooling: Spring Boot automatically configures HikariCP when it detects spring-boot-starter-data-jpa on the classpath. HikariCP is the fastest connection pool in the Java ecosystem, with zero-overhead and lightning-fast initialization. Alternatives include Tomcat JDBC Pool (slower, more features we don't need) and DBCP2 (legacy, performance issues). HikariCP is the undisputed default.

Redis — Caching & Session Store

Dependency: spring-boot-starter-data-redis

What it does: Provides Redis integration via Lettuce (a non-blocking, scalable Redis client). We use Redis for caching product listings, storing refresh tokens, rate-limiting counters, and optional cart persistence.

Why Lettuce over Jedis: Jedis is a blocking client — each thread holds a connection while waiting for a response. Lettuce is asynchronous and uses Netty, supporting multiplexing (one connection handles multiple requests concurrently). This is crucial for high-throughput applications. Spring Boot 2.x+ defaults to Lettuce.

Alternatives: You could use a simple ConcurrentHashMap for in-memory caching, but it won't survive restarts or scale across multiple instances. Redis is the standard for distributed caching.

Security & JWT — Authentication & Authorization

Dependencies: spring-boot-starter-security, jjwt-api, jjwt-impl (runtime), jjwt-jackson (runtime)

What it does: Spring Security provides the filter chain architecture that intercepts every HTTP request. It handles authentication (who you are) and authorization (what you can do). The jjwt libraries handle JWT token generation (signing with HMAC-SHA256) and validation (parsing claims, checking expiry).

Why jjwt over alternatives: Auth0's java-jwt is simpler but lacks features like JWT compression or custom claim validation. Nimbus JOSE + JWT is more comprehensive but complex. jjwt strikes the right balance — clean builder API, built-in Jackson integration (jjwt-jackson for POJO deserialization), and separate API/impl JARs (api is compile-time, impl is runtime — follows the "interface segregation" principle).

Why separate jjwt JARs: jjwt-api is the public API (compile-scope), jjwt-impl and jjwt-jackson are runtime implementations. This means your code only depends on the API; the implementation can be swapped without recompiling consumer code. This is a security-conscious design pattern.

Validation — Bean Validation API

Dependency: spring-boot-starter-validation

What it does: Brings in Hibernate Validator (the reference implementation of Jakarta Bean Validation 3.0). It enables @Valid and @Validated annotations on controller request bodies, plus field-level annotations like @NotBlank, @Email, @Size, @Min, @Pattern.

Why we need it: Without validation, every controller method would need manual null-checking and format verification. Declarative validation keeps controllers clean and ensures consistent error messages.

Alternatives: Manual validation in service classes — which leads to code duplication. Jakarta Validation is the standard, portable, and testable approach.

Email — JavaMailSender

Dependency: spring-boot-starter-mail

What it does: Provides JavaMailSender interface with SMTP support for sending transactional emails (order confirmation, password reset, shipping updates). Under the hood, it uses Jakarta Mail (formerly JavaMail).

Alternatives: SendGrid API (third-party, needs separate SDK), Amazon SES (AWS-specific). Spring Mail is framework-native, simpler to configure, and uses standard SMTP which works with any provider.

API Documentation — SpringDoc OpenAPI

Dependency: org.springdoc:springdoc-openapi-starter-webmvc-ui:${springdoc.version}

What it does: Automatically generates OpenAPI 3.0 specification from your Spring MVC controllers. Provides a Swagger UI at /swagger-ui.html and an OpenAPI JSON at /api-docs. Every @RestController, @RequestMapping, and @RequestParam is documented without manual effort.

Why SpringDoc over Springfox: Springfox (the older library) has not kept pace with Spring Boot 3.x. It requires complex configuration, has annotation processing bugs, and breaks with newer Spring versions. SpringDoc is the modern, maintained alternative with seamless Spring Boot 3 support.

Lombok — Boilerplate Reduction

Dependency: org.projectlombok:lombok (optional)

What it does: Lombok is a compile-time annotation processor that generates getters, setters, constructors, builders, toString(), equals(), and hashCode() automatically. Annotations like @Getter, @Setter, @Builder, @NoArgsConstructor, @AllArgsConstructor, and @Data eliminate hundreds of lines of repetitive code.

Why we use it: Entity classes in an e-commerce project have 10-20 fields each. Writing getters/setters/constructors manually for 15+ entities is tedious, error-prone, and clutters the codebase. Lombok keeps entity files concise and readable.

Alternatives: Java 16+ records (immutable data carriers — but entities are mutable, so records don't work for JPA). Kotlin (language-level data classes — but requires switching the whole project). Manually generated code (IDEs can auto-generate, but the source file becomes bloated). Lombok is the pragmatic choice for Java + JPA projects.

How it works: Lombok hooks into the Java compiler's annotation processing API (JSR 269). During compilation, when the compiler encounters @Getter on a class, Lombok's annotation processor generates the getter methods directly into the AST (Abstract Syntax Tree) before bytecode generation. The generated code is invisible to you but present in the compiled .class file. This is why Lombok is an optional dependency — it's only needed at compile time.

Razorpay — Payment Gateway

Dependency: com.razorpay:razorpay-java:${razorpay.version}

What it does: Official Java SDK for Razorpay, India's leading payment gateway. It provides APIs for creating orders, capturing payments, refunds, and webhook signature verification.

Why Razorpay: For an e-commerce project targeting the Indian market, Razorpay is the de facto standard. It supports UPI, credit/debit cards, net banking, EMI, and wallets — all through a single integration. Alternatives include Stripe (not fully India-optimized), PayU, and CCAvenue. Razorpay has the cleanest API and best developer experience among Indian payment gateways.

Development Tools

Dependency: spring-boot-devtools (runtime, optional)

What it does: Provides automatic restart (when classpath files change), LiveReload (browser auto-refresh), and remote debugging support. Speeds up development by eliminating manual restarts.

Note: This is runtime scope and always excluded from the final JAR when building for production.

Test Dependencies — JUnit, Mockito, TestContainers, H2

Dependencies: spring-boot-starter-test, spring-security-test, spring-boot-testcontainers, testcontainers, testcontainers:postgresql, testcontainers:junit-jupiter, com.h2database:h2

What they do:

  • spring-boot-starter-test — Brings JUnit 5, Mockito, AssertJ, Hamcrest, and Spring Test utilities in one starter.
  • spring-security-test — Utility classes for testing secured endpoints (@WithMockUser, @WithAnonymousUser, SecurityMockMvcRequestPostProcessors).
  • testcontainers — JUnit extension for spinning up disposable Docker containers (PostgreSQL) during integration tests.
  • h2 — In-memory database for unit tests where we don't need a real PostgreSQL container.

Why both H2 and TestContainers: H2 is fast for unit tests (milliseconds to start). TestContainers is for integration tests that must run against a real PostgreSQL to catch database-specific issues. This layered testing strategy balances speed and accuracy.

Dependency Management — TestContainers BOM

<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.testcontainers</groupId>
            <artifactId>testcontainers-bom</artifactId>
            <version>${testcontainers.version}</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
    </dependencies>
</dependencyManagement>

What is a BOM (Bill of Materials)? A BOM is a POM that lists dependency versions but does not add them to your project. You declare the BOM in <dependencyManagement>, and then any TestContainers dependency you use (postgresql, junit-jupiter, mysql, etc.) inherits the version from the BOM. This ensures all TestContainers modules are on the same version, preventing classpath conflicts.

Why we use the TestContainers BOM: Without it, you would manually specify <version>1.20.4</version> on every TestContainers dependency. If you forget to update one, you could end up with incompatible versions. The BOM eliminates this risk.

Plugins

<build>
    <plugins>
        <plugin>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-maven-plugin</artifactId>
            <configuration>
                <excludes>
                    <exclude>
                        <groupId>org.projectlombok</groupId>
                        <artifactId>lombok</artifactId>
                    </exclude>
                </excludes>
            </configuration>
        </plugin>
        <plugin>
            <groupId>org.apache.maven.plugins</groupId>
            <artifactId>maven-compiler-plugin</artifactId>
            <configuration>
                <source>17</source>
                <target>17</target>
                <annotationProcessorPaths>
                    <path>
                        <groupId>org.projectlombok</groupId>
                        <artifactId>lombok</artifactId>
                    </path>
                </annotationProcessorPaths>
            </configuration>
        </plugin>
    </plugins>
</build>

spring-boot-maven-plugin:

  • What it does: Packages the application as an executable fat JAR. It combines all dependencies into a single archive with a nested JAR layout, and adds a META-INF/MANIFEST.MF pointing to the main class. Running mvn package produces target/ecommerce-backend-0.0.1-SNAPSHOT.jar which can be launched with java -jar.
  • Lombok exclusion: The <excludes> configuration prevents Lombok from being included in the final JAR. Lombok is only needed at compile time — including it in production would waste space and serve no purpose.

maven-compiler-plugin:

  • What it does: Configures the Java compiler. We set <source>17</source> and <target>17</target> explicitly (though the parent POM already sets these — explicit is safe).
  • annotationProcessorPaths: This is where we declare Lombok as an annotation processor. The <annotationProcessorPaths> configuration ensures Lombok runs during compilation. Without this, Lombok annotations like @Getter would be ignored and cause compilation errors. This is the standard Maven configuration for Lombok (alternatively, you could add it to <annotationProcessors> or let IDEs handle it, but the explicit path ensures consistent builds across all environments).
Key Interview Insight: The separation of spring-boot-maven-plugin (packaging the app) and maven-compiler-plugin (compiling the code) follows Maven's single-responsibility principle. Each plugin handles one phase of the build lifecycle. Neither plugin is optional — without the compiler plugin, your source won't compile; without the boot plugin, the JAR won't be executable.

3 Spring Boot Starters — Auto-Configuration Explained

What Are Starters?

A Starter is a curated set of dependencies bundled into a single Maven artifact. Instead of adding 10 individual JARs for building a web application (Spring MVC, Tomcat, Jackson, validation, etc.), you add one dependency:

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
</dependency>

This single line pulls in:

  • spring-boot-starter (core auto-configuration)
  • spring-boot-starter-tomcat (embedded Tomcat)
  • spring-webmvc (Spring MVC framework)
  • jackson-databind (JSON serialization)
  • hibernate-validator (bean validation)
  • spring-boot-starter-logging (Logback + SLF4J)
  • And their transitive dependencies

Why starters simplify setup:

  1. Single dependency — One line replaces 10+. Your pom.xml stays clean and readable.
  2. Version-managed — All versions are managed by the parent POM. No need to track which Jackson version is compatible with which Spring version.
  3. Auto-discovery — Adding spring-boot-starter-data-jpa automatically enables Hibernate, HikariCP, and @EnableJpaRepositories.
  4. Opinionated defaults — Spring Boot auto-configures sensible defaults. You only override what you need in application.yml.

The Auto-Configuration Mechanism

Spring Boot's auto-configuration works through a multi-step process:

1. The @SpringBootApplication Annotation

@SpringBootApplication  // (at the bottom of this file)
public class EcommerceApplication {
    public static void main(String[] args) {
        SpringApplication.run(EcommerceApplication.class, args);
    }
}

@SpringBootApplication is a meta-annotation that combines:

  • @EnableAutoConfiguration — Tells Spring Boot to automatically configure beans based on classpath dependencies.
  • @ComponentScan — Scans the current package and sub-packages for @Component, @Service, @Repository, @Controller beans.
  • @Configuration — Marks the class as a source of bean definitions.

2. How Auto-Configuration Reads Your Classpath

When the application starts:

  1. Spring Boot scans all JARs for files named META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports (previously spring.factories).
  2. Each auto-configuration class in these files is loaded conditionally.
  3. Conditions are evaluated using @ConditionalOnClass (does the class exist on the classpath?), @ConditionalOnMissingBean (is the bean already defined?), @ConditionalOnProperty (is the property set?), etc.

Concrete Example — How Spring Boot Configures JPA

1. pom.xml includes spring-boot-starter-data-jpa
2. JAR on classpath contains:
   META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports
   → Contains: org.springframework.boot.autoconfigure.orm.jpa.HibernateJpaAutoConfiguration
3. On startup:
   @ConditionalOnClass({ DataSource.class, EntityManagerFactory.class })
   @ConditionalOnMissingBean({ JpaVendorAdapter.class })
   → TRUE — Spring Boot automatically configures:
      • LocalContainerEntityManagerFactoryBean
      • JpaVendorAdapter (Hibernate)
      • JpaTransactionManager
      • PersistenceExceptionTranslationPostProcessor
4. Additionally:
   @ConditionalOnClass({ HikariDataSource.class })
   → TRUE — HikariCP connection pool is auto-configured
Interview Answer: "Spring Boot starters use a convention-over-configuration approach. Each starter is a pre-bundled set of dependencies. Behind the scenes, auto-configuration classes use @Conditional annotations to check what libraries are on the classpath and configure them automatically. You only write configuration for things that deviate from the defaults."

4 Maven Build Lifecycle

Maven follows a fixed sequence of build phases called the default lifecycle. Each phase represents a stage in the build process. Here are the key phases and what happens in each:

PhaseWhat HappensOur Plugin Hooks
compile Compiles Java source files from src/main/java to target/classes maven-compiler-plugin runs here, invoking javac with Java 17 and Lombok annotation processing
test-compile Compiles test source files from src/test/java to target/test-classes maven-compiler-plugin runs again for test sources
test Runs unit/integration tests using the test framework (JUnit 5 via maven-surefire-plugin) maven-surefire-plugin (inherited from parent POM) executes all @Test methods. Fails the build if any test fails.
package Packages compiled code into a distributable format (JAR, WAR, etc.) spring-boot-maven-plugin repackages the standard JAR into an executable fat JAR with all dependencies nested inside
install Copies the built JAR into the local Maven repository (~/.m2/repository) so other local projects can use it No custom plugin — standard Maven behavior

How Our Plugins Hook Into the Lifecycle

mvn clean install  →  [clean] validate → compile → test-compile → test → package → verify → install

Step by step for our project:

1. mvn compile
   → maven-compiler-plugin:compile (Java 17 + Lombok processing)
   → Output: target/classes/com/ecommerce/*.class

2. mvn test-compile
   → maven-compiler-plugin:testCompile (test sources)
   → Output: target/test-classes/com/ecommerce/*Test.class

3. mvn test
   → maven-surefire-plugin:test (runs all @Test methods)
   → If any test fails → BUILD FAILURE

4. mvn package
   → jar:jar (standard JAR packaging from maven-jar-plugin, inherited)
   → spring-boot-maven-plugin:repackage (converts to executable fat JAR)
   → Output: target/ecommerce-backend-0.0.1-SNAPSHOT.jar

5. mvn install
   → Copies JAR to ~/.m2/repository/com/ecommerce/ecommerce-backend/0.0.1-SNAPSHOT/
Key Interview Detail: The spring-boot-maven-plugin does NOT replace the standard JAR — it repackages it. The original JAR (ecommerce-backend-0.0.1-SNAPSHOT.jar.original) is preserved alongside the executable JAR. The original JAR contains only your project's classes (no dependencies), while the executable JAR includes all dependency JARs in a nested layout.

5 Spring Profiles — Dev, Test, Prod

What Are Profiles?

Spring Profiles allow you to define different configurations for different environments. Instead of manually editing application.yml before each deployment, you create separate profile-specific files and activate them with a single property or environment variable.

Our Profile Strategy

ProfileFileActive WhenDatabaseRedis
default (dev) application.yml Local development (mvn spring-boot:run) PostgreSQL (local) Redis (local)
test application-test.yml Running tests (mvn test) H2 (in-memory) Disabled
prod application-prod.yml Production deployment PostgreSQL (RDS/Cloud SQL) Redis (ElastiCache)

How Profile Activation Works

# Method 1: application.yml
spring:
  profiles:
    active: prod

# Method 2: Command-line argument
java -jar app.jar --spring.profiles.active=prod

# Method 3: Environment variable (production standard)
export SPRING_PROFILES_ACTIVE=prod

# Method 4: Maven test (automatically uses "test" profile)
mvn test  # Spring Boot's Maven plugin activates "test" profile automatically

How Spring Resolves Properties

Spring Boot loads properties in a specific priority order. Profile-specific files take precedence over the default file, and the order of precedence (highest first) is:

  1. Command-line arguments (--spring.profiles.active=prod)
  2. OS environment variables (SPRING_PROFILES_ACTIVE)
  3. Profile-specific application-{profile}.yml
  4. Default application.yml

This means properties in application-prod.yml override those in application.yml when the prod profile is active.

Interview Tip: "Profiles are not just for databases. We use them to configure different logging levels, different external service endpoints, feature flags, and even different bean definitions. For example, we can define a @Profile("dev") bean that seeds dummy data, and it won't be created in production."

Why Each Test Configuration is Set This Way

PropertyValueWhy
spring.autoconfigure.exclude Redis auto-configuration classes Redis requires a running server. In tests, we don't want to start Redis (it would fail if not installed). By excluding both RedisAutoConfiguration and RedisRepositoriesAutoConfiguration, we disable all Redis beans. If any code tries to use Redis, it will throw an error — which is desired because tests should not depend on an external cache.
spring.datasource.url jdbc:h2:mem:testdb;DB_CLOSE_DELAY=-1 H2 in-memory database. DB_CLOSE_DELAY=-1 keeps the database alive as long as the JVM is running (prevents premature shutdown). DB_CLOSE_ON_EXIT=FALSE prevents H2 from closing the database when the last connection is closed. H2 is fast (starts in milliseconds), requires no installation, and is perfectly suited for unit tests that don't need PostgreSQL-specific features (JSONB, tsvector, etc.).
spring.jpa.hibernate.ddl-auto create-drop Creates tables at the start of each test and drops them at the end. This ensures every test class runs against a clean, predictable schema. No test pollution from previous tests. create-drop is stricter than update and is the standard choice for testing.
spring.jpa.show-sql false Disables SQL logging in tests. Test output would be unreadable with hundreds of SQL statements. If you need to debug SQL in tests, enable it selectively per test class.
spring.jpa.properties.hibernate.dialect H2Dialect Informs Hibernate to generate H2-compatible SQL. H2 supports most PostgreSQL syntax, but there are differences (sequences, JSON functions). The dialect ensures compatibility.
spring.mail.* localhost:25 with no auth Disables real email sending. By default, spring-boot-starter-mail will try to connect to an SMTP server on localhost:25, which fails fast. The greenmail library (or mocking) is used for actual email testing.
server.port 0 Random port. When running integration tests with @SpringBootTest(webEnvironment = RANDOM_PORT), the server starts on a random available port to avoid port conflicts when running tests in parallel.
jwt.secret Test-specific secret Uses a separate test key. This ensures test JWT tokens are not validated by the production secret, and vice versa.

Why This Matters for CI/CD

In a CI/CD pipeline (e.g., GitHub Actions), the test profile ensures:

  • No external dependencies: Tests don't need Redis or PostgreSQL installed on the CI runner. H2 is an in-memory database with zero setup.
  • Fast execution: H2 starts in ~100ms. TestContainers PostgreSQL takes 5-10 seconds to start (used only for specific PostgreSQL integration tests).
  • Deterministic environment: Every CI run is identical — same database, same config, no state pollution.
  • Parallel safety: Random ports and in-memory databases mean tests can run in parallel without conflicts.
Interview Answer: "For unit tests and most integration tests, we use H2 in-memory with the test profile. This eliminates external dependencies (no Redis, no PostgreSQL) from the CI pipeline, making tests fast, deterministic, and parallel-safe. We use TestContainers only for the subset of tests that specifically need PostgreSQL features like JSONB queries or full-text search."

7 Interview Questions & Answers

Beginner Level

Q1: What is spring-boot-starter-parent and why do we need it?
spring-boot-starter-parent is a special parent POM provided by Spring Boot that serves as the foundation for any Spring Boot project. It provides three key benefits:

Dependency Management: It declares versions for hundreds of commonly used dependencies (Spring, Jackson, Hibernate, etc.) via Maven's <dependencyManagement> section. When you add spring-boot-starter-web, you don't specify a version — the parent POM provides it. This prevents version conflicts across compatible libraries.

Plugin Configuration: It pre-configures essential Maven plugins — maven-compiler-plugin (Java source/target level), maven-surefire-plugin (JUnit integration), spring-boot-maven-plugin (executable JAR packaging) — with sensible defaults.

Resource filtering and placeholders: It enables Maven resource filtering, allowing you to use @project.version@ placeholders in application properties.

Alternative: If your organization already has a corporate parent POM, you can import spring-boot-dependencies as a BOM in <dependencyManagement> instead of using the parent. However, you then lose the plugin defaults and resource filtering.
Q2: What is the difference between @SpringBootApplication and @EnableAutoConfiguration + @ComponentScan + @Configuration individually?
@SpringBootApplication is a composed annotation — a convenience annotation that bundles three annotations together:

- @EnableAutoConfiguration — Enables Spring Boot's auto-configuration mechanism, which automatically configures beans based on classpath dependencies (e.g., if H2 is on the classpath, auto-configure a DataSource for it).
- @ComponentScan — Tells Spring to scan the current package and all sub-packages for @Component, @Service, @Repository, @Controller beans and register them.
- @Configuration — Marks the class as a source of bean definitions via @Bean methods.

Functionally, they are equivalent. You could replace @SpringBootApplication with these three annotations and the application would behave identically. However, @SpringBootApplication is preferred because:
1. It's more concise (1 annotation vs 3)
2. It's self-documenting — anyone reading the code immediately knows this is the Spring Boot entry point
3. It's a single point of change if Spring Boot introduces additional defaults in future versions

There is one subtle difference: @SpringBootApplication's @ComponentScan excludes certain Spring-internal types. In practice, this rarely matters.

Intermediate Level

Q3: How does Spring Boot know which database driver and dialect to use?
Spring Boot uses its auto-configuration mechanism combined with classpath detection and @ConditionalOnClass annotations.

The process:
1. When you add org.postgresql:postgresql to your pom.xml, the PostgreSQL JDBC driver JAR is on the classpath.
2. Spring Boot's DataSourceAutoConfiguration class has @ConditionalOnClass({ DataSource.class, EmbeddedDatabaseType.class }) — it's activated because Spring JDBC is present via spring-boot-starter-data-jpa.
3. Based on the spring.datasource.url property (jdbc:postgresql://...), Spring Boot infers the driver class name as org.postgresql.Driver.
4. Similarly, HibernateJpaAutoConfiguration detects the PostgreSQL URL and selects PostgreSQLDialect automatically.
5. If you had added the H2 dependency instead, Spring Boot would auto-configure an H2 DataSource with H2Dialect — zero configuration changes needed.

Why we still specify driver-class-name and dialect explicitly: Auto-detection works 99% of the time, but explicit configuration removes ambiguity. If a new Hibernate version renames the dialect class, explicit configuration protects against breakage.
Q4: What is HikariCP and why does Spring Boot use it as the default connection pool?
HikariCP is a high-performance JDBC connection pool library. Spring Boot 2.x+ uses it as the default connection pool when it detects spring-boot-starter-data-jpa or spring-boot-starter-jdbc on the classpath.

Why HikariCP is the best choice:
1. Performance: HikariCP is the fastest connection pool in the Java ecosystem. Benchmarks show it is 2-3x faster than Tomcat JDBC Pool and DBCP2 for connection acquisition and release. It achieves this through code optimization (bytecode-level tweaks, fewer synchronized blocks, lock-free handoff where possible).
2. Small footprint: The library is only ~200KB. It has minimal dependencies (just SLF4J).
3. Zero-config defaults: HikariCP works well out of the box with sensible defaults (max pool size = 10, connection timeout = 30 seconds).
4. Reliability: It actively tests connections before handing them out, handles connection timeouts gracefully, and provides detailed logging for pool metrics.

Alternatives and why they are not default:
- Tomcat JDBC Pool: Was the default in Spring Boot 1.x. It is slower and more complex. Still used when running inside a traditional Tomcat container (JNDI datasource).
- DBCP2 (Commons DBCP): Older, slower, and has known issues with connection leaks under load. Not recommended for production.
- Vibur: A good alternative but less widely adopted. HikariCP is the community standard.
Q5: What are Spring Boot Starters? Can you explain the naming convention and give examples?
Spring Boot Starters are curated dependency descriptors. They are Maven POMs that bundle a coherent set of compatible dependencies for a specific purpose. Instead of manually adding 10 individual dependencies and ensuring they are all on compatible versions, you add one starter dependency.

Naming convention:
- Official Spring starters: spring-boot-starter-{name}
Examples: spring-boot-starter-web, spring-boot-starter-data-jpa, spring-boot-starter-security
- Third-party starters: {name}-spring-boot-starter
Examples: springdoc-openapi-starter-webmvc-ui, mybatis-spring-boot-starter

Examples of what starters include:
- spring-boot-starter-web includes: spring-webmvc, spring-boot-starter-tomcat, jackson-databind, hibernate-validator, spring-boot-starter-logging, and their transitive deps (~15 JARs total from 1 line).
- spring-boot-starter-data-jpa includes: spring-orm, hibernate-core, spring-data-jpa, spring-boot-starter-jdbc (which brings HikariCP), and all transitive deps.
- spring-boot-starter-test includes: JUnit 5, Mockito, AssertJ, Hamcrest, JSONassert, and Spring Test utilities.

Why starters matter: They enforce version compatibility. If you mix Spring Boot 3.4.1 with an incompatible Jackson version, you get runtime errors. Starters + parent POM guarantee all versions are tested and compatible.

Advanced Level

Q6: How does @SpringBootApplication work internally? Walk through the startup process.
Step-by-step breakdown from main() to a running application:

1. SpringApplication.run() is called:
- Creates a StopWatch to track startup time
- Detects the web application type (SERVLET — based on Tomcat on classpath)
- Reads spring.factories / AutoConfiguration.imports from all JARs to find all auto-configuration classes
- Creates an ApplicationContext (specifically AnnotationConfigServletWebServerApplicationContext)

2. Prepare the environment:
- Loads application.yml/properties from classpath
- Resolves active profiles from spring.profiles.active
- Binds all properties to a unified Environment object

3. @SpringBootApplication is processed:
- @ComponentScan creates a ClassPathScanningCandidateComponentProvider that scans the base package and sub-packages
- Every class annotated with @Component, @Service, @Repository, @Controller, @Configuration is discovered and registered as a bean definition

4. Auto-configuration phase:
- All auto-configuration classes (found in step 1) are loaded
- Each class has @Conditional* annotations evaluated at runtime:
→ @ConditionalOnClass: Is a specific class on the classpath? (e.g., DataSource.class)
→ @ConditionalOnMissingBean: Has the user already defined this bean? (user beans take priority)
→ @ConditionalOnProperty: Is a specific property set? (e.g., spring.datasource.url)
- Matching auto-configurations register their beans (DataSource, EntityManagerFactory, JwtDecoder, etc.)

5. Servlet container starts:
- Tomcat is embedded (or Undertow/Jetty depending on starter)
- DispatcherServlet is registered at /
- Server port (8080 by default) is bound

6. ApplicationReadyEvent is published:
- CommandLineRunner and ApplicationRunner beans execute
- The application is now ready to accept HTTP requests

Total time: Typically 2-5 seconds for a Spring Boot 3 application with JPA + Security.
Q7: What is the difference between @Configuration and @Component in Spring?
While both annotations register classes as Spring beans, they serve different purposes and have different internal behavior.

@Component:
- A generic stereotype annotation for any Spring-managed bean
- Used for auto-detection via classpath scanning (each class is instantiated directly)
- @Bean methods inside a @Component class are NOT processed as configuration methods — they are treated as regular factory methods
- Use for: @Service, @Repository, @Controller, utility beans

@Configuration:
- A specialized form of @Component that signals the class contains @Bean definitions
- @Configuration classes are CGLIB-proxied — Spring creates a subclass of the configuration class at runtime. This proxy intercepts calls to @Bean methods to ensure they return singleton beans (not new instances each time)
- If a @Bean method inside a @Configuration calls another @Bean method, the proxy returns the existing singleton, not a new instance
- In a @Component, calling one @Bean method from another would create a new instance each time — the singleton scope is NOT enforced

Concrete example:
@Configuration
public class AppConfig {
    @Bean
    public DataSource dataSource() { return new HikariDataSource(); }

    @Bean
    public JdbcTemplate jdbcTemplate() {
        return new JdbcTemplate(dataSource()); // singleton guaranteed
    }
}
In @Configuration, dataSource() is called once and the same instance is used for jdbcTemplate(). In @Component, dataSource() would be called twice, creating two separate DataSource instances — a bug.

When to use each:
- @Configuration: For classes that define @Bean methods (e.g., SecurityConfig, RedisConfig)
- @Component (and its specializations): For classes that are themselves beans and don't define factory methods
Q8: How does Lombok work at compile time? What is annotation processing in Java?
Lombok works through JSR 269 Pluggable Annotation Processing API — a standardized mechanism in Java for processing annotations at compile time.

How it works step-by-step:

1. Write code: You annotate a class with @Getter, @Setter, @Builder, etc.

2. Compilation starts: javac begins compiling your source files. It reads annotations and checks if any registered annotation processors handle them.

3. Lombok's processor runs: Lombok's LombokProcessor implements javax.annotation.processing.Processor. When javac encounters @Getter, it invokes Lombok's processor.

4. AST modification: Lombok does NOT generate a separate source file. Instead, it directly modifies the Abstract Syntax Tree (AST) — the in-memory tree structure representing your source code. It injects new method declarations (getters, setters, toString, etc.) into the AST.

5. Bytecode generation: javac continues compilation with the modified AST. The injected methods are compiled into the .class file just as if you had written them manually.

6. Result: The compiled .class file contains all the generated methods. This is why Lombok is a compile-time only dependency (marked <optional>true</optional>) — the runtime doesn't need Lombok; the methods are already in the bytecode.

Why this matters:
- No runtime overhead: Lombok isn't loaded at runtime. Generated code is ordinary bytecode.
- IDE support: IDEs (IntelliJ, Eclipse) run annotation processors during editing, so you see the generated getters/setters in code completion and navigation.
- Limitation: Lombok uses undocumented internal javac APIs to modify the AST. This means it may break with new Java versions. However, Project Lombok is actively maintained and typically updates within weeks of a new Java release.

Maven configuration required: In pom.xml, we must add Lombok to <annotationProcessorPaths> in the maven-compiler-plugin configuration. This tells javac to include Lombok's processor in the annotation processing round.
