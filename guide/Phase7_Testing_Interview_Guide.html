1 Testing Overview

The Testing Pyramid

Testing Pyramid (Bottom to Top): /\ / \ /E2E \ Few (smoke tests, critical flows) /------\ / \ /Integration\ Some (API + DB combined) /--------------\ / \ / Unit Tests \ Many (isolated, fast, reliable) /--------------------\

Our project uses a layered testing strategy aligned with the testing pyramid:

LayerTest TypeFrameworkCountScope
ServiceUnitJUnit 5 + Mockito7 classes / 18 testsBusiness logic, edge cases, exception paths. Dependencies mocked.
ControllerSlice@WebMvcTest + MockMvc7 classes / 17 testsHTTP mapping, validation, JSON serialization. Services mocked.
RepositorySlice@DataJpaTest + H23 classes / 13 testsCustom queries, derived methods, pagination. Real DB.
IntegrationFull context@SpringBootTest1 class / 3 testsFull application context with H2. End-to-end flows.

What Each Type Tests

  • Service tests — Business logic: validation rules, stock checks, authorization, pricing calculations, exception paths. Repositories are mocked so no real DB is needed.
  • Controller tests — HTTP concerns: status codes, JSON structure, validation errors, endpoint mapping. Services are mocked via @MockBean. Security filters are disabled with addFilters = false.
  • Repository tests — JPA queries: derived methods, custom @Query, pagination, sorting. Run against an in-memory H2 database with automatic rollback per test.
  • Integration test — Full request flow: register, login, access protected endpoints. Uses real H2 database with full application context.

2 Every Annotation Explained

JUnit 5 Core Annotations

AnnotationSyntaxHow It Works InternallyWhy We Use ItAlternatives
@Test @Test void method() JUnit Platform discovers methods annotated with @Test via reflection (using org.junit.jupiter.engine.descriptor.TestMethodTestDescriptor). Each method is executed as a separate test case. Marks a method as executable test. Required for test discovery. @RepeatedTest, @ParameterizedTest, @TestFactory
@BeforeEach @BeforeEach void setUp() Registered as a lifecycle callback in ExtensionContext.Store. Before each @Test method, the JUnit engine invokes all @BeforeEach methods (including inherited ones). Initialize test fixtures (create objects, set up mocks) before every test to ensure isolation. @BeforeAll (once per class, static)
@AfterEach @AfterEach void tearDown() Same lifecycle mechanism as @BeforeEach, but runs after each test. Executes even if the test throws an exception. Cleanup after each test: delete test data, close resources, reset state. @AfterAll (once per class, static)
@BeforeAll @BeforeAll static void init() Static method executed once before any @Test in the class. Must be static because the test instance may not have been created yet. One-time setup: start containers, initialize expensive resources. @BeforeEach (per test)
@AfterAll @AfterAll static void cleanup() Static method executed once after all tests in the class complete. Must be static. One-time teardown: close DB connections, stop containers. @AfterEach (per test)
@DisplayName @DisplayName("text") JUnit stores the display name in the TestDescriptor. Test engines and IDEs read this via TestDescriptor.getDisplayName() instead of the method name. Human-readable test names in reports, IDEs, and CI. Makes tests executable documentation. Method naming conventions
@Disabled @Disabled("reason") Test descriptor is created but execution is skipped. Engine checks TestDescriptor.isDisabled() before invoking the test. Reason is stored for reporting. Temporarily skip failing or incomplete tests. In our project, used on tests where @PreAuthorize is not enforced in @WebMvcTest. Conditional execution: @EnabledOnOs, @EnabledIfSystemProperty, assumeTrue()
@Nested @Nested class InnerTests Inner class is discovered as a separate test class. JUnit creates a separate instance of the inner class for each test run. All outer class state is accessible. Group related tests logically (e.g., CreateProductTests, SearchProductTests). Improves test organization and reporting. Separate test classes per feature
@ParameterizedTest @ParameterizedTest @ValueSource(strings = {"a","b"}) void test(String s) Test method is invoked multiple times with different arguments. The ParameterizedTestExtension provides argument resolution from the source annotation. Run the same test with multiple input values without duplicating code. Validates boundaries and edge cases. @CsvSource, @MethodSource, @EnumSource
@ValueSource @ValueSource(strings = {"", " ", "\t"}) Provides a single array of primitive/String values. Values are converted to the method parameter type via ArgumentConverter. Simple parameterized tests with literal values. Good for null/empty/blank input validation. @CsvSource (multiple params), @MethodSource (complex objects)
@CsvSource @CsvSource({"5,3,2", "10,10,0"}) Parses CSV lines into arguments. Each line is split by comma (configurable delimiter). Values are converted to parameter types. Multiple input arguments in a compact format. Great for data-driven business logic tests. @ValueSource, @MethodSource
@MethodSource @MethodSource("factoryMethod") static Stream<Arguments> factoryMethod() Calls a static (or instance if @TestInstance(Lifecycle.PER_CLASS)) factory method that returns a Stream of Arguments. JUnit calls Arguments.of() to pass them. Complex argument generation with reusable factory methods. Best for creating domain objects as test inputs. @CsvSource, @ValueSource

Mockito Annotations

AnnotationSyntaxHow It Works InternallyWhy We Use ItAlternatives
@ExtendWith(MockitoExtension.class) @ExtendWith(MockitoExtension.class) class TestClass Registers MockitoExtension which implements BeforeEachCallback and AfterEachCallback. Before each test, it processes @Mock, @InjectMocks, @Spy, @Captor fields via MockitoAnnotations.openMocks(). After each test, it validates Mockito.validateMockitoUsage() to detect unnecessary stubbing. Automatic mock initialization without calling MockitoAnnotations.openMocks() manually. Also enables strict stubbing (fails on unused stubs). MockitoAnnotations.openMocks(this) in @BeforeEach, MockitoRule (JUnit 4)
@Mock @Mock private Repository repo MockitoExtension calls Mockito.mock(Field.getType()). For classes, Mockito uses ByteBuddy to create a CGLIB-style proxy subclass. For interfaces, a JDK dynamic proxy is created. All methods return defaults unless stubbed. Create a test double that simulates dependencies. We control what methods return, what exceptions they throw, and verify interactions. @Spy (wraps real object), @MockBean (Spring-managed mock)
@InjectMocks @InjectMocks private Service service Mockito creates an instance of the field's type and injects @Mock fields into it. Injection strategy (in order): constructor injection (preferred) -> setter injection -> field injection. For constructor injection, Mockito resolves parameters by type matching. Create the system under test (SUT) with mocked dependencies automatically injected. Avoids manual new Service(mock1, mock2). Manual constructor call with mocks, Spring's @Autowired with @MockBean
@Spy @Spy private Service service Wraps a real object. By default delegates calls to the real method. Can selectively stub specific methods with doReturn(...).when(spy).method(). Created via Mockito.spy(). Partial mocking: keep real behavior for most methods but stub specific ones. Useful for legacy code or testing abstract classes. @Mock (full mock), @InjectMocks (real + mock deps)
@Captor @Captor private ArgumentCaptor<Type> captor ArgumentCaptor captures method arguments during verify(). After verify(mock).method(captor.capture()), call captor.getValue() to inspect the actual argument passed. Assert on complex objects passed to mocked methods. For example, capture the Order object saved to verify all fields. any(Class.class) with eq(), manual assertions on returned values

Spring Boot Test Annotations

AnnotationSyntaxHow It Works InternallyWhy We Use ItAlternatives
@WebMvcTest @WebMvcTest(Controller.class) A test slice annotation that applies @AutoConfigureMockMvc and @AutoConfigureWebMvc. It uses TypeExcludeFilter to include only web-related beans: @Controller, @ControllerAdvice, WebMvcConfigurer, Jackson modules, converters, and security config. @Service and @Repository beans are NOT loaded. Fast controller tests (loads only web layer). Focuses on HTTP concerns without loading the full context. Services must be mocked with @MockBean. @SpringBootTest + @AutoConfigureMockMvc (loads full context, slower)
@AutoConfigureMockMvc @AutoConfigureMockMvc(addFilters = false) Auto-configures the MockMvc bean using MockMvcAutoConfiguration. When addFilters = false, the Spring Security filter chain is NOT applied, so security annotations (@PreAuthorize) are not enforced. Requests bypass authentication. Provides MockMvc for perform/andExpect/andReturn chain. Disabling filters allows testing endpoint logic without authentication setup, but @WithMockPrincipal is still needed if controllers access SecurityContext. Default (filters enabled), custom security config
@MockBean @MockBean private Service service Creates a Mockito mock and registers it in the Spring ApplicationContext as a bean, replacing any existing bean of the same type. The mock is managed by Spring's MockitoPostProcessor (a BeanFactoryPostProcessor). Replace real @Service/@Repository beans with mocks in the test context. Required in @WebMvcTest because real services are not loaded. @Mock (standalone, not Spring-managed), @SpringBootTest (real beans)
@DataJpaTest @DataJpaTest class RepositoryTest Test slice that auto-configures an in-memory H2 database, enables JPA repositories, and configures TestEntityManager. Uses EntityScanExcluder to scan only @Entity classes. Applies @Transactional to each test method — all changes roll back after the test. Test repository queries with a real database without loading services or controllers. Fast and isolated. @SpringBootTest (full context, slower), @JdbcTest (JDBC only)
@AutoConfigureTestDatabase @AutoConfigureTestDatabase(replace = Replace.ANY) Replaces the configured DataSource with an embedded database (H2 by default). Replace.ANY forces H2 even if PostgreSQL is configured. Replace.NONE keeps the configured database. Ensures @DataJpaTest always uses an embedded DB regardless of the active profile. Prevents accidental connection to production DB during tests. @AutoConfigureTestDatabase(replace = Replace.NONE) to test against real DB
@SpringBootTest @SpringBootTest(webEnvironment = WebEnvironment.RANDOM_PORT) Loads the full ApplicationContext via SpringApplication.run(). Scans all beans: @Service, @Repository, @Controller, @Configuration, etc. With RANDOM_PORT, it starts an embedded web server on a random port, and TestRestTemplate or MockMvc can be used. Integration tests that need the complete application context. Tests real bean wiring, transaction behavior, and full request flows. Test slices (@WebMvcTest, @DataJpaTest), @ContextConfiguration (partial context)
@TestConfiguration @TestConfiguration static class Config { @Bean ... } A special @Configuration that is excluded from component scanning but can be imported in tests. Beans defined here override production beans. Define test-specific beans (e.g., mock external service clients) without affecting production configuration. @MockBean, @DynamicPropertySource

Security Context Annotations

AnnotationSyntaxHow It Works InternallyWhy We Use ItAlternatives
@WithMockUser @WithMockUser(roles = "ADMIN") Spring Security Test annotation. Uses WithMockUserSecurityContextFactory to create a UsernamePasswordAuthenticationToken with a User principal (Spring Security's org.springframework.security.core.userdetails.User). Sets it in SecurityContextHolder. Quickly set up a mock authenticated user in controller tests. The principal is User (username, password, authorities). @WithMockPrincipal (custom), @WithUserDetails, manual SecurityContext setup
@WithMockPrincipal (custom) @WithMockPrincipal(userId = 1L, role = "SELLER") Custom annotation. Uses WithMockPrincipalSecurityContextFactory which creates a real User entity, wraps it in UserPrincipal (our custom class), creates a UsernamePasswordAuthenticationToken with that principal, and sets it in SecurityContextHolder. Our controllers expect UserPrincipal (not User) in SecurityContext. Required because @WithMockUser creates Spring Security's User, not our UserPrincipal, causing ClassCastException in controllers. @WithMockUser (creates User, not UserPrincipal), @WithUserDetails
Key Insight: @WebMvcTest with addFilters = false disables the Spring Security filter chain — this means authentication is not enforced. But controllers that read SecurityContextHolder.getContext().getAuthentication().getPrincipal() will get null or throw NPE. The custom @WithMockPrincipal solves this by populating SecurityContext with our custom UserPrincipal object.

3 Service Testing with Mockito

Service tests verify business logic in isolation. All external dependencies (repositories, other services, infrastructure clients) are mocked. We have 7 service test classes: AuthServiceTest, CartServiceTest, CategoryServiceTest, EmailServiceTest, OrderServiceTest, PaymentServiceTest, ProductServiceTest.

Structure Pattern

Every service test follows the same pattern:

@ExtendWith(MockitoExtension.class)
class XxxServiceTest {

    @Mock private XxxRepository repository;   // Dependencies are mocked
    @Mock private OtherService otherService;

    @InjectMocks private XxxService service;   // SUT is real, with mocks injected

    private Entity entity;                     // Test fixtures in fields

    @BeforeEach
    void setUp() {
        entity = Entity.builder().id(1L).build();  // Create test data
    }

    @Test
    void method_ShouldDoX_WhenCondition() {
        when(repository.findById(1L)).thenReturn(Optional.of(entity));

        Result result = service.method(1L);

        assertNotNull(result);
        verify(repository).findById(1L);
    }
}

Mockito Stubbing: when/thenReturn

Used when a dependency method returns a value and we want to control what it returns:

// From ProductServiceTest (line 89-91)
when(categoryRepository.findById(1L)).thenReturn(Optional.of(category));
when(userRepository.findById(2L)).thenReturn(Optional.of(seller));
when(productRepository.save(any(Product.class))).thenReturn(product);

// From CartServiceTest (line 74-76)
when(cartRepository.findWithItemsByUserId(1L)).thenReturn(Optional.of(cart));
when(productRepository.findById(1L)).thenReturn(Optional.of(product));
when(cartRepository.save(any(Cart.class))).thenReturn(cart);

Mockito Stubbing: doThrow/when for Void Methods

Void methods cannot be used inside when(). Use doThrow().when() instead:

// From AuthServiceTest (line 147-148) — AuthenticationManager.authenticate() returns void
doThrow(new RuntimeException("Bad credentials"))
    .when(authenticationManager).authenticate(any());

ArgumentMatchers: any(), anyLong(), eq()

Matchers allow flexible or precise argument matching when stubbing and verifying:

import static org.mockito.ArgumentMatchers.*;

// any() — match any object of any type
when(productRepository.save(any(Product.class))).thenReturn(product);

// anyLong() — match any long value
when(categoryRepository.findById(anyLong())).thenReturn(Optional.empty());

// anyString() — match any String
when(userRepository.existsByEmail(anyString())).thenReturn(false);

// eq() — match an exact value (needed when mixing matchers and literals)
// When using matchers, ALL arguments must use matchers
when(productService.updateProduct(eq(1L), any(ProductRequest.class), anyLong()))
    .thenReturn(response);

// any() with specific type
when(cartService.addItemToCart(anyLong(), any(CartItemRequest.class)))
    .thenReturn(createCartResponse());

verify() — Interaction Testing

Verify that the service actually called the mocked method. This is critical for side-effect methods (save, delete, send).

// From ProductServiceTest (line 103) — verify save was called
verify(productRepository).save(any(Product.class));

// From AuthServiceTest (line 94-97) — verify exact arguments
verify(userRepository).existsByEmail("test@example.com");
verify(userRepository).save(any(User.class));
verify(jwtTokenProvider).generateAccessToken(1L, "test@example.com", "CUSTOMER");

// From ProductServiceTest (line 113) — verify a method was NEVER called
verify(productRepository, never()).save(any());

// From CartServiceTest (line 135) — verify exact call count
verify(cartRepository, times(2)).save(any(Cart.class));

// From EmailServiceTest (line 26) — verify void method was called
verify(mailSender).send(any(SimpleMailMessage.class));

assertThrows — Exception Testing

Always test that exceptions are thrown with the correct type. Also verify that no data was persisted:

// From CartServiceTest (line 112-113) — exception type assertion
assertThrows(BadRequestException.class,
        () -> cartService.addItemToCart(1L, request));

// From ProductServiceTest (line 110-113) — exception + verify no save
assertThrows(ResourceNotFoundException.class,
        () -> productService.createProduct(productRequest, 2L));
verify(productRepository, never()).save(any());

// From OrderServiceTest (line 129-132) — exception + verify no side effect
assertThrows(BadRequestException.class,
        () -> orderService.placeOrder(1L, "123 Main St"));
verify(orderRepository, never()).save(any());

Common Mockito Mistakes

MistakeWhy It's WrongHow to Fix
Unnecessary stubbingA stub is set up but never called. With strict stubs (default in MockitoExtension), this fails the test with UnnecessaryStubbingException.Remove unused stubs. Only stub methods that are actually called by the SUT.
Calling real method on mockIf a method is not stubbed, a mock returns default values (null, 0, false). If the SUT depends on a non-null return, it gets NPE.Stub the method with when(x).thenReturn(y). Or use Mockito.CALLS_REAL_METHODS (rare).
Mixing matchers and literalswhen(repo.findById(1L, any())) throws InvalidUseOfMatchersException. Once you use a matcher for one arg, ALL args must use matchers.Use eq(1L) instead of 1L when any other arg uses a matcher: when(repo.findById(eq(1L), any())).
Not verifying side effectsTest passes but the save()/delete() method was never called because an exception was silently caught.Always add verify(repo).save(any()) for methods that should persist data.
Mocking value objectsMocking entities with mock(Product.class) breaks equals/hashCode/toString. Using real objects via builder pattern is cleaner.Use Product.builder().id(1L).name("Laptop").build() (real object) instead of mocks.

Full Example: ProductServiceTest

// From src/test/java/com/ecommerce/service/ProductServiceTest.java
@ExtendWith(MockitoExtension.class)
class ProductServiceTest {

    @Mock private ProductRepository productRepository;
    @Mock private CategoryRepository categoryRepository;
    @Mock private UserRepository userRepository;
    @InjectMocks private ProductService productService;

    private ProductRequest productRequest;
    private Product product;
    private Category category;
    private User seller;

    @BeforeEach
    void setUp() {
        category = Category.builder().id(1L).name("Electronics").build();
        seller = User.builder().id(2L).name("Test Seller").build();
        product = Product.builder().id(1L).name("Laptop")
                .price(new BigDecimal("999.99")).stock(10)
                .category(category).seller(seller).build();
        productRequest = ProductRequest.builder().name("Laptop")
                .price(new BigDecimal("999.99")).stock(10).categoryId(1L).build();
    }

    @Test
    void createProduct_ShouldReturnProductResponse_WhenSuccessful() {
        when(categoryRepository.findById(1L)).thenReturn(Optional.of(category));
        when(userRepository.findById(2L)).thenReturn(Optional.of(seller));
        when(productRepository.save(any(Product.class))).thenReturn(product);

        ProductResponse response = productService.createProduct(productRequest, 2L);

        assertNotNull(response);
        assertEquals("Laptop", response.getName());
        assertEquals(new BigDecimal("999.99"), response.getPrice());
        assertEquals("Electronics", response.getCategoryName());
        verify(productRepository).save(any(Product.class));
    }

    @Test
    void createProduct_ShouldThrowException_WhenCategoryNotFound() {
        when(categoryRepository.findById(anyLong())).thenReturn(Optional.empty());

        assertThrows(ResourceNotFoundException.class,
                () -> productService.createProduct(productRequest, 2L));
        verify(productRepository, never()).save(any());
    }

    @Test
    void deleteProduct_ShouldDelete_WhenAuthorized() {
        when(productRepository.findById(1L)).thenReturn(Optional.of(product));

        productService.deleteProduct(1L, 2L);

        verify(productRepository).delete(product);
    }

    @Test
    void deleteProduct_ShouldThrowException_WhenUnauthorized() {
        when(productRepository.findById(1L)).thenReturn(Optional.of(product));

        assertThrows(BadRequestException.class,
                () -> productService.deleteProduct(1L, 99L));
        verify(productRepository, never()).delete(any());
    }
}
Key Insight: Every service method should have at least one happy-path test (assertions on return values + verify interactions) and one exception-path test (assertThrows + verify never). Notice how verify(repository, never()) confirms that no data was persisted when an exception is thrown.

Controller tests use @WebMvcTest to load only the web layer. The controller is real, but all services are mocked via @MockBean. We have 7 controller test classes: AuthControllerTest, CartControllerTest, CategoryControllerTest, HealthControllerTest, OrderControllerTest, PaymentControllerTest, ProductControllerTest.

String json = mockMvc.perform(get("/api/products"))

Security Context Setup

Our controllers use @WithMockPrincipal (custom) instead of @WithMockUser because the controllers cast the principal to UserPrincipal. Here's the difference:

// This would fail with ClassCastException:
@WithMockUser(roles = "CUSTOMER")   // Creates User principal
void getCart_ShouldReturnCart() {
    // Inside controller: UserPrincipal principal = (UserPrincipal) SecurityContext...
    // ClassCastException: User cannot be cast to UserPrincipal
}

// This works:
@WithMockPrincipal                   // Creates UserPrincipal principal
void getCart_ShouldReturnCart() {
    // Inside controller: UserPrincipal principal = (UserPrincipal) SecurityContext...
    // Works correctly
}

// @WithMockUser is used only in PaymentControllerTest which doesn't cast to UserPrincipal

5 Repository Testing with @DataJpaTest

@DataJpaTest loads only JPA beans: entities, repositories, and the data source. It auto-configures an in-memory H2 database (not PostgreSQL). Each test method is @Transactional — all changes are rolled back after the test completes, so tests never interfere with each other.

What @DataJpaTest Does Internally

  • Auto-configures an embedded H2 database via EmbeddedDataSourceConfiguration
  • Scans for @Entity classes and initializes the schema using ddl-auto: create-drop
  • Enables Spring Data JPA repositories
  • Provides TestEntityManager for convenient test data setup and flushing
  • Wraps each test in a @Transactional proxy — rollback after test completes
  • Does NOT load @Service, @Controller, @Component, or any web-related beans

Why H2 Instead of PostgreSQL for Repository Tests

FactorH2PostgreSQL (TestContainers)

TestEntityManager

TestEntityManager is a test-friendly wrapper around JPA's EntityManager. Key methods:

entityManager.persist(entity);          // Persist and assign ID
entityManager.persistFlushFind(entity);  // Persist, flush, and return the managed entity
entityManager.flush();                   // Flush to DB (forces SQL execution for query testing)
entityManager.clear();                   // Clear the persistence context (detach all entities)
entityManager.find(Entity.class, id);    // Find by primary key

6 Integration Testing with @SpringBootTest

Our project has one integration test: EcommerceApplicationIntegrationTest. It uses @SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT) to load the full application context and start an embedded web server on a random port.

Full Application Context Loading

@SpringBootTest calls SpringApplication.run() internally, which:

  • Creates the complete ApplicationContext with all beans (@Service, @Repository, @Controller, @Configuration)
  • Applies all auto-configuration (JPA, Security, MVC, etc.)
  • Creates real beans — no mocks. The real database, real JPA repositories, real security filter chain.

H2 Decision (Not TestContainers)

This project uses H2 in-memory database for integration tests through application-test.yml, not TestContainers. The decision is based on:

  • CI simplicity — No Docker requirement. Tests run anywhere without Docker installed.
  • Speed — H2 starts in milliseconds vs 10-30s for Docker containers.
  • Adequate for current queries — The project's JPQL queries (pagination, filtering, derived methods) work identically on H2 + PostgreSQL mode.
  • Trade-off — If the project adds PostgreSQL-specific features (JSONB, full-text search, tsvector, window functions), TestContainers would become necessary.

7 Custom @WithMockPrincipal Annotation

Why It Was Needed

Our controllers retrieve the authenticated user from SecurityContextHolder and cast the principal to UserPrincipal:

// Typical pattern in our controllers:
UserPrincipal principal = (UserPrincipal) SecurityContextHolder
        .getContext().getAuthentication().getPrincipal();
Long userId = principal.getId();  // UserPrincipal has getId(), getEmail(), getRole()

The standard @WithMockUser annotation creates a UsernamePasswordAuthenticationToken with a org.springframework.security.core.userdetails.User as the principal — not our custom UserPrincipal. This causes a ClassCastException at runtime.

Custom @WithMockPrincipal solves this by creating a real User entity, wrapping it in UserPrincipal, and using that as the authentication principal — exactly what happens in production.

How It Works: Annotation to SecurityContext Flow

1. @WithMockPrincipal(userId = 1L, role = "SELLER")
   placed on a test method or class.

2. Spring Security Test framework detects @WithSecurityContext(factory = ...)
   and calls the factory's createSecurityContext() method.

3. WithMockPrincipalSecurityContextFactory.createSecurityContext():
   - Reads annotation attributes (userId, email, role)
   - Creates a User entity (our JPA entity)
   - Wraps it in UserPrincipal (our custom UserDetails implementation)
   - Creates UsernamePasswordAuthenticationToken with UserPrincipal
   - Sets it in SecurityContextHolder

4. When the controller runs, SecurityContextHolder.getContext()
   .getAuthentication().getPrincipal() returns UserPrincipal. Success!

The Annotation

// From src/test/java/com/ecommerce/support/WithMockPrincipal.java
@Target({ElementType.METHOD, ElementType.TYPE})
@Retention(RetentionPolicy.RUNTIME)
@WithSecurityContext(factory = WithMockPrincipalSecurityContextFactory.class)
public @interface WithMockPrincipal {
    long userId() default 1L;
    String email() default "user@example.com";
    String role() default "CUSTOMER";
}

The SecurityContextFactory

// From src/test/java/com/ecommerce/support/WithMockPrincipalSecurityContextFactory.java
public class WithMockPrincipalSecurityContextFactory
        implements WithSecurityContextFactory<WithMockPrincipal> {

    @Override
    public SecurityContext createSecurityContext(WithMockPrincipal annotation) {
        // Create a real User entity (our JPA entity)
        User user = User.builder()
                .id(annotation.userId())
                .email(annotation.email())
                .name("Test User")
                .password("password")
                .role(Role.valueOf(annotation.role()))
                .build();

        // Wrap in UserPrincipal (our custom UserDetails)
        UserPrincipal principal = new UserPrincipal(user);

        // Create authentication token
        Authentication auth = new UsernamePasswordAuthenticationToken(
                principal, null, principal.getAuthorities());

        // Set in SecurityContext
        SecurityContext context = SecurityContextHolder.createEmptyContext();
        context.setAuthentication(auth);
        return context;
    }
}

Usage in Tests

// Default: userId=1, email=user@example.com, role=CUSTOMER
@WithMockPrincipal
void getCart_ShouldReturnCart() { ... }

// Custom role:
@WithMockPrincipal(role = "SELLER")
void createProduct_ShouldReturn201_WhenSeller() throws Exception { ... }

// Custom role and userId:
@WithMockPrincipal(role = "ADMIN")
void getAllOrders_ShouldReturnAll_WhenAdmin() throws Exception { ... }

The SecurityContextFactory Interface

WithSecurityContextFactory<T> is a Spring Security Test interface with a single method:

public interface WithSecurityContextFactory<T extends Annotation> {
    SecurityContext createSecurityContext(T annotation);
}

Spring's SecurityContextTestExecutionListener discovers the annotation, finds the factory via @WithSecurityContext(factory = ...), invokes createSecurityContext(), and stores the returned SecurityContext in SecurityContextHolder before the test runs. After the test, it clears SecurityContextHolder.

9 Interview Q&A

1. What is the difference between @Mock and @MockBean?
@Mock (Mockito) creates a standalone mock object that is NOT managed by Spring. It's used in unit tests with @ExtendWith(MockitoExtension.class). The mock is created via Mockito.mock() and injected into @InjectMocks targets via field/constructor injection.

@MockBean (Spring Boot) creates a Mockito mock AND registers it as a bean in the Spring ApplicationContext, replacing any existing bean of the same type. It's used in @WebMvcTest and @SpringBootTest to replace real service/repository beans with mocks. Internally uses MockitoPostProcessor (a BeanFactoryPostProcessor) to intercept bean definitions.

When to use which: Use @Mock + @InjectMocks for pure unit tests (no Spring context). Use @MockBean in slice tests (@WebMvcTest) where Spring loads controllers but services must be mocked.
2. How does Mockito create mocks internally — CGLIB vs JDK Proxy?
Mockito uses ByteBuddy (replaced CGLIB in recent versions) to create mock objects at runtime.

For interfaces: Mockito creates a JDK dynamic proxy via java.lang.reflect.Proxy. The proxy implements the interface and intercepts all method calls through an InvocationHandler. This is lightweight and requires no additional library.

For classes: Mockito uses ByteBuddy to subclass the target class. ByteBuddy generates a new class at runtime that extends the target class, overrides all non-final methods, and delegates to MethodInterceptor. This is necessary because JDK Proxy can only proxy interfaces.

Important: Final classes and final methods CANNOT be mocked. ByteBuddy cannot override final methods. If you encounter this, either remove the final keyword or use PowerMock (not recommended).
3. What is the difference between @WebMvcTest and @SpringBootTest for controller testing?
@WebMvcTest(Controller.class) loads only the web slice: the specified controller, @ControllerAdvice, filters, Jackson config, and security config. Services and repositories are NOT loaded (must be mocked with @MockBean). Startup is fast (2-5 seconds).

@SpringBootTest loads the FULL application context: all beans, all auto-configuration, real database, real repositories. Startup is slower (10-30 seconds).

When to use each:
- @WebMvcTest for 90% of controller tests — focus on HTTP mapping, validation, JSON serialization
- @SpringBootTest for integration tests that test real security filters, real JWT flow, or end-to-end scenarios

Key limitation of @WebMvcTest: With addFilters = false, security annotations like @PreAuthorize on service methods are not enforced. This is why tests like createProduct_ShouldReturn403_WhenNotSeller() are marked @Disabled in our project.
4. How does MockMvc work internally?
MockMvc is a server-side mock of the Spring MVC dispatcher. It does NOT start an HTTP server. Instead:

1. mockMvc.perform(request) creates a MockHttpServletRequest and MockHttpServletResponse
2. The request is processed by the actual DispatcherServlet — the same servlet that handles real HTTP requests
3. DispatcherServlet runs through the full chain: handler mapping -> interceptor -> argument resolver -> controller method -> return value handler -> response
4. But all of this happens in-memory, without network I/O or a real server socket

Why this is fast: No TCP connection, no thread pool, no socket bind. The entire request-response cycle runs synchronously in the test thread. The cost is equivalent to calling the controller method directly, plus the overhead of the dispatcher infrastructure.

Limitation: MockMvc cannot test the complete HTTP contract (headers like Set-Cookie, redirects, or HTTP/2). For those, use @SpringBootTest(webEnvironment = RANDOM_PORT) with TestRestTemplate.
5. Mockito verify() vs assertions — when to use each?
Assertions (assertEquals, assertNotNull, assertThrows) test the result of a method call. Use when the method returns a value that reflects its behavior.

verify() tests interactions with dependencies. Use when:
- The method has side effects (save, delete, sendEmail)
- You need to confirm the correct arguments were passed
- You need to confirm a method was never called (exception paths)

Example — when both are needed:
@Test
void createProduct_ShouldSucceed() {
    when(productRepository.save(any(Product.class))).thenReturn(product);

    ProductResponse response = productService.createProduct(request);

    // Assertion: verify the return value
    assertEquals("Laptop", response.getName());

    // verify(): confirm save was actually called
    verify(productRepository).save(any(Product.class));
}
Rule of thumb: Assertions on the return value + verify() on side effects. Both are usually needed for thorough testing.
6. What is the difference between @InjectMocks and @Autowired?
@InjectMocks (Mockito) creates a new instance of the class under test and injects @Mock fields into it. It is NOT Spring-managed — no bean post-processing, no AOP proxies, no @Transactional behavior. The class is a plain Java object with mocked dependencies.

@Autowired (Spring) injects a bean from the Spring ApplicationContext. The bean is fully managed: it can be proxied for @Transactional, @Cacheable, or @Async; it goes through bean post-processors; it participates in dependency injection.

Why @InjectMocks for unit tests: We don't want Spring proxies or transaction management in pure unit tests. We want to test the business logic in isolation. @InjectMocks gives us a plain instance where we control every dependency.

Why not @InjectMocks for integration tests: Integration tests need real Spring beans with transactions, security, and all infrastructure. @Autowired provides the fully proxied, real bean.
7. How does @DataJpaTest achieve isolation?
@DataJpaTest achieves isolation through several mechanisms:

1. Embedded database: Each test class gets its own in-memory H2 instance. No shared database state between test classes.

2. @Transactional per test: Each test method runs inside a transaction that is rolled back after the test completes. Any data created during the test (even in @BeforeEach) is discarded.

3. Slice loading: Only JPA-related beans are loaded (@Entity, Repository). @Service, @Controller, and other beans are excluded. This prevents accidental side effects from unrelated components.

4. Clean schema: With ddl-auto: create-drop, the schema is created from scratch at the start of the test class and dropped at the end. No migration scripts are applied.

What this means in practice: Tests are truly isolated. Test A creates 10 products. Test B runs after and sees a clean database with 0 products. Tests cannot interfere with each other regardless of execution order.
8. What is the @Transactional rollback behavior in tests?
By default, @Transactional in tests rolls back after each test method. This is different from production, where @Transactional commits on success.

How it works:
1. Spring's TransactionalTestExecutionListener starts a transaction before the test
2. The test runs (including @BeforeEach setup)
3. After the test completes (including @AfterEach), TransactionAspectSupport calls TransactionStatus.setRollbackOnly()
4. The transaction is rolled back — no data persists

Override rollback behavior:
@Rollback(false)  // Commit changes (rare — useful for debugging)
@Transactional(propagation = Propagation.NEVER)  // Don't wrap in transaction
@Commit          // Alternative to @Rollback(false)
Important: @DataJpaTest automatically applies @Transactional. Our integration test EcommerceApplicationIntegrationTest also uses @Transactional explicitly. This means even the full-context integration test rolls back, keeping the test database clean.
9. When should you use TestContainers (real PostgreSQL) vs H2 for testing?
Use H2 (embedded, in-memory) when:
- Testing standard JPA queries: derived methods, JPQL, pagination, sorting
- Running fast feedback tests on developer machines
- CI environments without Docker
- Testing repository layer slice (@DataJpaTest)

Use TestContainers (real PostgreSQL) when:
- Using PostgreSQL-specific features: JSONB columns, full-text search (tsvector), array columns, ILIKE queries
- Need to verify Hibernate dialect compatibility
- Testing stored procedures, triggers, or database functions
- Running critical integration tests before production deployment
- Testing database migration scripts (Flyway/Liquibase)

Decision in this project: H2 is sufficient because queries use standard JPA (derived methods, JPQL, pagination). No PostgreSQL-specific features are used. If the project adds JSONB product attributes or full-text product search, TestContainers would become necessary.

Compromise approach: Use H2 for @DataJpaTest (fast feedback) and TestContainers for @SpringBootTest (full confidence). This gives you both speed and reliability.
10. How does @WithMockUser create the security context?
@WithMockUser uses WithMockUserSecurityContextFactory (part of spring-security-test):

1. Reads annotation attributes: username (default "user"), password (default "password"), roles (default "USER"), authorities
2. Creates a SimpleGrantedAuthority list from roles/authorities (roles are automatically prefixed with "ROLE_")
3. Creates a User object: new org.springframework.security.core.userdetails.User(username, password, authorities)
4. Creates UsernamePasswordAuthenticationToken with the User as principal, password as credentials, and authorities
5. Sets this Authentication in SecurityContextHolder via SecurityContextHolder.getContext().setAuthentication(auth)

The key limitation: The principal is Spring Security's User class — a simple POJO with username, password, and authorities. It does NOT have custom fields like getId(), getEmail(), or getRole(). If your controller casts the principal to a custom class like UserPrincipal, you get ClassCastException.

This is exactly why our project created @WithMockPrincipal — it sets the principal to our custom UserPrincipal which wraps a real User entity with all our custom fields.
