1 Why Testing Matters in E-Commerce

In an e-commerce system, bugs don't just crash the application — they cost real money. A single production bug in the order flow can result in lost revenue, corrupted data, and permanently damaged customer trust.

The Cost of Bugs in Production

Bug TypeImpactReal-World Example
Order ProcessingOrders not created, duplicate charges, wrong amountsCustomer charged twice but order never placed — lost sale + refund cost + support cost
InventoryOverselling, underselling, negative stock100 customers buy same item, only 50 in stock — 50 cancellations, angry customers, bad reviews
AuthenticationUsers cannot login, sessions hijackedFlash sale starts, users get 401 errors — $1M revenue lost in 10 minutes
PaymentMoney deducted but order not created, double charges$500K in incorrect charges — fines, chargebacks, reputation damage
Data CorruptionInconsistent database state, orphaned recordsOrder items without orders, payments without order references

The Testing Pyramid

Testing Pyramid (Bottom to Top): ⧑ ⧒ ⧒ ⧒ E2E⧒ ← Few (smoke tests, critical flows) ⧒------⧒ ⧒ ⧒ ⧒Integration⧒ ← Some (API + DB combined) ⧒--------------⧒ ⧒ ⧒ ⧒ Unit Tests ⧒ ← Many (isolated, fast, reliable) ⧒--------------------⧒ - BASE: Unit tests (JUnit + Mockito) — thousands, milliseconds each - MIDDLE: Integration tests (@SpringBootTest + TestContainers) — hundreds, seconds each - TOP: End-to-end tests (Cypress / Selenium / REST Assured) — dozens, minutes each

2 Unit Testing with JUnit 5

JUnit 5 is the standard testing framework for Java applications. It is composed of three sub-projects:

JUnit 5 Architecture

ComponentDescription
JUnit PlatformThe foundation — launches test frameworks on the JVM. Integrates with Maven Surefire, Gradle, IDEs, and CI tools.
JUnit JupiterThe new programming model — contains all annotations (@Test, @BeforeEach, @Nested) and the assertion/assumption APIs.
JUnit VintageBackward compatibility layer — allows running JUnit 4 tests on the JUnit 5 platform.

Key Annotations

AnnotationPurposeExample
@TestMarks a method as a test case@Test void shouldReturnProduct() { ... }
@BeforeEachRuns before each test method — used for setup@BeforeEach void setUp() { mockProduct = new Product(); }
@AfterEachRuns after each test method — used for cleanup@AfterEach void tearDown() { productRepo.deleteAll(); }
@BeforeAllRuns once before all tests — must be static@BeforeAll static void init() { ... }
@AfterAllRuns once after all tests — must be static@AfterAll static void cleanup() { ... }

@DisabledTemporarily skip a test@Disabled("Until we fix the bug")

Test Lifecycle and Order

Tests should be independent — never rely on execution order. However, JUnit 5 provides @TestMethodOrder for ordered execution when absolutely needed:

@TestMethodOrder(MethodOrderer.OrderAnnotation.class)
class OrderedProductServiceTest {

    private static Product savedProduct;

    @Test
    @Order(1)
    void createProduct_shouldSucceed() {
        savedProduct = productService.createProduct(request);
        assertNotNull(savedProduct.getId());
    }

    @Test
    @Order(2)
    void getProduct_shouldReturnCreatedProduct() {
        Product found = productService.getProduct(savedProduct.getId());
        assertEquals(savedProduct.getId(), found.getId());
    }
}
⚠️ Caution: Ordered tests are a code smell. Tests should be independent. Use ordered tests only for integration scenarios where order is meaningful.

3 Mockito — Mocking Dependencies

Why Mock?

  • Isolate the unit under test — test only the service logic, not the repository, not the database
  • Control behavior — simulate success, failure, timeout, and edge cases that are hard to reproduce with real dependencies
  • Verify interactions — confirm that the service called the right methods with the right arguments
  • Speed — unit tests with mocks run in milliseconds, not seconds

Core Annotations

@ExtendWith(MockitoExtension.class)
class ProductServiceTest {

    @Mock
    private ProductRepository productRepository;

    @Mock
    private CategoryRepository categoryRepository;

    @InjectMocks
    private ProductService productService;

    @Test
    void getProductById_shouldReturnProduct() {
        Product mockProduct = new Product();
        mockProduct.setId(1L);
        mockProduct.setName("iPhone 15");
        when(productRepository.findById(1L)).thenReturn(Optional.of(mockProduct));

        Product result = productService.getProduct(1L);

        assertEquals("iPhone 15", result.getName());
        verify(productRepository).findById(1L);
    }
}

when().thenReturn() vs doThrow().when()

PatternWhen to UseExample
when(x).thenReturn(y)Mock returns a value — standard casewhen(repo.findById(1L)).thenReturn(Optional.of(product))
when(x).thenThrow(y)Mock throws exception on method callwhen(repo.save(any())).thenThrow(DataIntegrityViolationException.class)
when(x).thenAnswer(y)Dynamic response based on argumentswhen(repo.save(any())).thenAnswer(inv -> inv.getArgument(0))
doThrow(y).when(x).method()Mock void method to throw exceptiondoThrow(new RuntimeException()).when(repo).delete(any())
doNothing().when(x).method()Explicitly do nothing for void methoddoNothing().when(emailService).sendEmail(any())
doReturn(x).when(y).method()Alternative to when().thenReturn() (avoids side effects)doReturn(product).when(repo).save(any())

ArgumentMatchers

import static org.mockito.ArgumentMatchers.*;

// Match any value
when(productRepository.findById(anyLong())).thenReturn(Optional.of(product));
when(productRepository.save(any(Product.class))).thenReturn(product);

// Match specific values
when(productRepository.findById(eq(1L))).thenReturn(Optional.of(product));
when(categoryRepository.findByName(eq("Electronics"))).thenReturn(Optional.of(category));

// Complex matchers
when(productRepository.findAll(any(Specification.class), any(Pageable.class)))
    .thenReturn(new PageImpl<>(List.of(product)));

// String matchers
when(userRepository.findByEmail(contains("@gmail.com"))).thenReturn(Optional.of(user));
when(userRepository.findByEmail(startsWith("admin"))).thenReturn(Optional.of(admin));

verify() — Verifying Interactions

// Verify method was called exactly once (default)
verify(productRepository).findById(1L);

// Verify number of invocations
verify(productRepository, times(1)).findById(1L);
verify(productRepository, never()).delete(any());
verify(productRepository, atLeast(1)).findAll(any(Pageable.class));
verify(productRepository, atMost(3)).save(any(Product.class));

// Verify with timeout (useful for async)
verify(emailService, timeout(1000)).sendOrderConfirmation(any());

// Verify no other interactions happened
verifyNoMoreInteractions(productRepository);

// Verify order of interactions
InOrder inOrder = inOrder(productRepository, categoryRepository);
inOrder.verify(productRepository).findById(1L);
inOrder.verify(categoryRepository).findByName("Electronics");

BDDMockito — Given/WillReturn Pattern

BDD (Behavior-Driven Development) style uses given() instead of when() for more readable tests:

import static org.mockito.BDDMockito.*;

@ExtendWith(MockitoExtension.class)
class ProductServiceBDDTest {

    @Mock private ProductRepository productRepository;
    @InjectMocks private ProductService productService;

    @Test
    void getProductByName_shouldReturnProduct() {
        // Given
        Product product = new Product("MacBook Pro");
        given(productRepository.findByName("MacBook Pro"))
            .willReturn(Optional.of(product));

        // When
        Product found = productService.findByName("MacBook Pro");

        // Then
        assertThat(found.getName()).isEqualTo("MacBook Pro");
        then(productRepository).should().findByName("MacBook Pro");
    }

    @Test
    void getProductByName_shouldThrowWhenNotFound() {
        given(productRepository.findByName("NonExistent"))
            .willReturn(Optional.empty());

        assertThrows(ResourceNotFoundException.class,
            () -> productService.findByName("NonExistent"));
        then(productRepository).should().findByName("NonExistent");
    }
}
💡 Interview Tip: "when().thenReturn() is the standard Mockito pattern. BDDMockito.given().willReturn() is preferred for BDD-style tests. Both achieve the same result — choose based on team convention."

4 Testing Service Layer

The service layer contains all business logic. Testing it thoroughly is the highest priority because bugs here directly affect revenue and data.

Key Principles

  • Isolate from infrastructure — mock all repositories, external clients (Razorpay), and dependent services (EmailService)
  • Test every code path — happy path, validation errors, insufficient stock, not found, unauthorized, duplicate data
  • Test transaction behavior — if a service is @Transactional, ensure rollback happens on failure

6 Repository Testing with @DataJpaTest

@DataJpaTest loads only the JPA layer — entities, repositories, and data source. It uses an embedded H2 in-memory database by default, making tests fast and isolated.

Key Features

  • Auto-configures an embedded H2 database (or you can override with a real DB)
  • Scans only @Entity classes and Spring Data JPA repositories
  • Provides TestEntityManager for convenient test data setup
  • Each test is transactional — rolled back after execution
  • Does NOT load services, controllers, or web layer

7 Integration Testing with TestContainers

What is TestContainers?

TestContainers is a Java library that allows you to programmatically start Docker containers for your tests. Instead of using an in-memory H2 database for integration tests, you spin up a real PostgreSQL container that mirrors your production environment.

Why TestContainers?

ApproachProCon
H2 (In-Memory)Fast startup, no Docker neededDifferent SQL dialect, missing features (JSONB, full-text search), false confidence
TestContainers (PostgreSQL)Same database as production, real SQL, catches dialect issuesSlower startup (Docker), needs Docker installed, slightly longer test time
Embedded PostgreSQLNo Docker neededLimited features, not identical to real PostgreSQL

10 Mockito Best Practices & Pitfalls

✅ Do This

1. Don't Mock What You Don't Own

// BAD: Mocking third-party library class
RazorpayClient mockClient = mock(RazorpayClient.class);

// GOOD: Wrap in your own service and mock the wrapper
@Mock private RazorpayPaymentGateway paymentGateway;
// paymentGateway is YOUR interface/class, not Razorpay's

2. Don't Mock Value Objects

// BAD: Mocking a value object
Product product = mock(Product.class);
when(product.getName()).thenReturn("iPhone");
// product.equals() and product.hashCode() are broken!

// GOOD: Use real objects
Product product = new Product();
product.setName("iPhone");

3. Keep Mocks Simple

// BAD: Overly complex mock setup
when(repo.findByCategoryAndPriceBetweenAndActiveTrueOrderByNameDesc(
    any(), any(), any(), any())).thenReturn(...);

// GOOD: Simplify queries or test separately
// Test complex queries in @DataJpaTest, not in service unit tests

4. Avoid Over-Mocking

// BAD: 7 mocks for one test
@Mock private ServiceA a;
@Mock private ServiceB b;
@Mock private ServiceC c;
@Mock private ServiceD d;
@Mock private RepositoryE e;
@Mock private RepositoryF f;
@Mock private ClientG g;

// GOOD: Split tests or refactor service
// A service with 7+ dependencies should be refactored

5. Use @InjectMocks Carefully

// BAD: @InjectMocks with field injection
@Service
public class OrderService {
    @Autowired private OrderRepository orderRepository;
    @Autowired private ProductRepository productRepository;
}

// GOOD: Constructor injection (works better with mocks)
@Service
public class OrderService {
    private final OrderRepository orderRepository;
    private final ProductRepository productRepository;

    public OrderService(OrderRepository orderRepository,
                        ProductRepository productRepository) {
        this.orderRepository = orderRepository;
        this.productRepository = productRepository;
    }
}

6. Never Mock the System Under Test

// BAD: Mocking the class being tested
OrderService orderService = mock(OrderService.class);
when(orderService.placeOrder(any(), any())).thenReturn(something);

// GOOD: Use real instance with mocked dependencies
@InjectMocks private OrderService orderService;
@Mock private OrderRepository orderRepository;

⚠️ Common Pitfalls

PitfallWhy It's BadSolution
Mocking concrete classes instead of interfacesFragile — changes to concrete class break mocksPrefer mocking interfaces; use constructor injection
Using when().thenReturn() on void methodsWon't compile — void methods don't returnUse doNothing().when() or doThrow().when()
Not verifying interactionsTest passes but method was never calledUse verify() to confirm interactions
Too many verify() callsBrittle tests — internal refactoring breaks themVerify only key interactions, not every call
Mocking static methods without MockedStaticTest fails or mocks not appliedUse MockedStatic for static mocking
Not resetting mocks between testsTest pollution — one test affects anotherUse @Mock with @ExtendWith(MockitoExtension.class) for automatic reset
🔑 Golden Rule: "A test should mock only the direct dependencies of the class under test. If you're mocking classes that are dependencies of dependencies, your test design needs improvement."

11 Interview Questions — Beginner Level

Q1: What's the difference between a unit test and an integration test?
Answer:

Unit Test: Tests a single unit (class/method) in isolation. Dependencies are mocked. Fast (milliseconds). Focuses on business logic correctness.
Example: Testing OrderService.placeOrder() with mocked OrderRepository and ProductRepository.

Integration Test: Tests multiple components working together. Real dependencies (database, external services). Slower (seconds). Focuses on component interaction correctness.
Example: Testing the full "Register -> Add to Cart -> Place Order" flow against a real PostgreSQL database via TestContainers.

Key difference: Isolation vs interaction. Unit tests tell you WHAT broke. Integration tests tell you WHERE the components don't fit together.
Q2: What is Mockito and why use it?
Answer: Mockito is a mocking framework for Java unit tests. It creates mock objects that simulate the behavior of real dependencies.

Why use it:
1. Isolation — Test only the class under test, not its dependencies
2. Speed — Mocked tests run in milliseconds (no DB, no network)
3. Control — Simulate edge cases: exceptions, timeouts, null returns
4. Verification — Confirm that methods were called with correct arguments

Example: You want to test OrderService calls PaymentService when placing an order. Without Mockito, you need a real Payment integration. With Mockito, you mock PaymentService and verify the call.
Q3: Explain @Mock vs @InjectMocks in Mockito.
Answer:

@Mock creates a mock instance of a dependency. The mock has no real behavior — every method returns default values (null, 0, false) unless you stub it.
@Mock private ProductRepository productRepository;

@InjectMocks creates an instance of the class under test and injects the mocks into it. It tries constructor injection first, then setter, then field injection.
@InjectMocks private ProductService productService;

How they work together: @Mock creates the fake dependencies. @InjectMocks creates the real object and injects those fakes into it.
Q4: What is JUnit 5 @DisplayName used for?
Answer: @DisplayName provides a custom, human-readable name for a test class or test method. Instead of seeing testProductSearchWithEmptyQueryReturnsEmpty() in test reports, you see "Should return empty results when search query is empty".

Benefits: Readability (non-technical stakeholders understand reports), Documentation (tests become executable specifications), Better CI/CD reporting.

Best practice: Use format: "Should [expected behavior] when [condition/scenario]".
Q5: How do you test exceptions in JUnit 5?
Answer: Use assertThrows().

// Assert that exception is thrown
assertThrows(ResourceNotFoundException.class,
    () -> productService.getProduct(999L));

// Assert + capture exception for further assertions
ResourceNotFoundException ex = assertThrows(
    ResourceNotFoundException.class,
    () -> productService.getProduct(999L));
assertEquals("Product not found with id: 999", ex.getMessage());
Key points:
1. If no exception is thrown, the test FAILS
2. If a DIFFERENT exception is thrown, the test FAILS
3. assertThrows returns the exception instance for additional assertions
4. For testing NO exception: assertDoesNotThrow(() -> productService.getProduct(1L));
Q6: What is the testing pyramid?
Answer: The testing pyramid describes the ideal distribution of tests:

Base — Unit Tests (70-80%): Test individual classes/methods in isolation. Milliseconds each. Caught: Business logic bugs.

Middle — Integration Tests (15-25%): Test interactions between components. Seconds each. Caught: Component interaction bugs.

Top — End-to-End Tests (5-10%): Test the complete system. Minutes each. Caught: System-level integration bugs.

Key principle: Write more unit tests, fewer integration tests, and even fewer E2E tests. The pyramid ensures fast feedback while maintaining confidence.
Q7: What does @WebMvcTest do?
Answer: @WebMvcTest is a test slice annotation that loads only the web layer of a Spring Boot application. It auto-configures MockMvc and loads only controllers, filters, interceptors, converters, and security configuration.

What it does NOT load: Services, repositories, full application context, database configuration.

Why use it: Speed (loads only web beans), Focus (tests only HTTP concerns), Isolation (services are mocked with @MockBean).
Q8: What does @DataJpaTest do?
Answer: @DataJpaTest is a test slice annotation that loads only the JPA/data layer. It auto-configures an embedded H2 database, scans @Entity classes, and enables Spring Data JPA repositories.

What it does NOT load: Services, controllers, web layer, security configuration.

Key features:
1. Each test is transactional — automatically rolls back after execution
2. Provides TestEntityManager for convenient test data setup
3. Can be configured to use a real database instead of H2
Q9: How do you verify a method was called with Mockito?
Answer: Use Mockito's verify() method.

// Verify method was called exactly once (default)
verify(productRepository).findById(1L);

// Verify with call count
verify(productRepository, times(1)).findById(1L);
verify(productRepository, never()).delete(any());
verify(productRepository, atLeast(1)).save(any());
verify(productRepository, atMost(3)).findAll();

// Verify with timeout (for async code)
verify(emailService, timeout(1000)).sendOrderConfirmation(any());

// Verify order of interactions
InOrder inOrder = inOrder(productRepository, categoryRepository);
inOrder.verify(productRepository).findById(1L);
inOrder.verify(categoryRepository).findByName("Electronics");

// Verify no unexpected interactions
verifyNoMoreInteractions(productRepository);
Important: verify() is most useful for checking side effects (save, sendEmail). For methods that return values, assertions on the return value are more important.
Q10: What is TDD (Test-Driven Development)?
Answer: TDD is a development approach where you write tests before production code. The cycle is:

Red -> Green -> Refactor

1. Red: Write a failing test for the feature you want to implement
2. Green: Write the MINIMUM production code to make the test pass
3. Refactor: Clean up the code while keeping the tests green

Example:
// Step 1 (RED): Write test first
@Test
void applyDiscount_shouldApply10Percent_ForOrdersAbove1000() {
    Order order = new Order(new BigDecimal("1500.00"));
    orderService.applyDiscount(order);
    assertEquals(new BigDecimal("1350.00"), order.getTotalAmount());
}

// Step 2 (GREEN): Write minimum code
public void applyDiscount(Order order) {
    if (order.getTotalAmount().compareTo(new BigDecimal("1000")) > 0) {
        order.setTotalAmount(order.getTotalAmount()
            .multiply(new BigDecimal("0.9")));
    }
}
Benefits: Better design (code must be testable), documentation (tests define behavior), confidence (every line is tested), fewer bugs.

Q12: What is the difference between when().thenThrow() and doThrow().when()?
Answer:

when().thenThrow() — Used for methods that return a value:
when(productRepository.findById(anyLong()))
    .thenThrow(new DatabaseException("Connection failed"));
doThrow().when() — Used for void methods (no return value):
doThrow(new RuntimeException())
    .when(productRepository).delete(any(Product.class));
Key difference: when().thenThrow() evaluates the method call inside when(), which requires the method to return something. Void methods can't be used inside when(), so use doThrow().when() instead.

Q14: What is TestContainers and when should you use it?

    // Then
    assertThat(result).isNotNull();

}

Q23: Explain test slicing and when to use each slice vs full context.
Answer: Test slices load only specific parts of the Spring context:

@WebMvcTest: Web layer only (controllers, filters, security). Use for HTTP mapping, validation, security. Mock all services.

@DataJpaTest: JPA layer only (entities, repositories). Use for custom queries, pagination, entity mapping. Auto-rolled back transactions.

@RestClientTest: REST client beans. Use for testing external API clients (Razorpay) with WireMock.

@JsonTest: JSON serialization only. Use for testing DTO serialization/deserialization.

@SpringBootTest (full context): Everything. Use only for integration tests that genuinely need all layers. Slower but catches cross-layer issues.

Rule of thumb: Use slices for 90% of tests (fast, focused). Use full context for 10% (integration, E2E flows).

Q26: How to avoid flaky tests in CI/CD?
Answer: Flaky tests (sometimes pass, sometimes fail) destroy trust in the test suite. Prevention strategies:

1. Avoid shared mutable state — Each test should create and clean up its own data
2. Use TestContainers, not embedded databases — H2 behaves differently than PostgreSQL
3. Add timeouts to all async operations — verify(service, timeout(5000)).method();
4. Use fixed data, not random — Random dates, random strings can cause edge cases
5. Don't depend on system clock — Use Clock injection or MockedStatic for time
6. Avoid thread.sleep() — Use CountDownLatch, CompletableFuture, or Awaitility
7. Database cleanup between tests — @Transactional rollback or @Sql cleanup scripts
8. Retry flaky tests (last resort) — Use @RetryingTest or Surefire rerunFailingTestsCount

Q28: What is mutation testing and how does it improve test quality?
Answer: Mutation testing introduces small changes (mutations) into your code and checks if your tests detect them. It measures the quality of your tests, not just coverage.

Examples of mutations:
• if (x > 0) mutated to if (x >= 0) — boundary condition
• if (condition) mutated to if (true) — removed condition
• return a + b; mutated to return a - b; — arithmetic operator
• list.add(item) mutated to list.remove(item) — method call

How it improves quality:
1. Exposes missing assertions — Code is executed but not verified
2. Exposes weak assertions — Assertions that pass even with wrong output
3. Shows untested branches — Conditions that tests don't fully cover

Real example:
// Production code
public boolean isEligibleForDiscount(Order order) {
    return order.getTotalAmount().compareTo(new BigDecimal("1000")) > 0;
}

// Weak test (passes mutation testing poorly)
@Test
void testDiscount() {
    order.setTotalAmount(new BigDecimal("1500"));
    assertTrue(orderService.isEligibleForDiscount(order));
    // Mutation: change > to >= — test still passes! Bug hidden!
}

// Strong test (catches the mutation)
@Test
void testDiscount_boundary() {
    order.setTotalAmount(new BigDecimal("1000"));
    assertFalse(orderService.isEligibleForDiscount(order));
    // Mutation: change > to >= — test FAILS! Bug caught!
}
Tools: Pitest for Java. Aim for 80%+ mutation score.
Q29: How to test idempotency in payment endpoints?
Answer: Idempotency means sending the same request multiple times produces the same result. For payments, this is critical — duplicate requests must not charge the customer twice.

Implementation: Clients send an Idempotency-Key header. The server stores the key and response. Duplicate requests return the cached response.

Test scenarios:
@Test
@DisplayName("Same idempotency key returns cached response")
void duplicateRequest_shouldReturnSameResponse() {
    String idempotencyKey = UUID.randomUUID().toString();

    // First request
    mockMvc.perform(post("/api/payments")
            .header("Idempotency-Key", idempotencyKey)
            .contentType(MediaType.APPLICATION_JSON)
            .content(requestBody))
        .andExpect(status().isOk())
        .andExpect(jsonPath("$.status").value("created"));

    // Second request with same key — should return cached result
    mockMvc.perform(post("/api/payments")
            .header("Idempotency-Key", idempotencyKey)
            .contentType(MediaType.APPLICATION_JSON)
            .content(requestBody))
        .andExpect(status().isOk())
        .andExpect(jsonPath("$.status").value("created"));
    // Verify payment was processed only ONCE
    then(paymentService).should(times(1)).processPayment(any());
}
