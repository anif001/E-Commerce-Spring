1 High-Level Architecture

This project follows a 4-layer monolith architecture with strict layer separation. Each layer has a single responsibility and communicates only with the layer directly below it. This enforces Separation of Concerns and makes the codebase testable, maintainable, and ready for future microservice extraction.

The 4-Layer Architecture

HTTP/HTTPS | +-------------------------------v-------------------------------+ | CONTROLLER LAYER (REST) | | @RestController, @RequestMapping, @Valid, @PreAuthorize | | AuthController | ProductController | OrderController | | CartController | PaymentController | CategoryController | | Responsibilities: HTTP mapping, validation, response shape | +-------------------------------+-------------------------------+ | calls v +-------------------------------+-------------------------------+ | SERVICE LAYER (Business Logic) | | @Service, @Transactional | | AuthService | ProductService | CartService | OrderService | | PaymentService | EmailService | | Responsibilities: Business rules, orchestration, DTO map | +-------------------------------+-------------------------------+ | calls v +-------------------------------+-------------------------------+ | REPOSITORY LAYER (Data Access) | | @Repository, Spring Data JPA | | UserRepository | ProductRepository | CartRepository | | OrderRepository | CategoryRepository | PaymentRepository | | Responsibilities: SQL queries, pagination, sorting | +---------------+---------------+-------------------------------+ | | v v +-----------------------------+ +-----------------------------+ | PostgreSQL | | Redis | | ACID-compliant primary DB | | In-memory cache layer | | Users, Products, Orders, | | Product cache, | | Cart Items, Categories | | Refresh tokens, | | Payments | | Rate limiting counters | +-----------------------------+ +-----------------------------+

Layer Responsibilities (Top to Bottom)

LayerAnnotation / ClassResponsibility
Controller @RestController Receives HTTP requests, validates input (@Valid), enforces security (@PreAuthorize), delegates to service, returns shaped response (DTO). Never contains business logic.
Service @Service Contains all business rules, orchestrates calls to repositories and external services (Razorpay, Email). Manages transactions via @Transactional. Maps entities to DTOs and vice versa.
Repository @Repository Spring Data JPA interface. Defines derived query methods, custom @Query JPQL/SQL, pagination (Pageable), sorting, and locking (@Lock). Never contains business logic.
Entity / Model @Entity JPA entity mapping to database tables. Defines relationships (@ManyToOne, @OneToMany), column mappings, validation constraints, lifecycle callbacks (@PrePersist).
Key Rule: Controller never accesses Repository directly. Service never handles HttpServletRequest. This strict layering means each layer can be tested in isolation and replaced independently.

2 Tech Stack Decisions

Every technology in this stack was chosen deliberately over alternatives. Here is the reasoning a senior architect would provide in an interview.

2.1 Why Spring Boot (Not Node.js / Django)

FeatureHow Spring Boot DeliversWhy It Matters
Auto-Configuration @SpringBootApplication triggers classpath-scanning auto-config. Spring detects H2, PostgreSQL, Redis on classpath and configures beans automatically. spring-boot-starter-data-jpa auto-configures DataSource, EntityManagerFactory, TransactionManager. Zero boilerplate configuration. Focus on business code, not infrastructure wiring.
Dependency Injection @Autowired, @Inject, constructor injection. Spring container manages bean lifecycle, scopes (singleton, prototype, request), and proxy creation for @Transactional and @Cacheable. Loose coupling. Services depend on abstractions (interfaces), not concretions. Swap implementations without changing callers.
Declarative Transactions @Transactional on service methods creates a proxy that wraps the method in a JDBC transaction. Supports propagation (REQUIRED, REQUIRES_NEW), isolation levels, rollback rules. Critical for e-commerce: deduct inventory + create order + clear cart must be atomic. Without @Transactional, partial failures corrupt data.
Spring Security Filter chain architecture. SecurityFilterChain bean configures which URLs are public, which require auth, which roles are permitted. JWT authentication via custom OncePerRequestFilter. Comprehensive, battle-tested security. Method-level @PreAuthorize("hasRole('ADMIN')") for fine-grained access control.
Testing @SpringBootTest, @WebMvcTest, @DataJpaTest — slice tests load only relevant beans. MockMvc for controller testing, TestContainers for integration. Fast, focused tests. Unit test services in milliseconds, integration test against real PostgreSQL.
Interview Answer: "Spring Boot for e-commerce because of auto-configuration that reduces boilerplate, declarative @Transactional that ensures data consistency for financial operations, Spring Security's comprehensive filter chain for JWT auth, and slice-testing support that makes tests fast and reliable."

2.2 Why PostgreSQL (Not MySQL / MongoDB)

FeaturePostgreSQLWhy It Wins
ACID Compliance Full ACID out of the box — every operation is transactional. No engine switching. MySQL requires InnoDB for ACID; MyISAM (default in older versions) does not support transactions or foreign keys. PostgreSQL is always ACID.
JSONB Binary JSON with full indexing via GIN indexes. Query with @> (contains), ? (exists), #>> (path) operators. Product attributes vary by category (phone has RAM, clothing has size). JSONB stores this flexibly while remaining queryable and indexable — no need for EAV pattern.
Full-Text Search tsvector / tsquery with GIN indexes. Weighted search (name=A weight, description=D weight), stemming, ranking. Built-in search for product catalog up to ~100K products. No need for Elasticsearch initially. SELECT * FROM products WHERE to_tsvector('english', name) @@ to_tsquery('english', 'iphone').
MVCC Multi-Version Concurrency Control. Readers never block writers and vice versa. Each transaction sees a snapshot of data as of its start time. Critical for e-commerce: 1000 concurrent users browsing products while 10 are placing orders. No read locks, no deadlocks between reads and writes.
Partial / Expression Indexes CREATE INDEX ON products (stock) WHERE active = true. CREATE INDEX ON products (LOWER(name)). Index only active products (smaller index). Case-insensitive search without function overhead.

2.3 Why Redis (Not Memcached)

FeatureRedisMemcached
Data StructuresString, List, Set, SortedSet, Hash, Bitmap, HyperLogLog, Streams, GeospatialString only
PersistenceRDB (point-in-time snapshots) + AOF (append-only log, every write)None — pure cache, data lost on restart
Pub/SubNative publish/subscribe for real-time notificationsNot supported
TransactionsMULTI/EXEC + Lua scripting for atomic operationsNot supported
TTLPer-key expiry with EXPIRE / TTL commandsPer-key expiry

Why Redis wins: We use SortedSets for product rating leaderboards, Hashes for product cache (fields: name, price, stock), Pub/Sub for cache invalidation events, and persistence to survive restarts without cache cold-start. Memcached is a simpler cache; Redis is a data structure server with cache capabilities.

2.4 Why JWT (Not Sessions)

AspectJWT (Stateless)Sessions (Stateful)
Horizontal ScalingAny server validates token independently — no shared state needed. Add instances freely.Requires sticky sessions (load balancer affinity) or centralized session store (Redis). Both add complexity.
Mobile SupportToken stored in secure storage (Keychain/Keystore), sent in Authorization: Bearer <token> header. Works natively.Cookies are designed for browsers. Mobile apps need cookie management libraries, interceptors, and CSRF handling.
MicroservicesPass JWT between services. Each service verifies signature independently using the shared secret or public key. No session store call.Each service must call the central session store on every request — latency adds up, session store becomes a bottleneck.
PerformanceSignature verification is ~1ms HMAC-SHA256. No network call.Session lookup to Redis/DB adds 3-5ms per request across all services.
RevocationHarder — token valid until expiry. Mitigated with short-lived access tokens (15min) + refresh tokens (7 days) + Redis blacklist.Trivial — delete session from store.

2.5 Why Docker

  • Consistency Across Environments: The same Docker image runs on every developer's machine, in CI, staging, and production. Eliminates "works on my machine" bugs caused by Java version differences, OS differences, or dependency version mismatches.
  • CI/CD Pipeline: docker build produces a single immutable artifact. GitHub Actions builds, tests, then builds the Docker image. The same image is promoted through environments: CI test → staging → production. No rebuilds between stages.
  • Dependency Isolation: PostgreSQL 15, Redis 7, and the Spring Boot app each run in their own container. No port conflicts (each uses its own port namespace), no version conflicts (Java 17 in app container, no dependency on host Java).
  • Team Onboarding: New developer runs docker compose up — the entire stack (PostgreSQL + Redis + app) starts in 30 seconds. No manual installation of databases, no environment setup scripts.

3 System Design Concepts

3.1 Monolith vs Microservices Tradeoffs

DimensionMonolith (Our Choice)Microservices
Development SpeedFast for teams of 1-10. Single codebase, single build, single deploy.Slow initial velocity. Multiple repos, CI/CD pipelines, coordination overhead.
TestingSimple end-to-end tests. Start one JAR, test against real DB. No service mocking needed.Complex. Each service needs contract tests, integration tests with mocked downstream services, and end-to-end tests across all services.
DeploymentOne WAR/JAR. Rollback is a single command.Multiple services to coordinate. Canary deployments, blue-green, or rolling updates needed.
ScalingScale the entire application. Cannot scale only the order-processing component.Scale each service independently. Order service can have 10 instances while Auth service has 2.
Team AutonomyShared codebase requires coordination, merge conflicts, aligned release cadence.Each team owns their service independently. API contracts are the only coordination point.
Operational OverheadLow. One app to monitor, one set of logs, one JVM tuning.High. Service mesh, distributed tracing (Jaeger), centralized logging (ELK), container orchestration (Kubernetes).
Decision: Start monolith, organize for microservices. Use feature-based packages, no circular dependencies, interface-based service boundaries. When the team grows to 3+ teams or a domain requires independent scaling, extract that bounded context into a service using the Strangler Fig pattern.

3.2 Caching Strategies — Cache-Aside with TTL

This project uses Cache-Aside (Lazy Loading) as the primary caching pattern:

// Cache-Aside Pattern (pseudocode)
Product getProduct(Long id) {
    String cacheKey = "product:" + id;

    // 1. Check Redis cache
    Product cached = redisTemplate.opsForValue().get(cacheKey);
    if (cached != null) return cached;  // Cache HIT

    // 2. Cache MISS → query database
    Product product = productRepository.findById(id)
        .orElseThrow(() -> new ResourceNotFoundException("Product not found"));

    // 3. Populate cache with TTL
    redisTemplate.opsForValue().set(cacheKey, product, 30, TimeUnit.MINUTES);

    return product;
}
Cache StrategyUse CaseProsCons
Cache-Aside (this project)Product details, categories, search resultsOnly caches what is requested. Simple to implement. TTL handles staleness.First request always slow (cache miss). Stale data until TTL expires.
Write-ThroughInventory counts (consistency critical)Cache always up-to-date. No stale data served.Slower writes. Caches data that may never be read.
Write-BehindPage view counters, analytics eventsExtremely fast writes. Batched DB persistence.Risk of data loss if cache crashes before persistence.

3.3 Horizontal Scaling

Three design decisions enable horizontal scaling:

  1. Stateless JWT Authentication: No session state on servers. Any instance can handle any request. Add instances behind a load balancer with zero configuration.
  2. Stateless Spring Boot Application: No local file storage (images go to S3/CDN). No in-memory session storage. Application state lives only in PostgreSQL and Redis — both external and independently scalable.
  3. Docker Containerization: Each instance runs in an identical container. Orchestration (Docker Compose for dev, Kubernetes for production) manages scaling: kubectl scale deployment ecommerce-app --replicas=10.

3.4 Performance Optimization

TechniqueImplementationImpact
Connection PoolingHikariCP (default in Spring Boot 3.x). Configured in application.yml with maximum-pool-size: 10, minimum-idle: 5, connection-timeout: 30000.Reuses database connections instead of opening a new TCP connection per request. Reduces DB connection latency from ~20ms to ~1ms.
PaginationSpring Data Pageable interface. All list endpoints accept ?page=0&size=20&sort=name,asc. SQL translates to LIMIT 20 OFFSET 0.Prevents loading thousands of records into memory. A product catalog of 100K items returns 20 at a time — response size is ~2KB instead of ~20MB.
Database IndexingIndexes on product(name), product(category_id), order(user_id), order_item(order_id). Composite indexes for common query patterns.Turns full table scans into index seeks. A query on 1M products without index takes ~500ms; with B-tree index, ~2ms.
Redis CachingProduct details cached with 30-min TTL. Category tree cached with 1-hour TTL.Reads that hit Redis return in ~5ms instead of ~200ms from PostgreSQL. Reduces DB load by 90%+ for read-heavy workloads.

3.5 Security Architecture

// Security Configuration (SecurityConfig.java)
@Configuration
@EnableWebSecurity
@EnableMethodSecurity  // Enables @PreAuthorize on controller methods
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .csrf(csrf -> csrf.disable())  // Stateless JWT — no CSRF needed
            .sessionManagement(sm -> sm.sessionCreationPolicy(STATELESS))
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/auth/**").permitAll()           // Register, Login — public
                .requestMatchers("/api/products/**").permitAll()       // Product listing — public
                .requestMatchers("/api/admin/**").hasRole("ADMIN")    // Admin only
                .anyRequest().authenticated()                         // Everything else needs auth
            )
            .addFilterBefore(jwtAuthenticationFilter, UsernamePasswordAuthenticationFilter.class);
        return http.build();
    }
}
Security LayerImplementationWhy
JWT Filter ChainJwtAuthenticationFilter extends OncePerRequestFilter. Extracts token from Authorization: Bearer <jwt>, validates signature + expiry, sets SecurityContextHolder.Every authenticated request passes through this filter. No session lookup needed — token is self-validating.
Role-Based Access@PreAuthorize("hasRole('ADMIN')") on controller methods. Three roles: CUSTOMER, SELLER, ADMIN.Method-level security at the controller layer. Guards against calling admin endpoints from customer accounts even if JWT is valid.
Password HashingBCryptPasswordEncoder with strength=10. Salt is automatically generated and embedded in the hash.BCrypt is deliberately slow (~50ms per hash) to resist brute-force attacks. Spring Security's PasswordEncoder interface allows switching to SCrypt or Argon2 without changing service code.
CORS@CrossOrigin or CorsConfigurationSource bean. Allows requests from known frontend origins.Prevents unauthorized domains from accessing APIs via browser CORS policy.

Package Design Principles

PrincipleHow It Is Enforced
Separation of ConcernsEach package has a single responsibility. Controllers only handle HTTP. Services only contain business logic. Repositories only access data. DTOs never leak into entities or vice versa.
Dependency Rule (Clean Architecture)Controller depends on Service (interface). Service depends on Repository (interface). Repository depends on Entity. Outer layers depend on inner layers — never the reverse.
No Circular DependenciesService A never imports Service B if Service B imports Service A. Enforced via architecture tests (@ArchTest with ArchUnit).
DTO Mapping at Service BoundaryServices convert entities to DTOs and vice versa. Controllers never see entities. This decouples API contract from database schema.
Interface-Based Service AccessServices implement interfaces. Controllers depend on interfaces, not concrete classes. This enables mocking in tests and alternative implementations.
Interview Tip: "The package structure follows Clean Architecture principles. The controller layer adapts HTTP to the application. The service layer contains use cases. The repository layer handles persistence. This structure makes the codebase testable, maintainable, and ready for microservice extraction — each bounded context is already isolated in its own packages."

6 Interview Q&A

Q1: Why did you choose a monolith architecture for this e-commerce project? When would you switch to microservices?
Answer: We start with a monolith because our team is small (2-5 developers), we need to ship an MVP quickly, and the operational overhead of microservices (distributed transactions, service discovery, API gateways, distributed tracing) would slow us down without providing measurable benefit.

We would extract to microservices when:
  • The team grows to 3+ independent feature teams
  • A specific bounded context requires independent scaling (e.g., Product Service gets 10x more traffic than User Service)
  • Deployment velocity differs (Order Service needs to deploy daily, Auth Service deploys weekly)
Crucially, we organize the monolith as if it will become microservices — feature-based packages, no shared databases between domains, interface-based service boundaries. This makes the eventual extraction a mechanical exercise, not a rewrite.
Q2: Explain the caching strategy used in this project. Why cache-aside instead of write-through?
Answer: We use cache-aside with TTL. The application code explicitly checks Redis before querying PostgreSQL. On cache miss, it queries the database, populates Redis, and returns the result. A TTL of 10-30 minutes for product data handles staleness automatically.

Why not write-through? Write-through updates the cache on every database write. For product data that is read 1000x more often than written, write-through wastes cache writes on data that may not be read again before the next update. Additionally, write-through makes writes slower (must wait for both DB + Redis).

Exception: Inventory counts use a write-through-like pattern with Redis DECR for atomic stock deduction, because serving stale inventory (saying "in stock" when it is not) causes overselling — a direct revenue loss.
Q3: JWT vs Sessions — you chose JWT. What specific problems does JWT solve for e-commerce, and how do you handle its revocation weakness?
Answer: JWT solves three specific problems for e-commerce:
  1. Horizontal scaling: During flash sales, we auto-scale from 3 to 30 app instances. With JWT, adding instances requires zero configuration — every instance validates tokens independently. With sessions, we would need sticky sessions (load balancer affinity) or a centralized Redis session store, both adding complexity and latency.
  2. Mobile clients: 60%+ of e-commerce traffic comes from mobile apps. JWT works natively with mobile (store token in Keychain/Keystore, send in Authorization header). Sessions rely on cookies which are awkward on mobile.
  3. Microservices-ready: When we eventually extract services, JWT passes between them. Each service validates the signature independently without calling a shared session store.
Revocation strategy: Access tokens expire in 15 minutes (short window of vulnerability). Refresh tokens expire in 7 days and are stored in Redis — on logout, we delete the refresh token, preventing new access tokens from being issued. For immediate blacklisting, we maintain a short-lived Redis blacklist of revoked token IDs (JTI claim).
Q4: Walk me through the complete request flow when a customer places an order. What happens at each layer?
Answer: The flow spans 7 layers:
  1. Client sends POST /api/orders with JWT in Authorization header and shipping address in body.
  2. Load Balancer receives request, checks health endpoint (/actuator/health), forwards to an available app instance.
  3. Spring Security Filter Chain: JwtAuthenticationFilter extracts JWT from the Bearer header, verifies HMAC-SHA256 signature, checks expiry, parses userId and roles from claims, sets SecurityContextHolder. Invalid JWT returns 401.
  4. DispatcherServlet maps POST /api/orders to OrderController.placeOrder()
  5. OrderController: @PreAuthorize("hasRole('CUSTOMER')") checks authorization, @Valid validates request body, then calls orderService.placeOrder(userId, request)
  6. OrderService (inside @Transactional): fetches cart with items, validates stock for each item, deducts inventory with @Lock(PESSIMISTIC_WRITE), creates Order + OrderItem entities, clears cart, calls Razorpay for payment, triggers async email.
  7. PostgreSQL executes all writes in a single database transaction. If any step fails, everything rolls back — no partial orders, no phantom inventory deductions.
Response: HTTP 201 with order ID, status, total, and Razorpay order ID for payment on the client side.
Q5: Explain how Spring Boot auto-configuration works. How does @SpringBootApplication know to configure a DataSource for PostgreSQL without explicit configuration?
Answer: @SpringBootApplication is a composed annotation: @SpringBootConfiguration + @EnableAutoConfiguration + @ComponentScan.

Auto-configuration mechanism:
  1. Spring Boot scans the classpath for spring.factories files in every JAR under META-INF/spring/. These files list auto-configuration classes.
  2. Each auto-configuration class is annotated with @ConditionalOnClass, @ConditionalOnMissingBean, @ConditionalOnProperty etc. These conditions determine whether the configuration is applied.
  3. For PostgreSQL: When spring-boot-starter-data-jpa and postgresql driver are on the classpath, DataSourceAutoConfiguration checks for javax.sql.DataSource on classpath. If no DataSource bean is defined, it reads spring.datasource.* properties from application.yml and creates a HikariDataSource with those properties.
  4. Similarly, JpaRepositoriesAutoConfiguration detects Spring Data JPA on classpath, looks for @EnableJpaRepositories (which @SpringBootApplication implicitly enables), and creates EntityManagerFactory and PlatformTransactionManager beans.
In short: Auto-configuration uses conditional bean registration based on classpath dependencies and property values. You get a fully configured PostgreSQL + Hibernate stack from just a starter dependency and connection properties in application.yml.
Q6: How does @Transactional work internally in Spring? What happens if a nested method call within the same class is also @Transactional?
Answer: @Transactional works through AOP proxies:
  1. When Spring detects @Transactional on a bean, it creates a JDK dynamic proxy (or CGLIB proxy if the class does not implement an interface) wrapping the bean.
  2. When a method on the bean is called through the proxy, the proxy intercepts the call, checks if the method has @Transactional, and if so:
  3. Retrieves the transaction manager (e.g., JpaTransactionManager)
  4. Calls transactionManager.getTransaction() — starts a JDBC connection, sets autoCommit=false, begins transaction
  5. Invokes the actual target method
  6. On success: transactionManager.commit() — calls connection.commit(), closes connection
  7. On exception (matching rollbackFor): transactionManager.rollback() — calls connection.rollback()
Self-invocation problem: If method A (transactional) in the same class calls method B (also @Transactional), method B's @Transactional is ignored. This is because the call goes through this.methodB() — directly on the target object, not through the proxy. The proxy is only used when the method is called from outside the class.

Solutions: (a) Inject the bean into itself with @Autowired and call through that reference, (b) use AopContext.currentProxy(), or (c) extract method B into a separate @Service bean. The cleanest approach is (c) — keep transaction boundaries at the service layer, each service calling other services that have their own transactions.
Q7: What is connection pooling and why is HikariCP the default? How do you configure it for an e-commerce workload?
Answer: Connection pooling reuses database connections instead of opening a new TCP connection (handshake + auth) for every request. Opening a connection takes 15-30ms; reusing a pooled connection takes <1ms.

Why HikariCP: It is the fastest pool available — benchmarks show it is 2-3x faster than Tomcat JDBC pool and C3P0. Optimizations include: (a) use of FastList instead of ArrayList for statement cleanup, (b) optimized proxy classes that minimize bytecode size, (c) no synchronized blocks in the hot path.

Configuration for e-commerce: 
spring:
  datasource:
    hikari:
      maximum-pool-size: 20        # Max connections in pool
      minimum-idle: 5              # Keep 5 ready
      idle-timeout: 300000         # 5 minutes idle before eviction
      connection-timeout: 30000    # Wait 30s for a connection before throwing
      max-lifetime: 1800000        # 30 minutes max connection lifetime
      pool-name: EcommercePool
Sizing rule: A good starting point is pool_size = ((core_count * 2) + effective_spindle_count). For a typical 4-core, 1-disk server: pool of 9-10. We set 20 conservatively because we also have Redis cache reducing DB load by 90%+. Each connection uses ~1MB RAM, so 20 connections = ~20MB overhead — negligible.
Q8: Why PostgreSQL over MongoDB for an e-commerce platform? When would MongoDB be a better choice?
Answer: PostgreSQL is the right choice for an e-commerce backend because:
  • ACID transactions are non-negotiable for order + payment processing. MongoDB added multi-document transactions in v4.0 (2018), but they have higher overhead and limited throughput compared to PostgreSQL's MVCC-based transactions.
  • Relational integrity: Orders reference users, order_items reference products and orders. Foreign keys ensure referential integrity — no orphan records. MongoDB has no foreign keys; application-level referential integrity is error-prone.
  • JSONB: PostgreSQL provides the flexibility of NoSQL (store variable product attributes as JSON) while keeping relational query power (JOIN, GROUP BY, aggregation). Best of both worlds.
  • Full-text search: Built-in tsvector/tsquery with ranking, stemming, and phrase search. Sufficient for 100K products without adding Elasticsearch.
When to choose MongoDB: (a) Product catalog with highly variable, deeply nested attributes that are rarely joined, (b) Session storage / logging (write-heavy, no joins), (c) When schema flexibility is more important than relational integrity. For our core transactional data, PostgreSQL is the clear winner.
Q9: How do you handle concurrent inventory deduction during a flash sale? Explain pessimistic vs optimistic locking.
Answer: For our scale, we use pessimistic locking with @Lock(PESSIMISTIC_WRITE) on the inventory query. This executes SELECT ... FOR UPDATE which acquires an exclusive row-level lock — other transactions wait their turn to read/update the same row.

Pessimistic Locking: 
@Lock(PESSIMISTIC_WRITE)
@Query("SELECT p FROM Product p WHERE p.id = :id")
Optional<Product> findByIdWithLock(@Param("id") Long id);
Pros: Guarantees no overselling. Simple to implement.
Cons: All concurrent buyers queue up sequentially. If 1000 people buy the same item, they wait in line. Slow throughput.

Optimistic Locking: 
// Add @Version column to Product entity
@Version
private Long version;

// Update checks version
UPDATE product SET stock = stock - ?, version = version + 1
WHERE id = ? AND version = :expectedVersion
Pros: No locks. Higher throughput. Retry on version conflict.
Cons: High retry rate (many users see "in stock" but fail at order).

Recommended for flash sales: Use Redis DECR for atomic counting (microseconds, no DB locks). Only the first N successful DECR calls get queued for order processing. This combines the atomicity of pessimistic locking with the speed of in-memory operations.
Q10: Why start with a monolith when most companies use microservices? Isn't a monolith considered outdated?
Answer: Starting with a monolith is not outdated — it is the recommended approach by industry leaders (Martin Fowler, Sam Newman, and the teams at Amazon, Netflix, and Shopify who actually migrated from monoliths to microservices).

The key insight: Microservices are an operational pattern, not an architectural one. They solve organizational problems (team autonomy, independent scaling, independent deployment) — not technical problems. A 5-person startup with a monolith is more productive than a 5-person startup with 10 microservices, because:

  • One deployment instead of ten
  • One codebase to search and refactor
  • No network calls between services (local method calls are 1000x faster)
  • Simple transactions instead of Saga orchestration
  • Simple debugging instead of distributed tracing setup
The problem with monoliths is not the architecture itself — it is unstructured monoliths with circular dependencies, shared mutable state, and no clear boundaries. We avoid this by organizing our monolith as a modular monolith: each bounded context (auth, product, cart, order, payment) has clear interfaces, its own data access, and no cross-domain dependencies. When the time comes to extract a service, it is a matter of copying a package into a new project and adding a REST API on top — not a rewrite.
