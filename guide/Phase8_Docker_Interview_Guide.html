1. Why Docker?

Docker packages the application and its dependencies into a lightweight, portable container that runs identically across development, staging, and production environments.

Benefits for This Project

  • Consistency — the app runs the same on every machine
  • Dependency isolation — PostgreSQL, Redis, and the app each run in their own container
  • Simplified setup — one docker-compose up starts the entire stack
  • CI/CD ready — containers are the standard deployment artifact for pipelines

2. Dockerfile — Multi-Stage Build

We use a multi-stage build to keep the final image small:

  • Stage 1 (build): Maven + JDK 17 compiles the JAR
  • Stage 2 (runtime): JRE 17 runs the JAR — no Maven or source code included
FROM maven:3.9.9-eclipse-temurin-17-alpine AS build
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline -B
COPY src ./src
RUN mvn package -DskipTests -B

FROM eclipse-temurin:17-jre-alpine
WORKDIR /app
COPY --from=build /app/target/*.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]

Key Decisions

  • mvn dependency:go-offline — downloads dependencies before copying source, so source changes don't re-download deps (layer caching)
  • -DskipTests — tests run in CI, not during image build
  • eclipse-temurin — official, well-maintained OpenJDK builds
  • -alpine — minimal base image (~120 MB vs ~400 MB for full JDK)

3. Docker Compose — Full Stack Orchestration

services:
  postgres:
    image: postgres:15-alpine
    container_name: ecommerce-postgres
    environment:
      POSTGRES_DB: ecommerce
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: postgres
    ports:
      - "5432:5432"
    volumes:
      - postgres-data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 5s
      timeout: 3s
      retries: 5

  redis:
    image: redis:7-alpine
    container_name: ecommerce-redis
    ports:
      - "6379:6379"
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 3s
      retries: 5

  app:
    build: .
    container_name: ecommerce-app
    ports:
      - "8080:8080"
    environment:
      SPRING_DATASOURCE_URL: jdbc:postgresql://postgres:5432/ecommerce
      SPRING_DATASOURCE_USERNAME: postgres
      SPRING_DATASOURCE_PASSWORD: postgres
      SPRING_REDIS_HOST: redis
      SPRING_REDIS_PORT: 6379
      JWT_SECRET: my-super-secret-key...
      JWT_ACCESS_TOKEN_EXPIRATION: 900000
      JWT_REFRESH_TOKEN_EXPIRATION: 604800000
      RAZORPAY_KEY_ID: rzp_test_xxxxxxxxxxxx
      RAZORPAY_KEY_SECRET: your_razorpay_secret
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy

volumes:
  postgres-data:

How Spring Boot Env Var Override Works

Spring Boot automatically maps environment variables to properties using relaxed binding:

Environment VariableSpring Property
SPRING_DATASOURCE_URLspring.datasource.url
SPRING_DATASOURCE_USERNAMEspring.datasource.username
SPRING_REDIS_HOSTspring.redis.host
JWT_SECRETjwt.secret
RAZORPAY_KEY_IDrazorpay.key-id

This lets us keep application.yml unchanged — the Docker environment overrides localhost references to service names (postgres, redis).

Healthchecks & depends_on

The app container waits until both PostgreSQL and Redis pass their health checks before starting. Without this, the app would crash-loop trying to connect to unavailable services.

5. Running the Stack

# Build and start all services
docker compose up --build -d

# View logs
docker compose logs -f app

# Stop everything
docker compose down

# Remove volumes too (resets database)
docker compose down -v

6. Interview Questions & Answers

Q: Why multi-stage builds?
Multi-stage builds produce smaller, more secure runtime images. The build stage has Maven + JDK (hundreds of MB), but the runtime stage only contains the JRE and the JAR. No build tools, source code, or intermediate artifacts leak into the final image.
Q: Why mvn dependency:go-offline before copying source?
Docker builds cache layers. If we copy pom.xml first and run go-offline, the dependency layer only changes when the POM changes. Source code changes re-trigger only the compile + package steps, not dependency downloads. This dramatically speeds up rebuilds.
Q: How does the app connect to PostgreSQL and Redis in Docker?
Docker Compose creates an internal network. Services reach each other by container name (postgres, redis). Environment variables in the app service override the localhost values in application.yml, pointing to the container names instead.
Q: What's the difference between depends_on with and without condition?
Basic depends_on only waits for the container process to start, not for the service to be ready. With condition: service_healthy, it waits until the healthcheck passes (e.g., PostgreSQL accepting queries), preventing the app from crashing on startup.
Q: Why use a named volume for PostgreSQL data?
Containers are ephemeral — removing them deletes all data. A named volume (postgres-data) persists database files on the host, surviving container restarts and rebuilds. Without it, every docker compose down would wipe the database.
Q: How would you deploy this to production?
Options include (a) running docker compose on a VPS with environment variables provided via a .env file, (b) using Kubernetes with Helm charts for orchestration, or (c) using a container platform like AWS ECS/EKS, GCP Cloud Run, or Azure Container Apps. A CI/CD pipeline builds the image, pushes to a registry, and deploys the new image.
Q: How do you handle secrets in Docker?
For development, environment variables in docker-compose.yml suffice. For production, use Docker secrets (swarm mode), a .env file (gitignored), or external secret managers like HashiCorp Vault, AWS Secrets Manager, or sealed secrets in Kubernetes.
Q: What's the difference between an image and a container?
An image is a read-only template with the application code, runtime, and dependencies. A container is a running instance of an image. You can start, stop, restart, and delete containers without affecting the image. Similar to how a class is a blueprint and an object is an instance.
Q: What is layer caching and why does it matter?
Each Dockerfile instruction creates a layer. Docker caches layers so unchanged steps reuse the cache. We structure the build so that rarely-changing steps (dependency downloads) come before frequently-changing steps (source code). This way, changing one Java file doesn't re-download all Maven dependencies.
