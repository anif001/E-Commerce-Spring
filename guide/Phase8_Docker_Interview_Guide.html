1. Docker Basics

Image vs Container

An image is a read-only, immutable template containing the application code, runtime, libraries, and configuration. A container is a runnable instance of that image — an isolated process with its own filesystem, network, and process space. The relationship mirrors classes to objects in OOP: one image can spawn many containers, each with its own writable layer.

Dockerfile vs Docker Compose

A Dockerfile is a recipe for building a single image — it defines the base OS, dependencies, build steps, and runtime command. Docker Compose orchestrates multi-container applications: it declares services, networks, volumes, environment variables, and startup ordering in a single docker-compose.yml file. Compose is declarative infrastructure-as-code for local development and staging environments.

Why Docker for Development and Production

  • Environment parity: Eliminates "it works on my machine" — the same image runs identically on a developer's laptop, CI runner, and production server.
  • Dependency isolation: Each service (PostgreSQL 15, Redis 7, Java 17) runs in its own container without conflicts. No need to install Postgres or Redis on the host machine.
  • Fast onboarding: A new developer runs docker compose up and the entire stack — database, cache, and app — is ready in minutes.
  • CI/CD standardization: Containers are the universal deployment artifact. Build once, deploy to any container runtime (Docker Engine, Kubernetes, ECS, Nomad).
  • Immutable infrastructure: Updates are deployed by replacing the entire container, not patching a running system. Rollbacks are instant — just redeploy the previous image tag.

2. Dockerfile — Multi-Stage Build

Full Dockerfile

FROM maven:3.9.9-eclipse-temurin-17-alpine AS build
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline -B
COPY src ./src
RUN mvn package -DskipTests -B

FROM eclipse-temurin:17-jre-alpine
WORKDIR /app
COPY --from=build /app/target/*.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]

Line-by-Line Explanation

Stage 1: Build

FROM maven:3.9.9-eclipse-temurin-17-alpine AS build

  • FROM initializes the build stage from a base image. Here we use maven:3.9.9-eclipse-temurin-17-alpine, which bundles Maven 3.9.9, Eclipse Temurin JDK 17, and Alpine Linux. The AS build names this stage so the runtime stage can copy artifacts from it.

WORKDIR /app

  • WORKDIR sets the working directory for all subsequent instructions. If it does not exist, Docker creates it. All relative paths (COPY, RUN) resolve from this directory. This is a best practice — avoids scattering files in the root filesystem and makes the build self-documenting.

COPY pom.xml .

  • COPY copies files from the build context (the directory passed to docker build) into the image filesystem. We copy only pom.xml first — this is a deliberate layer caching optimization (see below).

RUN mvn dependency:go-offline -B

  • RUN executes a command during the build and commits the result as a new layer. mvn dependency:go-offline -B downloads all Maven dependencies (from pom.xml) and stores them in the local Maven repository inside the image. The -B flag runs Maven in batch mode (no progress bars, cleaner logs).
  • Layer caching optimization: Docker caches each layer. If pom.xml has not changed between builds, Docker reuses the cached layer from COPY pom.xml and the entire RUN mvn dependency:go-offline layer. This means dependency downloads happen only when the POM changes. If we had copied src first, even a single Java file change would invalidate the cache and force a full re-download of all dependencies — a 5-10 minute penalty on every rebuild.

COPY src ./src

  • COPY copies the source code into the image. This layer changes whenever any source file changes, which is expected — only the compile and package steps will re-run.

RUN mvn package -DskipTests -B

  • Compiles the source and packages it into a fat JAR. -DskipTests skips test execution (tests run in CI, not during the image build — this keeps builds fast and avoids flaky test failures blocking deployments). The resulting JAR is written to /app/target/.

Stage 2: Runtime

FROM eclipse-temurin:17-jre-alpine

  • A new, clean stage starting from a minimal JRE image. This stage has no Maven, no JDK, no source code — only what we explicitly copy in. This is the core benefit of multi-stage builds.
  • Why eclipse-temurin: Eclipse Temurin (formerly AdoptOpenJDK) is the official OpenJDK build from the Eclipse Foundation. It is TCK-tested (Java Specification compliant), free for all uses, and has a strong security patch cadence. It is the recommended JDK distribution in the Java community.
  • Why Alpine: Alpine Linux is a security-oriented, lightweight Linux distribution based on musl libc and BusyBox. The JRE Alpine image is approximately 120 MB compared to ~300 MB for a full Ubuntu-based JRE image. Minimal size means faster pulls, less disk usage, and a smaller attack surface.

WORKDIR /app

  • Sets the runtime working directory.

COPY --from=build /app/target/*.jar app.jar

  • Copies the JAR from the build stage into this runtime stage. The --from=build flag references the named stage. Only the final JAR is copied — no intermediate artifacts, no Maven cache, no source code.

EXPOSE 8080

  • EXPOSE is documentation — it informs Docker that the container listens on port 8080 at runtime. It does not publish the port; -p 8080:8080 in docker run or docker-compose.yml does that. Some container runtimes and orchestrators use EXPOSE for automatic port mapping.

ENTRYPOINT ["java", "-jar", "app.jar"]

  • ENTRYPOINT configures the container's main process. The exec form (["java", "-jar", "app.jar"]) is preferred over the shell form (java -jar app.jar) because it runs the process directly as PID 1, receives Unix signals (SIGTERM), and avoids shell string splitting issues.

Why Multi-Stage?

  • Image size reduction: The build stage (Maven + JDK + dependencies) is approximately 600 MB. The runtime stage (JRE + JAR) is approximately 180 MB. A 70% reduction means faster registry pushes, quicker deployment pulls, and less storage cost.
  • Security: No build tools (Maven, compiler, debug symbols) or source code exist in the runtime image. If an attacker gains access to a running container, they cannot recompile code, run Maven plugins, or access source files.
  • Single Dockerfile: No need for separate build and runtime Dockerfiles or external build scripts. Everything is self-contained and versioned together.
Pro tip: You can verify the size savings with docker history <image> to see each layer's size, or docker image inspect for the total.

3. .dockerignore

The .dockerignore file excludes files and directories from the Docker build context. A smaller build context means faster builds (especially over slow network connections) and ensures no unintended files leak into the image.

target/
.git/
.gitignore
*.md
.idea/
*.iml
.settings/
.project
.classpath
node_modules/
  • target/ — Maven build output. Including it would cause dirty builds and balloon the context size with compiled classes and JARs that the Dockerfile re-creates anyway.
  • .git/ — The entire git history (often hundreds of MB). Not needed at build time. Including it is a common source of slow builds.
  • .gitignore and *.md — Project metadata and documentation. Unnecessary in the image.
  • .idea/, *.iml, .settings/, .project, .classpath — IDE configuration files (IntelliJ IDEA, Eclipse). Every developer's IDE settings are different and irrelevant to the build.
  • node_modules/ — If present, this can be huge. Excluded because the backend Java build does not need it.
Key insight: Even though COPY src ./src in the Dockerfile only copies src/, the entire build context is sent to the Docker daemon before any instruction runs. .dockerignore filters at the client side before the context is tarred and sent. This makes builds faster and prevents Docker cache invalidation caused by irrelevant file changes.

4. Docker Compose — Full Stack Orchestration

Full docker-compose.yml

services:

  postgres:
    image: postgres:15-alpine
    container_name: ecommerce-postgres
    environment:
      POSTGRES_DB: ecommerce
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: postgres
    ports:
      - "5432:5432"
    volumes:
      - postgres-data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 5s
      timeout: 3s
      retries: 5

  redis:
    image: redis:7-alpine
    container_name: ecommerce-redis
    ports:
      - "6379:6379"
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 3s
      retries: 5

  app:
    build: .
    container_name: ecommerce-app
    ports:
      - "8080:8080"
    environment:
      SPRING_DATASOURCE_URL: jdbc:postgresql://postgres:5432/ecommerce
      SPRING_DATASOURCE_USERNAME: postgres
      SPRING_DATASOURCE_PASSWORD: postgres
      SPRING_REDIS_HOST: redis
      SPRING_REDIS_PORT: 6379
      JWT_SECRET: my-super-secret-key-that-is-at-least-32-chars-long-for-hs256
      JWT_ACCESS_TOKEN_EXPIRATION: 900000
      JWT_REFRESH_TOKEN_EXPIRATION: 604800000
      RAZORPAY_KEY_ID: rzp_test_xxxxxxxxxxxx
      RAZORPAY_KEY_SECRET: your_razorpay_secret
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy

volumes:
  postgres-data:

Service-by-Service Breakdown

postgres

  • image: postgres:15-alpine — Uses the official PostgreSQL 15 image on Alpine Linux. Lightweight and well-maintained by the Docker community.
  • environment — Sets the database name, user, and password. These are consumed by the PostgreSQL Docker entrypoint script, which creates the database and user on first startup.
  • ports: "5432:5432" — Maps host port 5432 to container port 5432. This allows connecting from the host using tools like pgAdmin, psql, or IntelliJ database viewer. Without this, Postgres would still be accessible internally to other containers but not from the host.
  • volumes: - postgres-data:/var/lib/postgresql/data — A named volume mounted to the Postgres data directory. This ensures database files persist across container restarts and rebuilds. Named volumes are managed by Docker and stored under /var/lib/docker/volumes/ (Linux) or the Docker VM (macOS/Windows).
  • healthcheck — Runs pg_isready -U postgres every 5 seconds. The check passes when PostgreSQL accepts connections. This is critical for the depends_on condition (see below).

redis

  • image: redis:7-alpine — Official Redis 7 image on Alpine.
  • ports: "6379:6379" — Exposes Redis on the host for debugging or external tools.
  • healthcheck — Runs redis-cli ping. Redis replies with PONG when healthy.

app

  • build: . — Builds the image from the Dockerfile in the project root. Docker Compose will run docker build automatically when you run docker compose up --build.
  • container_name: ecommerce-app — Gives the container a fixed name instead of a randomly generated one. Useful for log identification and docker exec commands.
  • environment — All Spring Boot properties and application secrets are injected as environment variables. Spring Boot's relaxed binding maps these to application.yml properties (see section 5).
  • depends_on with condition: service_healthy — Docker Compose will not start the app until the postgres and redis healthchecks return success. This prevents a crash-loop where the app starts, tries to connect to unavailable databases, fails, and restarts repeatedly.

volumes

postgres-data is a named volume declared at the top level. Named volumes survive docker compose down but are removed with docker compose down -v. This gives explicit control over data lifecycle.

Why Healthchecks Matter

Without healthchecks, depends_on only ensures the target container's process is running — not that the service inside is ready. PostgreSQL may be starting, running recovery, or accepting connections only after a few seconds. The app container would start immediately, fail to connect, and exit. Docker Compose would restart it (depending on restart policy), creating a crash-loop. A healthcheck with depends_on: condition: service_healthy solves this cleanly.

Docker Networking — How Services Communicate

Docker Compose creates a default bridge network for the project. Every service is reachable by its service name (which acts as a DNS hostname). The app connects to PostgreSQL using jdbc:postgresql://postgres:5432/ecommerce — postgres resolves to the container's internal IP. The ports directive maps internal ports to the host, but inter-service communication uses the internal network and the service name, never the host mapping.

  • Internal communication: postgres:5432 — works within the Docker network.
  • Host access: localhost:5432 — works from the host machine because of ports: "5432:5432".
  • No port conflict: You can run multiple Compose projects simultaneously; each gets its own network. Port conflicts only matter for host-mapped ports.
Pro tip: You can inspect the default network with docker network ls and docker network inspect <name>. Compose names it <project>_default.

5. How Spring Boot Integrates with Docker

Spring Boot's relaxed binding (also called "relaxed property resolution") automatically maps environment variables to application.yml properties. The mapping follows a convention: uppercase names, underscores instead of dots, and dashes/camelCase are all normalized.

SPRING_DATASOURCE_URL=jdbc:postgresql://postgres:5432/ecommerce in the Docker environment overrides spring.datasource.url in application.yml without modifying the file. This is a clean separation of code and environment — the same JAR can run against local Postgres (localhost:5432) or containerized Postgres (postgres:5432) depending on which environment variables are set.

Environment variables take priority over application.yml, which takes priority over application.properties. Spring Boot's property resolution order is:

  1. Command line arguments (--spring.datasource.url=...)
  2. Environment variables (SPRING_DATASOURCE_URL)
  3. application-{profile}.yml
  4. application.yml
  5. Defaults

The YAML property razorpay.key-id maps to RAZORPAY_KEY_ID (underscores map to dots and dashes). Always use uppercase with underscores for environment variables for maximum compatibility.

6. Docker Networking in Depth

Default Network Created by Docker Compose

When you run docker compose up, Compose automatically creates a bridge network named <project-dir>_default. All services defined in the Compose file are attached to this network. Containers on the same bridge network can communicate freely; containers not on the network cannot reach them.

Service Discovery by Container Name

Docker's embedded DNS server resolves service names to container IPs. When the app container connects to jdbc:postgresql://postgres:5432/ecommerce, the hostname postgres is resolved to the Postgres container's internal IP (e.g., 172.18.0.2). This is transparent — no need for service registries, hardcoded IPs, or host file modifications.

Port Mapping: Host vs Internal Ports

The ports directive maps HOST:CONTAINER. For example, "5432:5432" means port 5432 on the host is forwarded to port 5432 inside the container. Key points:

  • Internal ports (the container side) are used for inter-service communication. Containers talk to each other on the internal network using the service name and the container port (e.g., postgres:5432).
  • External (host) ports allow connections from outside Docker — your local machine, a load balancer, or another service not on the Docker network.
  • You can remap ports if needed: "5433:5432" would expose Postgres on host port 5433 while the internal port remains 5432. Internal communication is unaffected.

DNS Resolution Details

Docker Compose configures /etc/resolv.conf in each container to point to the built-in DNS resolver at 127.0.0.11. This resolver knows the service names defined in the Compose file. It does not require a separate DNS service or Consul — everything is automatic.

Debugging tip: Run docker compose exec app ping postgres to verify connectivity. Run docker compose exec app cat /etc/resolv.conf to see the DNS configuration. Use docker compose exec app nslookup postgres (if nslookup is available) to see the resolved IP.

7. Running the Stack

CommandDescriptionWhen to Use
docker compose up -d Creates and starts all services in detached (background) mode. Builds images if they do not exist. Normal startup after initial build or when images are up to date.
docker compose build Builds (or rebuilds) the app image without starting containers. Does not affect running services. After changing the Dockerfile or source code when you want to rebuild before starting.
docker compose up --build -d Forces a rebuild of all images before starting. Combines build and up -d in one step. Best for development — ensures the image is fresh every time you start.
docker compose logs -f Streams logs from all services. Append a service name (e.g., docker compose logs -f app) to filter. Debugging startup issues or monitoring real-time output.
docker compose down Stops and removes containers, networks, and any tmpfs volumes. Does not remove named volumes by default. Clean shutdown. Data persists in named volumes.
docker compose down -v Same as down but also removes named volumes (including postgres-data). Full reset — wipes the database. Use sparingly, ideally in development only.
docker compose ps Lists all containers managed by the Compose file with their status and ports. Quick health check for the entire stack.
docker compose exec app sh Opens an interactive shell inside the app container. Useful for debugging. Inspecting files, running ad-hoc commands, or verifying environment variables with env.
Important: Use docker compose (with a space) — the hyphenated form docker-compose is the legacy Python version. Modern Docker Engine includes Compose as a plugin.

8. Production Considerations

Secrets Management

The docker-compose.yml currently hardcodes secrets (JWT secret, Razorpay keys) in plaintext. This is acceptable for local development but never for production.

  • .env file: Store sensitive values in a .env file (added to .gitignore) and reference them with ${VARIABLE} syntax in docker-compose.yml. Example: JWT_SECRET: ${JWT_SECRET}
  • Docker Secrets (Swarm): For Docker Swarm deployments, use secrets with files mounted at /run/secrets/. The application reads them at runtime.
  • External secret stores: HashiCorp Vault, AWS Secrets Manager, or Azure Key Vault. The application fetches secrets on startup via SDK calls, keeping them out of Compose files entirely.
  • Kubernetes: Use Secrets objects (base64-encoded) or external secrets operators (e.g., External Secrets Operator).

Resource Limits

Without limits, a single container can consume all host memory and starve other containers. In production, add resource constraints:

services:
  app:
    deploy:
      resources:
        limits:
          memory: 512M
          cpus: '1.0'
        reservations:
          memory: 256M
          cpus: '0.5'

For docker compose (not swarm), use the --compatibility flag or specify limits per-service with the v3 format.

Logging Driver

The default json-file logging driver can grow unbounded. Configure log rotation or use a centralized logging driver:

services:
  app:
    logging:
      driver: "json-file"
      options:
        max-size: "10m"
        max-file: "3"

For production, consider awslogs (CloudWatch), gcplogs, or fluentd / syslog for centralized log aggregation.

Image Tagging Strategy

Never tag production images as latest. Use a deterministic, immutable tag strategy:

  • Git SHA: registry.example.com/app:abc123 — traceable to the exact commit.
  • Semantic version: registry.example.com/app:1.4.2 — for release branches.
  • CI build number: registry.example.com/app:build-1423 — sequential, easy to identify.
  • Combined: registry.example.com/app:v1.4.2-abc123 — both human-readable and traceable.

Set the tag in your CI/CD pipeline and pass it to docker build -t. Never hardcode a tag in the Compose file for production.

Additional Production Patterns

  • Healthcheck endpoints: Add a Spring Boot Actuator /actuator/health endpoint and use it as the container healthcheck instead of relying on process-level checks.
  • Graceful shutdown: Spring Boot handles SIGTERM gracefully, but set spring.lifecycle.timeout-per-shutdown-phase: "30s" to allow in-flight requests to complete.
  • Security scanning: Scan images with Trivy, Snyk, or Docker Scout before pushing to a registry.
  • Read-only root filesystem: Where possible, set read_only: true on containers that do not need to write to the filesystem.
  • Non-root user: Add a USER instruction in the Dockerfile to run the application as a non-root user, reducing the blast radius of container escape vulnerabilities.

9. Interview Q&A

Q1: What are the benefits of multi-stage builds, and when would you avoid them?

Benefits: Smaller final images (build tools excluded), improved security (no compilers or debug symbols in runtime), and a single Dockerfile that documents the full build pipeline. Image size drops from ~600 MB to ~180 MB in this project.

When to avoid: Simple applications with no build step (e.g., a Python script that runs directly from the base image) do not benefit from multi-stage builds. Also, if your build tools are already minimal (e.g., a Go binary compiled with go build produces a static binary — a single FROM scratch or FROM alpine with just the binary is already optimal).

Q2: Explain layer caching. Why do we run mvn dependency:go-offline before copying source code?

Layer caching: Each Dockerfile instruction produces a layer. Docker caches layers and reuses them on subsequent builds if the instruction and its context (files, command text) have not changed. This is the primary mechanism for fast incremental builds.

By copying pom.xml first and running dependency:go-offline, we create a layer containing all Maven dependencies. This layer is cached and reused as long as pom.xml does not change. If we copied src first, any source file change would invalidate the cache for everything after it — including the dependency download, which takes 5-10 minutes. The ordering makes frequent source changes cheap (only compile + package re-run) and infrequent dependency changes expensive (full download), which is the correct trade-off.

Q3: Why use Alpine-based images? What are the trade-offs?

Pros: Minimal size (~5 MB base, ~120 MB with JRE), smaller attack surface (fewer installed packages), fast pulls.

Cons: Uses musl libc instead of glibc. Some native libraries (e.g., JDBC drivers with native components, certain JNI libraries) may have compatibility issues. DNS resolution and networking behave slightly differently. Build toolchains may require additional dependencies. For pure Java applications (no native code), Alpine is safe and recommended.

Q4: What is the difference between COPY and ADD in a Dockerfile?

COPY copies files/directories from the build context into the image. It is simple, predictable, and the preferred choice for all file copying.

ADD does everything COPY does, plus two additional features: (1) automatically extracts local .tar archives into the image, and (2) supports remote URLs as sources (fetches the file during build). These extras make ADD opaque — the build behavior depends on the source file type. Docker's documentation recommends using COPY unless you specifically need tar extraction or URL fetching. Using ADD for plain files can cause unexpected extraction behavior if someone adds a tar archive to the build context.

Q5: What is the difference between depends_on with and without condition: service_healthy?

Basic depends_on (depends_on: - postgres) only waits for the PostgreSQL container to be running — i.e., the Docker engine has started the process. It does not wait for PostgreSQL to be ready to accept connections. The app container could start, attempt to connect, fail, and restart (often in a crash-loop).

With condition: service_healthy, Compose waits until the healthcheck command (pg_isready) returns a zero exit code — meaning the database is actively accepting connections. This is a proper readiness check, not just a liveness check. The app starts only when its dependencies are genuinely ready.

Q6: Compare bridge network mode vs host network mode. Which do you use and why?

Bridge (default): Containers run in an isolated network namespace with their own IP. Port mapping (-p) exposes specific ports to the host. Containers communicate via the bridge network. This is the standard for development and production — it provides network isolation, prevents port conflicts, and works with Docker Compose service discovery.

Host: The container uses the host's network stack directly — no network isolation, no separate IP. Port mapping is ignored; the container listens on the host's IP. This offers better performance (no NAT overhead) and is sometimes used for high-throughput applications. However, it breaks Docker Compose networking (service names do not resolve), prevents running multiple container instances on the same port, and is a security concern (the container has full access to the host's network interfaces).

Verdict: Use bridge (default) for almost everything. Reserve host mode for specialized use cases like network performance benchmarks or applications that need to bind to specific host IPs directly.

Q7: What is the difference between a named volume and a bind mount? When would you use each?

Named volume: Docker manages the storage location (/var/lib/docker/volumes/). Created with docker volume create or declared in docker-compose.yml under volumes:. Platform-independent, portable, and the recommended way to persist data in production.

Bind mount: You specify an explicit host path (e.g., ./data:/var/lib/postgresql/data). The container can read and write any file on the host at that path. Useful for development (live-reload source code) but less portable and a security risk in production.

Use cases: Named volumes for database data, cache directories, and any persisted production data. Bind mounts for development-time hot-reloading, configuration injection, or sharing Unix sockets.

In this project, postgres-data is a named volume — correct for database persistence. A bind mount would be appropriate for mounting ./src into a dev container for live reload with tools like spring-boot-devtools.

Q8: How does Spring Boot resolve environment variables from Docker Compose to override application.yml? Explain the mechanics.

Spring Boot implements the Environment abstraction backed by a chain of PropertySource objects. When the application starts, Spring Boot registers multiple property sources in order of priority:

  1. Command-line arguments (highest priority)
  2. JNDI attributes
  3. System properties (-Dkey=value)
  4. OS environment variables
  5. application-{profile}.yml
  6. application.yml
  7. Default values (@Value defaults)

Environment variables are in position 4 — higher than application.yml. When the app reads spring.datasource.url, Spring Boot searches property sources in order and finds SPRING_DATASOURCE_URL from the OS environment before reaching application.yml.

The relaxed binding mechanism normalizes the variable name: SPRING_DATASOURCE_URL is converted to spring.datasource.url by splitting on underscores and lowercasing. This supports multiple naming conventions: spring.datasource.url, spring.datasource.URL, SPRING_DATASOURCE_URL, and spring_datasource_url all resolve to the same property.
