1. What is CI/CD?

Continuous Integration (CI)

CI is the practice of automatically building and testing every code change pushed to the repository. In this project, every push or pull request to main triggers a workflow that compiles the code, runs all unit/integration tests against a real PostgreSQL database, and verifies the Docker image builds successfully. The key benefits are immediate feedback on breakage, early detection of integration conflicts, and a consistent quality gate that prevents broken code from reaching production.

Continuous Deployment (CD)

CD extends CI by automatically deploying the validated artifact to a production or staging environment. Here, after the CI workflow completes successfully on main, a separate CD workflow builds the Docker image, pushes it to GitHub Container Registry (GHCR), and SSHes into the deployment server to pull and restart the application with zero-downtime via docker compose up --force-recreate.

Why CI/CD Matters at Scale

  • Catch bugs early: A failing test on a PR costs minutes to fix; a bug in production costs hours of incident response, rollback, and customer trust.
  • Consistent process: Every push follows the exact same build, test, and deploy pipeline, eliminating "works on my machine" discrepancies and manual steps that cause environment drift.
  • Fast feedback: Developers know within minutes (not days) whether their change is safe, enabling rapid iteration and confident shipping.
  • Audit trail: Every deployment is tied to a specific commit, workflow run, and set of test results, making rollback and root-cause analysis straightforward.
Senior-level insight: CI/CD is not just about automation tools. It is a cultural discipline. The pipeline encodes your team's quality standards, release process, and compliance checks as code. A well-designed pipeline catches human error before it reaches customers and lets developers focus on feature work rather than release mechanics.

2. CI Pipeline (ci.yml)

The CI workflow is defined at .github/workflows/ci.yml. It runs on every push and pull request targeting main and consists of two sequential jobs: test and docker. The full file is reproduced below with line-by-line analysis.

Full ci.yml

name: CI — Build, Test & Docker

on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

env:
  IMAGE_NAME: ecommerce-backend

jobs:
  test:
    name: Build & Test
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:15-alpine
        env:
          POSTGRES_DB: ecommerce
          POSTGRES_USER: postgres
          POSTGRES_PASSWORD: postgres
        options: >-
          --health-cmd "pg_isready -U postgres"
          --health-interval 5s
          --health-timeout 3s
          --health-retries 5
        ports:
          - 5432:5432
    steps:
      - uses: actions/checkout@v4

      - name: Set up JDK 17
        uses: actions/setup-java@v4
        with:
          java-version: '17'
          distribution: 'temurin'
          cache: maven

      - name: Run tests
        run: mvn test -B
        env:
          SPRING_DATASOURCE_URL: jdbc:postgresql://localhost:5432/ecommerce
          SPRING_DATASOURCE_USERNAME: postgres
          SPRING_DATASOURCE_PASSWORD: postgres
          SPRING_JPA_HIBERNATE_DDL_AUTO: update
          SPRING_REDIS_HOST: localhost
          SPRING_REDIS_PORT: 6379
          JWT_SECRET: ${{ secrets.JWT_SECRET || 'ci-test-secret-key-that-is-at-least-32-chars-long' }}
          JWT_ACCESS_TOKEN_EXPIRATION: 900000
          JWT_REFRESH_TOKEN_EXPIRATION: 604800000
          RAZORPAY_KEY_ID: rzp_test_ci
          RAZORPAY_KEY_SECRET: ci_secret

      - name: Upload test reports
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: test-reports
          path: target/surefire-reports/

  docker:
    name: Build Docker Image
    runs-on: ubuntu-latest
    needs: [ test ]
    steps:
      - uses: actions/checkout@v4

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Build image
        uses: docker/build-push-action@v5
        with:
          context: .
          push: false
          tags: ${{ env.IMAGE_NAME }}:latest
          cache-from: type=gha
          cache-to: type=gha,mode=max

Line-by-Line Breakdown

Workflow Name and Triggers

name: CI — Build, Test & Docker — This name appears in the GitHub Actions tab and, critically, is referenced by the CD workflow's workflow_run trigger. The name must match exactly between files.

on: push: branches: [ main ] and pull_request: branches: [ main ] — Two events trigger this workflow: direct pushes to main and pull requests opened against main. The pull_request trigger is essential for branch protection — it runs tests on the merge commit before the PR is merged, giving reviewers confidence that the target branch won't break.

Environment Variables

env: IMAGE_NAME: ecommerce-backend — A workflow-level variable reused by the docker job. Referenced as ${{ env.IMAGE_NAME }}. Using env for values used across jobs avoids duplication and makes the pipeline easier to maintain.

Job 1: test

runs-on: ubuntu-latest — The job executes on a fresh Ubuntu virtual machine provided by GitHub. Each job gets its own isolated runner; state does not persist between jobs (except via artifacts or caches).

services: postgres: — This is a service container. GitHub Actions spins up a separate Docker container alongside the job's primary container. The service container is reachable via localhost from the job steps. It lives for the duration of the job and is automatically torn down when the job completes.

  • image: postgres:15-alpine — The official PostgreSQL 15 image on Alpine Linux (small footprint).
  • env: POSTGRES_DB: ecommerce, POSTGRES_USER: postgres, POSTGRES_PASSWORD: postgres — Environment variables passed to the container to initialize the database.
  • options: >- --health-cmd "pg_isready -U postgres" ... — Health check configuration. GitHub Actions waits until the container reports healthy before running job steps. This prevents race conditions where tests start before the database is accepting connections. --health-interval 5s checks every 5 seconds; --health-retries 5 gives up after 5 failures.
  • ports: - 5432:5432 — Maps the container's PostgreSQL port to the host (runner) on the standard port. The Spring Boot test configuration then points to localhost:5432.
Why a real PostgreSQL container instead of H2? Using the exact same database dialect as production ensures that SQL syntax, data type behavior, constraint enforcement, and migration scripts behave identically in CI and production. H2 has subtle incompatibilities (e.g., boolean handling, sequence generation, JSONB support) that can produce false-positive test passes. For a production system using PostgreSQL-specific features (JSONB columns, array types, ON CONFLICT clauses), real-PostgreSQL testing is non-negotiable.

Steps in the test job

Step 1: actions/checkout@v4 — The official GitHub Action that checks out the repository's source code into the runner's workspace. Without this, subsequent steps have no code to work with.

Step 2: actions/setup-java@v4 — Installs JDK 17 from the Adoptium (Temurin) distribution. The cache: maven parameter tells the action to save and restore the Maven ~/.m2/repository directory using GitHub Actions cache, so subsequent runs don't need to re-download all dependencies from Maven Central. This typically shaves 60-80% off the setup time.

Step 3: Run tests — mvn test -B executes the full Maven test suite in batch mode (non-interactive). The -B flag suppresses progress output, making logs more concise. Environment variables override the application's application.properties or application.yml at runtime:

  • SPRING_DATASOURCE_URL: jdbc:postgresql://localhost:5432/ecommerce — Points to the service container on localhost.
  • SPRING_DATASOURCE_USERNAME/PASSWORD: postgres — Matches the service container credentials.
  • SPRING_JPA_HIBERNATE_DDL_AUTO: update — Allows Hibernate to auto-create tables based on entities, avoiding the need for Flyway/Liquibase in CI.
  • SPRING_REDIS_HOST/PORT — Configured even though Redis is not spun up as a service container here (tests requiring Redis would need an additional service entry).
  • JWT_SECRET: ${{ secrets.JWT_SECRET || 'ci-test-secret...' }} — Uses the GitHub secret if available; falls back to a literal string for local/PR testing where secrets are not exposed.
  • RAZORPAY_KEY_ID/SECRET — Test-only values for the payment gateway integration. These are not real credentials — they are placeholders that allow the test context to load without errors.

Step 4: Upload test reports — actions/upload-artifact@v4 saves the Surefire test report XML files. The if: always() condition ensures this step runs even if tests fail, so developers can download and inspect the failure details. Artifacts are stored per-workflow-run in GitHub and expire after 90 days by default.

Job 2: docker

needs: [ test ] — This keyword creates a hard dependency. The docker job waits for the test job to complete successfully. If tests fail, the docker job is skipped entirely (shown as "skipped" with a grey check in the UI). This prevents wasting runner minutes building a Docker image from broken code.

docker/setup-buildx-action@v3 — Configures Docker Buildx, which provides advanced build features like multi-platform builds, build cache export/import, and better layer caching. Without this step, docker/build-push-action would fall back to the legacy builder.

docker/build-push-action@v5 — The core build step:

  • context: . — The build context is the repository root (where the Dockerfile lives).
  • push: false — The image is built but not pushed to any registry. This job purely validates that the Dockerfile can be built successfully (no missing files, no syntax errors, all layers resolve).
  • tags: ${{ env.IMAGE_NAME }}:latest — Tags the image locally as ecommerce-backend:latest.
  • cache-from: type=gha and cache-to: type=gha,mode=max — Uses GitHub Actions cache as the backend for Docker layer caching. mode=max caches all layers (not just the final image), which makes rebuilds significantly faster when only a few layers change. The cache is keyed by the workflow, so different branches/PRs get separate cache entries.
Why two jobs instead of one? Separating test and docker into two jobs with a needs dependency means tests can run in parallel with other work (if there were more jobs) and the docker job can be reused or side-stepped independently. It also makes the pipeline visualisation clearer in the GitHub Actions UI — you can see at a glance whether the failure was in tests or the build. Practically, if the docker job fails, you know the issue is in the Dockerfile or build context, not in the application code.

Pipeline Flow (CI)

Push/PR to main
       |
       v
  +-------------------+
  |  Job: test        |
  |  - PostgreSQL     |---- If fail -> upload reports -> stop
  |  - JDK 17 + Maven |
  |  - mvn test -B    |
  +--------+----------+
           |
           v (needs: [test])
  +-------------------+
  |  Job: docker      |
  |  - Buildx setup   |---- Build image locally (push: false)

3. CD Pipeline (cd.yml)

The CD workflow is defined at .github/workflows/cd.yml. It is not triggered by push events directly. Instead, it uses the workflow_run trigger to run after the CI workflow completes on main. This two-pipeline architecture is a deliberate security and organisational pattern.

Full cd.yml

name: CD — Deploy to Server

on:
  workflow_run:
    workflows: [ "CI — Build, Test & Docker" ]
    branches: [ main ]
    types:
      - completed

env:
  IMAGE_NAME: ecommerce-backend
  REGISTRY: ghcr.io
  IMAGE_TAG: ${{ github.repository_owner }}/ecommerce-backend:latest

jobs:
  deploy:
    if: ${{ github.event.workflow_run.conclusion == 'success' }}
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Log in to GitHub Container Registry
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Build & push image
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: ${{ env.REGISTRY }}/${{ env.IMAGE_TAG }}
          cache-from: type=gha
          cache-to: type=gha,mode=max

      - name: Deploy via SSH
        uses: appleboy/ssh-action@v1.0.3
        with:
          host: ${{ secrets.DEPLOY_HOST }}
          username: ${{ secrets.DEPLOY_USER }}
          key: ${{ secrets.DEPLOY_SSH_KEY }}
          script: |
            cd ~/ecommerce
            docker compose pull app
            docker compose up -d --force-recreate app
            docker image prune -f

Line-by-Line Breakdown

Trigger: workflow_run

on: workflow_run: workflows: [ "CI — Build, Test & Docker" ] — This workflow does not trigger on pushes or PRs. Instead, it triggers when the named workflow (CI — Build, Test & Docker) completes, but only on the main branch. The types: [ completed ] event fires regardless of whether the triggering workflow succeeded or failed — hence the explicit conclusion check in the job.

This design provides several advantages over a single monolithic workflow:

  • Separation of concerns: CI and CD are independent YAML files. The CD pipeline can be maintained, reviewed, and permissioned separately.
  • Security: The CD workflow requires additional secrets (SSH keys, deploy credentials). By keeping it in a separate workflow triggered only on CI success, you avoid exposing these secrets to pull requests from forks.
  • Reusability: The same pattern can be extended to deploy to multiple environments (staging, production) by adding more workflows that listen for the same CI completion event but filter by branch or other criteria.

Conclusion Check

if: ${{ github.event.workflow_run.conclusion == 'success' }} — This is the safety gate. Even though the workflow triggers on completion, we explicitly verify that the CI workflow concluded with success (not failure, cancelled, skipped, or timed_out). The workflow_run event payload contains a conclusion field reflecting the triggering workflow's final status.

Common gotcha: Never use on: workflow_run: ... types: [completed] without also checking conclusion == 'success'. Without that check, the CD pipeline would trigger on CI failures too, potentially deploying broken or untested code.

Environment Variables

REGISTRY: ghcr.io — The target container registry is GitHub Container Registry. ghcr.io is GitHub's own OCI-compliant registry integrated with GitHub Packages.

IMAGE_TAG: ${{ github.repository_owner }}/ecommerce-backend:latest — Push tag format for GHCR is ghcr.io/<owner>/<image>:<tag>. github.repository_owner is a built-in context variable that resolves to the repository owner's GitHub username (lowercase).

Steps

Step 1: Checkout — Standard checkout. The source code is needed because docker/build-push-action reads the Dockerfile from the workspace.

Step 2: docker/login-action@v3 — Authenticates with GHCR using the automatically-provided secrets.GITHUB_TOKEN. The GITHUB_TOKEN is a temporary token that GitHub automatically injects into every workflow run. It has permissions scoped to the current repository and expires when the workflow completes. No manual credential management is needed.

GitHub Container Registry vs Docker Hub: GHCR is deeply integrated with the GitHub ecosystem. Authentication is handled via the built-in GITHUB_TOKEN, so there are no separate registry credentials to rotate or store. GHCR supports fine-grained package permissions at the repository level, has no pull rate limits for authenticated users, and keeps the image alongside the source code in the GitHub UI. Docker Hub, by contrast, requires a separate account/token, has rate limits on free tiers (200 pulls/hour for anonymous, higher for authenticated), and stores images in a separate namespace outside your GitHub repository.

Step 3: docker/build-push-action@v5 — Nearly identical to the CI docker job, but with two critical differences:

  • push: true — The built image is pushed to GHCR, making it available for the deployment server to pull.
  • tags: ${{ env.REGISTRY }}/${{ env.IMAGE_TAG }} — The full push URL, e.g., ghcr.io/myuser/ecommerce-backend:latest.

Step 4: appleboy/ssh-action@v1.0.3 — This popular action SSHes into the deployment server and runs shell commands:

  • cd ~/ecommerce — Navigate to the project directory (where docker-compose.yml lives).
  • docker compose pull app — Pull the latest image from GHCR.
  • docker compose up -d --force-recreate app — Stop the current container, create a new one from the pulled image, and start it in detached mode. The --force-recreate flag ensures a fresh container even if the Compose configuration hasn't changed.
  • docker image prune -f — Remove unused Docker images from the server to free disk space.

The SSH action requires three secrets: DEPLOY_HOST (the server's public IP or hostname), DEPLOY_USER (the SSH username, e.g., ubuntu or deploy), and DEPLOY_SSH_KEY (the SSH private key that matches an authorized public key on the server). These secrets are set in the repository's GitHub Settings and never appear in the workflow file or logs.

4. GitHub Actions Concepts

A senior engineer must understand these primitives to design, debug, and optimise pipelines. Each concept is defined below with its role in this project.

ConceptDefinitionIn This Project
Workflow A YAML file in .github/workflows/ that defines an automated process. It is the top-level unit of CI/CD configuration. ci.yml (build/test/docker) and cd.yml (deploy). Each workflow is independent but linked via workflow_run.
Job A set of steps that execute on the same runner. Jobs in the same workflow run in parallel unless they declare needs dependencies. CI has two jobs (test, docker) running sequentially. CD has one job (deploy).
Step An individual unit of work within a job — either a shell command (run:) or a reusable action (uses:). Examples: uses: actions/checkout@v4, run: mvn test -B.
Action A reusable, versioned package of logic published to the GitHub Marketplace or a public repository. actions/checkout, actions/setup-java, docker/build-push-action, appleboy/ssh-action.
Runner A virtual machine (or self-hosted machine) that executes jobs. Each job gets a fresh runner with a clean filesystem. All jobs use runs-on: ubuntu-latest, GitHub's hosted Ubuntu runner.
Event A trigger that starts a workflow. Events include push, pull_request, workflow_run, schedule, and many more. CI is triggered by push/pull_request to main. CD is triggered by workflow_run completion.
Needs Declares that a job depends on one or more other jobs. The dependent job starts only after all its dependencies succeed. needs: [ test ] on the docker job ensures tests pass first.
Service Container A Docker container that runs alongside the job steps, accessible via localhost. Used to provide databases, caches, or other dependencies without external infrastructure. The PostgreSQL 15 container in the test job provides a real database for integration tests.
Artifact Files persisted after a job completes, downloadable from the GitHub Actions UI. Artifacts live for configurable retention (default 90 days). target/surefire-reports/ uploaded in the test job to make test results inspectable even on failure.
Cache A key-value store for reusing dependency trees and build outputs across workflow runs, reducing execution time. Maven cache via setup-java: cache: maven and Docker layer cache via type=gha.

5. Secrets Management

Secrets are sensitive values that the pipeline needs but must never appear in plaintext in the repository or logs. GitHub provides a secrets store at the repository, environment, and organisation level.

How to Configure Secrets

Navigate to Settings > Secrets and variables > Actions in your GitHub repository. Click "New repository secret" and enter the name and value. Secrets are encrypted with AES-256 before storage and are only decrypted at runtime within the workflow runner. They are masked in all log output (shown as ***).

Secrets Used in This Project

SecretUsed InPurpose
JWT_SECRETCI (test step)JWT token signing key. The CI workflow falls back to a hardcoded test value if the secret is not set (via || operator), but production deployments require a strong, non-guessable secret.
RAZORPAY_KEY_IDCI (test step)Razorpay payment gateway test key. CI uses a hardcoded placeholder (rzp_test_ci) for integration tests that mock the payment provider.
RAZORPAY_KEY_SECRETCI (test step)Razorpay payment gateway test secret. Same pattern as the key.
GITHUB_TOKENCD (login step)Automatically injected by GitHub — not a manual secret. Scoped to the current repository, expires when the workflow ends. Used to authenticate with GHCR.
DEPLOY_HOSTCD (SSH step)The IP address or hostname of the deployment server.
DEPLOY_USERCD (SSH step)The SSH username for the deployment server (e.g., ubuntu, ec2-user, deploy).
DEPLOY_SSH_KEYCD (SSH step)The private SSH key used to authenticate with the server. The corresponding public key must be in the server's ~/.ssh/authorized_keys.

Why Never Hardcode Secrets

  • Exposure in logs: If a secret value appears in a run: command, it will be printed in plaintext in the workflow log.
  • Repository history: A hardcoded secret committed to application.properties or a workflow file lives forever in Git history, even if later removed.
  • Forked PRs: Pull requests from forked repositories cannot access repository secrets (by design). Hardcoded values would bypass this security boundary.
  • Rotation difficulty: Changing a hardcoded secret requires a code change, commit, and deployment. Changing a GitHub secret is a 30-second operation in the Settings UI.
Best practice: Use the ${{ secrets.SECRET_NAME || 'fallback' }} pattern in CI test steps. This allows PRs from forks (which lack secret access) to run tests with a non-sensitive fallback value, while the real secret is used when the workflow runs from the main repository. Never use this pattern in the CD workflow, where secrets are always required and should not have fallbacks.

6. Pipeline Design Patterns

The CI/CD pipeline in this project demonstrates several fundamental design patterns that appear in production-grade pipelines across the industry.

Sequential Job Dependency (Test > Docker)

The needs: [ test ] declaration creates a strict ordering: the Docker build job waits for the test job to succeed. This is a fail-fast pattern. If the tests fail, the pipeline stops immediately and does not waste resources building an image from broken code. The alternative (running test and docker in parallel) would waste compute on every failure and potentially produce a container image from code that failed its tests.

Conditional Deployment (CI Must Pass)

The CD pipeline uses workflow_run with a conclusion == 'success' check. This is a quality gate pattern. No code is deployed unless the entire CI pipeline (build, test, docker build) has completed successfully on the main branch. This prevents scenarios where a developer merges partially broken code that CI flagged but someone forgot to check.

Artifact Sharing (Test Reports)

The test job uploads target/surefire-reports/ as an artifact with if: always(). This always-upload pattern ensures failure analysis data is preserved even when the job fails. Without this, a test failure would stop the job and the detailed Surefire XML reports would be lost when the runner is torn down.

Caching (Maven and Docker Layers)

Two levels of caching are employed:

  • Maven dependency cache — actions/setup-java@v4 with cache: maven automatically saves and restores ~/.m2/repository. The cache key is derived from the pom.xml hash; when dependencies change, the cache is invalidated and rebuilt.
  • Docker layer cache — cache-from: type=gha and cache-to: type=gha,mode=max export and import Docker build cache to GitHub's cache service. This avoids re-downloading base images and re-running unchanged layers on every build.

Both caches are keyed per branch and shareable across workflow runs. Cache hits reduce the test job setup by 60-80% and the Docker build by 40-60%.

Two-Workflow Separation (CI/CD Split)

Using separate YAML files for CI and CD with a workflow_run bridge is a separation of concerns pattern. It allows different permission sets, separate maintenance schedules, and independent failure domains. If the CD workflow has a syntax error, CI still runs for developers. If CI is temporarily failing, you can disable it without affecting the CD configuration.

7. Interview Q&A

Q1: What is the difference between CI and CD, and how do they interact in this project?
CI (Continuous Integration) is the automated process of building and testing every code change. In this project, the CI workflow compiles the application with Maven, runs all tests against a real PostgreSQL service container, and verifies the Docker image builds. CD (Continuous Deployment) takes the validated artifact from CI and deploys it automatically. Here, the CD workflow triggers on CI completion, builds and pushes the Docker image to GHCR, then SSHes into the server to pull and restart the application. CI answers "does this code work?" and CD answers "can we ship it safely?" CI is a prerequisite for CD — you should never deploy code that hasn't passed CI.
Q2: Why does the CI pipeline use a PostgreSQL service container instead of H2 or an embedded database?
A real PostgreSQL container ensures tests run against the exact same database dialect, data types, and constraint behaviour as production. H2 has well-documented incompatibilities with PostgreSQL (JSONB handling, sequence generation, boolean literals, ON CONFLICT clauses, array types). A test that passes on H2 but fails on PostgreSQL is a false positive that erodes confidence in the pipeline. For a production application using PostgreSQL-specific features, the service container guarantees that the test environment mirrors production as closely as possible. Additionally, the health check configuration (pg_isready) avoids race conditions where the test runner connects before the database is ready to accept connections.
Q3: How does the needs keyword work, and what happens if a needed job fails?
needs: [ test ] on the docker job creates a directed dependency. The docker job starts only after the test job completes with a conclusion of success. If the test job fails (any test failure, compilation error, or infrastructure issue), the docker job is automatically skipped — it does not run at all, and the workflow is marked as failed. This is visible in the GitHub Actions UI as a grey checkmark for the skipped job. The needs keyword supports multiple dependencies (needs: [test, lint, security-scan]) and creates a fan-in pattern where the dependent job waits for all its dependencies to succeed. There is no built-in timeout for needs — the job waits indefinitely until its dependencies complete or the workflow times out (6 hours by default).
Q4: What is the difference between GitHub Secrets and environment variables in a workflow?
Secrets (${{ secrets.SECRET_NAME }}) are encrypted values stored in the repository settings. They are AES-256 encrypted at rest, decrypted only at runtime within the runner, and masked in all log output. Environment variables (${{ env.VAR_NAME }} or defined in the workflow's env: block) are plaintext values visible in logs and accessible to all steps. Secrets should be used for credentials, API keys, tokens, and any value that would cause harm if exposed. Environment variables are appropriate for non-sensitive configuration like IMAGE_NAME: ecommerce-backend. A subtle point: secrets are not available to workflows triggered by pull requests from forked repositories — this is a security feature that prevents PRs from exfiltrating secrets. Environment variables, by contrast, are always available. This is why the CI test step uses the pattern ${{ secrets.JWT_SECRET || 'fallback' }} to handle both cases.
Q5: Why does the CD workflow use workflow_run instead of push as its trigger?
The workflow_run trigger decouples the CD pipeline from direct push events and ties it to the completion of the CI workflow. This provides several advantages. First, safety: deployment only happens after the full CI pipeline (build, test, docker build) has completed successfully on main — not merely because a push happened. Second, security: the CD workflow requires access to deployment secrets (SSH keys, host addresses). If CD were triggered by push, those secrets would be accessible within any workflow run, including ones that bypassed the CI quality gate. Third, separation of concerns: the CD workflow can be maintained independently, with different permissions and reviewers. The workflow_run event is the idiomatic way to chain workflows in GitHub Actions when one workflow must complete before the next begins. We also add the explicit if: github.event.workflow_run.conclusion == 'success' guard to handle edge cases where the CI workflow completed but failed.
Q6: Compare GitHub Container Registry (GHCR) with Docker Hub for this pipeline.
GHCR (ghcr.io) is tightly integrated with the GitHub ecosystem. It uses the built-in GITHUB_TOKEN for authentication, eliminating the need to store separate registry credentials as secrets. GHCR supports fine-grained permissions at the repository level, so you can control who can read and write packages using the same GitHub teams and roles. There are no pull rate limits for authenticated users. Docker Hub, while more widely known, requires a separate Docker Hub account and access token stored as a GitHub secret, has pull rate limits (200 pulls/hour for unauthenticated, higher for authenticated but still bounded), and manages packages in a namespace separate from the repository source code. For a project already on GitHub, GHCR reduces operational complexity and credential surface area. The main advantage of Docker Hub is broader ecosystem support and integration with third-party tools that may not support GHCR.
Q7: How does the CD workflow know whether the CI workflow passed? Walk through the mechanism.
The CD workflow uses the workflow_run event trigger with workflows: ["CI — Build, Test & Docker"] and types: [completed]. When GitHub completes any run of the named workflow, it generates a workflow_run event that contains a workflow_run object in the event payload. This object has a conclusion field that reflects the final status of the triggering workflow run (success, failure, cancelled, skipped, timed_out). The CD workflow's deploy job includes the condition if: ${{ github.event.workflow_run.conclusion == 'success' }}, which evaluates the payload field at runtime. If the conclusion is anything other than success, the job is skipped. This is important because the workflow_run event fires on every completion regardless of outcome — without the explicit check, the CD workflow would trigger on CI failures too. The event payload also includes head_branch, head_sha, and other contextual data, but the conclusion check is the primary safety gate.
Q8: What would you improve in this pipeline if you were designing it for a production system with 10+ developers?
Several improvements would be appropriate for a team-scale pipeline. (a) Add a linting/formatting job: Introduce mvn checkstyle:check or a SpotBugs analysis as a fast, parallel job that fails quickly on style or static-analysis issues without waiting for the full test suite. (b) Parallelise the test job: Split tests by module or use Maven's -T option for parallel test execution, reducing CI cycle time. (c) Add a dedicated integration test phase: Use Maven profiles (-Pintegration-tests) or mvn verify with the Failsafe plugin to separate fast unit tests from slower integration tests, running them sequentially or only on certain branches. (d) Introduce a staging deployment: Deploy every successful main build to a staging environment for automated smoke tests before the production deployment. (e) Add security scanning: Integrate trivy or snyk to scan the Docker image for known CVEs during the CI docker job. (f) Implement matrix builds: If the application needs to support multiple JDK versions or platforms, use a build matrix strategy. (g) Notification integration: Send Slack or email notifications on deployment success or failure so the team doesn't need to watch the Actions UI. (h) Version tagging: Replace the latest tag with a semantic version derived from the commit or a Git tag (v1.2.3), enabling rollback to a specific known-good version.
