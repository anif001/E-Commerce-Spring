1. What is CI/CD?

CI (Continuous Integration): Every push to the repository triggers an automated build and test suite. If tests fail, the developer knows immediately.

CD (Continuous Deployment): After CI passes, the application is automatically deployed to a server or cloud platform.

Why it matters for a fresher

  • Shows you understand modern DevOps workflows
  • Automates the tedious parts of testing and deployment
  • Catches integration bugs early
  • Most companies ask about CI/CD in interviews

2. CI Pipeline (.github/workflows/ci.yml)

Triggered on push/PR to main. Two jobs run in sequence:

Job 1: Build & Test

jobs:
  test:
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:15-alpine
        env:
          POSTGRES_DB: ecommerce
          POSTGRES_USER: postgres
          POSTGRES_PASSWORD: postgres
        ports:
          - 5432:5432
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with:
          java-version: '17'
          distribution: 'temurin'
          cache: maven
      - run: mvn test -B
        env:
          SPRING_DATASOURCE_URL: jdbc:postgresql://localhost:5432/ecommerce
          JWT_SECRET: ${{ secrets.JWT_SECRET || 'fallback-secret' }}
          ...

What happens:

  • GitHub spins up a PostgreSQL 15 container as a service
  • JDK 17 is installed with Maven caching
  • mvn test -B runs ALL 135+ tests
  • Failures upload test reports as artifacts for debugging

Job 2: Build Docker Image

  docker:
    runs-on: ubuntu-latest
    needs: [ test ]
    steps:
      - uses: actions/checkout@v4
      - uses: docker/setup-buildx-action@v3
      - uses: docker/build-push-action@v5
        with:
          context: .
          push: false
          tags: ecommerce-backend:latest
          cache-from: type=gha
          cache-to: type=gha,mode=max

The Docker image is built (but not pushed) to verify the Dockerfile works. Uses GitHub Actions cache for faster rebuilds.

3. CD Pipeline (.github/workflows/cd.yml)

on:
  workflow_run:
    workflows: [ "CI — Build, Test & Docker" ]
    branches: [ main ]
    types:
      - completed

jobs:
  deploy:
    if: ${{ github.event.workflow_run.conclusion == 'success' }}
    steps:
      - uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - uses: docker/build-push-action@v5
        with:
          push: true
          tags: ghcr.io/${{ github.repository_owner }}/ecommerce-backend:latest

      - uses: appleboy/ssh-action@v1.0.3
        with:
          host: ${{ secrets.DEPLOY_HOST }}
          username: ${{ secrets.DEPLOY_USER }}
          key: ${{ secrets.DEPLOY_SSH_KEY }}
          script: |
            cd ~/ecommerce
            docker compose pull app
            docker compose up -d --force-recreate app

How it works:

  • Only triggers after the CI workflow succeeds on main
  • Builds the Docker image and pushes it to GitHub Container Registry (ghcr.io)
  • SSHes into your server, pulls the new image, and restarts the app
  • Zero-downtime deployment via --force-recreate
Note: The CD workflow requires server secrets to be configured in your GitHub repository settings (DEPLOY_HOST, DEPLOY_USER, DEPLOY_SSH_KEY).

4. Setting Up Secrets in GitHub

Go to Settings → Secrets and variables → Actions in your repository and add:

Secret NamePurpose
JWT_SECRETJWT signing key (at least 32 chars)
RAZORPAY_KEY_IDRazorpay test key
RAZORPAY_KEY_SECRETRazorpay test secret
DEPLOY_HOSTServer IP (e.g., 123.456.78.90)
DEPLOY_USERSSH username (e.g., ubuntu)
DEPLOY_SSH_KEYSSH private key for server access

Secrets are encrypted and never exposed in logs.

6. Interview Questions & Answers

Q: What is CI and why is it important?
CI (Continuous Integration) means merging code changes frequently and automatically running builds and tests on each merge. It catches bugs early, prevents integration conflicts, and gives immediate feedback to developers. Without CI, bugs might only surface days later during manual testing.
Q: Why use a service container for PostgreSQL in CI instead of an in-memory database?
Using the real PostgreSQL container ensures tests run against the same database dialect and features as production. In-memory databases like H2 have subtle differences that can hide SQL compatibility bugs. For a project using PostgreSQL-specific features, testing against real PostgreSQL is more reliable.
Q: How does needs: [ test ] work in the CI pipeline?
The needs keyword creates a dependency. The Docker job only starts after the test job completes successfully. If tests fail, the Docker job is skipped entirely, preventing wasted time building an image from broken code.
Q: What are GitHub secrets and why use them?
Secrets are encrypted environment variables stored in the repository settings. They are never printed in logs or exposed to forks. We use them for sensitive values like JWT secrets, API keys, and SSH private keys instead of hardcoding them in the workflow file.
Q: How would you trigger a deployment only after CI passes?
The CD workflow uses workflow_run trigger with a filter on workflows: ["CI — Build, Test & Docker"] and types: [completed]. It then checks conclusion == 'success' to ensure CI passed. This guarantees only tested code gets deployed.
Q: What's the difference between GitHub Container Registry (ghcr.io) and Docker Hub?
ghcr.io is GitHub's own registry integrated with GitHub Packages. It uses GITHUB_TOKEN for authentication (no separate credentials needed), supports fine-grained permissions per repository, and has no pull rate limits. Docker Hub is the public default registry but has pull rate limits on free tiers.
Q: What would you add to improve this pipeline?
Possible improvements: (a) parallelize test jobs across different JVM versions, (b) add SonarCloud for code quality checks, (c) add Snyk/Dependabot for vulnerability scanning, (d) add Slack/email notifications on failure, (e) trigger deployment to a staging environment for PR previews, (f) add end-to-end tests with a real browser.
