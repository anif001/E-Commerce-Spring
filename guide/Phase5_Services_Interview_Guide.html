1. Service Layer Role

The service layer is the heart of the application. It sits between controllers (HTTP) and repositories (data) and owns all business logic, transaction boundaries, and security checks.

Architectural Pattern
Controller (HTTP)  --->  Service (Business Logic)  --->  Repository (Data Access)
        |                         |                             |
   @RestController           @Service                     @Repository
   DTO in/out               @Transactional               Spring Data JPA
   JSON serialization       Exception throwing            Entity mapping

Why a separate service layer?

  • Separation of concerns — controllers handle HTTP; services handle logic. Controllers stay thin.
  • Transaction boundaries — @Transactional on service methods ensures atomicity across multiple repository calls.
  • Reusability — services can be called from controllers, scheduled tasks, event listeners, or other services.
  • Testability — business logic is testable without HTTP concerns. Mock repositories, test service methods in isolation.
  • Security enforcement — ownership checks, role validation, and authorization logic live centrally in services.

Service by Service — AuthService

AuthService.java
package com.ecommerce.service;

import com.ecommerce.dto.request.LoginRequest;
import com.ecommerce.dto.request.RegisterRequest;
import com.ecommerce.dto.response.AuthResponse;
import com.ecommerce.enums.Role;
import com.ecommerce.exception.BadRequestException;
import com.ecommerce.exception.UnauthorizedException;
import com.ecommerce.model.User;
import com.ecommerce.repository.UserRepository;
import com.ecommerce.security.JwtTokenProvider;
import lombok.RequiredArgsConstructor;
import org.springframework.security.authentication.AuthenticationManager;
import org.springframework.security.authentication.UsernamePasswordAuthenticationToken;
import org.springframework.security.crypto.password.PasswordEncoder;
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;

@Service
@RequiredArgsConstructor
public class AuthService {

    private final UserRepository userRepository;
    private final PasswordEncoder passwordEncoder;
    private final JwtTokenProvider jwtTokenProvider;
    private final AuthenticationManager authenticationManager;

    @Transactional
    public AuthResponse register(RegisterRequest request) {
        if (userRepository.existsByEmail(request.getEmail())) {
            throw new BadRequestException("Email already registered");
        }

        Role role = Role.CUSTOMER;
        if (request.getRole() != null) {
            try {
                role = Role.valueOf(request.getRole().toUpperCase());
                if (role == Role.ADMIN) {
                    throw new BadRequestException("Admin registration not allowed");
                }
            } catch (IllegalArgumentException e) {
                throw new BadRequestException("Invalid role: " + request.getRole());
            }
        }

        User user = User.builder()
                .name(request.getName())
                .email(request.getEmail())
                .password(passwordEncoder.encode(request.getPassword()))
                .role(role)
                .build();

        user = userRepository.save(user);

        String accessToken = jwtTokenProvider.generateAccessToken(
                user.getId(), user.getEmail(), user.getRole().name());
        String refreshToken = jwtTokenProvider.generateRefreshToken(user.getId());

        return buildAuthResponse(user, accessToken, refreshToken);
    }

    public AuthResponse login(LoginRequest request) {
        try {
            authenticationManager.authenticate(
                    new UsernamePasswordAuthenticationToken(
                            request.getEmail(), request.getPassword()));
        } catch (Exception e) {
            throw new UnauthorizedException("Invalid email or password");
        }

        User user = userRepository.findByEmail(request.getEmail())
                .orElseThrow(() -> new UnauthorizedException("User not found"));

        String accessToken = jwtTokenProvider.generateAccessToken(
                user.getId(), user.getEmail(), user.getRole().name());
        String refreshToken = jwtTokenProvider.generateRefreshToken(user.getId());

        return buildAuthResponse(user, accessToken, refreshToken);
    }

    public AuthResponse refreshToken(String refreshToken) {
        if (!jwtTokenProvider.validateToken(refreshToken)) {
            throw new UnauthorizedException("Invalid or expired refresh token");
        }

        Long userId = jwtTokenProvider.getUserIdFromToken(refreshToken);
        User user = userRepository.findById(userId)
                .orElseThrow(() -> new UnauthorizedException("User not found"));

        String newAccessToken = jwtTokenProvider.generateAccessToken(
                user.getId(), user.getEmail(), user.getRole().name());
        String newRefreshToken = jwtTokenProvider.generateRefreshToken(user.getId());

        return buildAuthResponse(user, newAccessToken, newRefreshToken);
    }

    private AuthResponse buildAuthResponse(User user, String accessToken, String refreshToken) {
        return AuthResponse.builder()
                .accessToken(accessToken)
                .refreshToken(refreshToken)
                .tokenType("Bearer")
                .userId(user.getId())
                .name(user.getName())
                .email(user.getEmail())
                .role(user.getRole().name())
                .expiresIn(jwtTokenProvider.getAccessTokenExpiration())
                .build();
    }
}

Annotations Explained

@Service A Spring stereotype annotation derived from @Component. It marks the class as a Spring-managed bean eligible for component scanning. The difference from @Component is semantic: @Service indicates the class holds business logic, making the architecture explicit. @Component is a generic annotation for any bean. @Service also provides a better target for AOP pointcuts (e.g., within(@org.springframework.stereotype.Service *)).

@RequiredArgsConstructor A Lombok annotation that generates a constructor with one parameter for every final field. Since all dependencies are declared private final, Spring performs constructor injection automatically. Constructor injection is preferred over @Autowired field injection because:

  • Immutability — fields can be final, ensuring they are set once during construction and never reassigned.
  • Testability — no reflection needed; construct objects with new AuthService(mockRepo, mockEncoder, ...).
  • Circular dependency detection — Spring detects cycles at context startup, not at runtime.
  • Null safety — the constructor enforces that all required dependencies are present.
  • No reflection — field injection uses reflection to set private fields, which is slower and can be suppressed by security managers.

@Transactional On the register() method only (method-level). This creates a transaction that wraps the userRepository.save() call. If anything fails before the save completes — such as a uniqueness constraint violation — the transaction rolls back. Since login() and refreshToken() are read-only operations that do not modify data, they have no @Transactional; each repository call will auto-commit in its own transaction by default (JPA's OPEN_IN_VIEW aside).

Key Methods — Reasoning

Password encoding: passwordEncoder.encode(request.getPassword()) hashes the password before persisting. Never store plaintext passwords. Spring Security's PasswordEncoder (typically BCryptPasswordEncoder) uses a salt + adaptive one-way hashing algorithm. An attacker who gains database access still cannot reverse passwords.

AuthenticationManager in login: authenticationManager.authenticate(...) delegates to Spring Security's ProviderManager, which iterates over configured AuthenticationProviders. The DaoAuthenticationProvider fetches the user by email, verifies the password via PasswordEncoder.matches(), and returns a fully populated Authentication token. By using the framework's mechanism, the application gets consistent credential validation, account locking, and event publishing without custom code.

Two DB calls in login: The authentication manager internally loads the user. After successful authentication, a second findByEmail() call fetches the user again to build the response. This is a known pattern where the authenticated principal (a UserDetails object) may not carry all entity fields. The second call is intentional to retrieve the full entity for token generation. In production, this can be optimized by caching or extracting the principal's fields, but the cost is negligible compared to the token generation time.

Refresh token rotation: Every call to refreshToken() issues a completely new access token and a new refresh token. The old refresh token is not invalidated (no blacklist), but since JWT is stateless, the rotation limits the window of exposure: if a refresh token is stolen, it can only be used once before a new one is issued. A production system would add a refresh token blacklist or version counter.

Exceptions Thrown

  • BadRequestException — email already exists, invalid role string, admin self-registration attempt.
  • UnauthorizedException — invalid credentials on login, invalid/expired refresh token, user not found during login/token refresh.

ProductService

ProductService.java
package com.ecommerce.service;

import com.ecommerce.dto.request.ProductRequest;
import com.ecommerce.dto.response.ProductResponse;
import com.ecommerce.exception.BadRequestException;
import com.ecommerce.exception.ResourceNotFoundException;
import com.ecommerce.model.Category;
import com.ecommerce.model.Product;
import com.ecommerce.model.User;
import com.ecommerce.repository.CategoryRepository;
import com.ecommerce.repository.ProductRepository;
import com.ecommerce.repository.UserRepository;
import lombok.RequiredArgsConstructor;
import org.springframework.data.domain.Page;
import org.springframework.data.domain.Pageable;
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;

@Service
@RequiredArgsConstructor
public class ProductService {

    private final ProductRepository productRepository;
    private final CategoryRepository categoryRepository;
    private final UserRepository userRepository;

    @Transactional
    public ProductResponse createProduct(ProductRequest request, Long sellerId) {
        Category category = categoryRepository.findById(request.getCategoryId())
                .orElseThrow(() -> new ResourceNotFoundException(
                        "Category not found: " + request.getCategoryId()));

        User seller = userRepository.findById(sellerId)
                .orElseThrow(() -> new ResourceNotFoundException(
                        "Seller not found: " + sellerId));

        Product product = Product.builder()
                .name(request.getName())
                .description(request.getDescription())
                .price(request.getPrice())
                .stock(request.getStock())
                .imageUrl(request.getImageUrl())
                .category(category)
                .seller(seller)
                .build();

        product = productRepository.save(product);
        return toProductResponse(product);
    }

    public ProductResponse getProductById(Long id) {
        Product product = productRepository.findById(id)
                .orElseThrow(() -> new ResourceNotFoundException(
                        "Product not found: " + id));
        return toProductResponse(product);
    }

    public Page getProductsByCategory(Long categoryId, Pageable pageable) {
        return productRepository.findByCategoryId(categoryId, pageable)
                .map(this::toProductResponse);
    }

    public Page searchProducts(String keyword, Pageable pageable) {
        return productRepository.searchByKeyword(keyword, pageable)
                .map(this::toProductResponse);
    }

    public Page getSellerProducts(Long sellerId, Pageable pageable) {
        return productRepository.findBySellerId(sellerId, pageable)
                .map(this::toProductResponse);
    }

    @Transactional
    public ProductResponse updateProduct(Long id, ProductRequest request, Long userId) {
        Product product = productRepository.findById(id)
                .orElseThrow(() -> new ResourceNotFoundException(
                        "Product not found: " + id));

        if (!product.getSeller().getId().equals(userId)) {
            throw new BadRequestException("Not authorized to update this product");
        }

        if (request.getCategoryId() != null) {
            Category category = categoryRepository.findById(request.getCategoryId())
                    .orElseThrow(() -> new ResourceNotFoundException(
                            "Category not found: " + request.getCategoryId()));
            product.setCategory(category);
        }

        product.setName(request.getName());
        product.setDescription(request.getDescription());
        product.setPrice(request.getPrice());
        product.setStock(request.getStock());
        product.setImageUrl(request.getImageUrl());

        product = productRepository.save(product);
        return toProductResponse(product);
    }

    @Transactional
    public void deleteProduct(Long id, Long userId) {
        Product product = productRepository.findById(id)
                .orElseThrow(() -> new ResourceNotFoundException(
                        "Product not found: " + id));

        if (!product.getSeller().getId().equals(userId)) {
            throw new BadRequestException("Not authorized to delete this product");
        }

        productRepository.delete(product);
    }

    private ProductResponse toProductResponse(Product product) {
        return ProductResponse.builder()
                .id(product.getId())
                .name(product.getName())
                .description(product.getDescription())
                .price(product.getPrice())
                .stock(product.getStock())
                .imageUrl(product.getImageUrl())
                .categoryName(product.getCategory().getName())
                .categoryId(product.getCategory().getId())
                .sellerName(product.getSeller().getName())
                .createdAt(product.getCreatedAt())
                .build();
    }
}

Annotations Explained

@Service @RequiredArgsConstructor Same as AuthService above. Constructor injection for the three repositories.

@Transactional Applied at method level on createProduct, updateProduct, deleteProduct. Read-only methods (getProductById, searchProducts, etc.) do not have the annotation, so each query runs in its own implicit transaction.

Key Methods — Reasoning

Seller ownership check: updateProduct and deleteProduct compare product.getSeller().getId().equals(userId) before allowing the operation. This is multi-tenant protection — a seller cannot modify or delete another seller's product. The check happens after the product is loaded, ensuring the product exists first.

Private toProductResponse(): Encapsulates the mapping from entity to DTO. It is private because it is an internal implementation detail. Keeping it private ensures all DTO conversion logic is centralized and not accidentally bypassed. The method uses the Builder pattern to construct the response, explicitly selecting only the fields needed by the API contract.

Category re-fetch: In updateProduct, the category is only re-fetched if request.getCategoryId() != null. This allows partial updates while still setting the category explicitly via a managed entity reference (avoiding a proxy ID set).

Exceptions Thrown

  • ResourceNotFoundException — category not found, seller not found, product not found.
  • BadRequestException — unauthorized seller attempting update/delete.

CategoryService

CategoryService.java
package com.ecommerce.service;

import com.ecommerce.exception.BadRequestException;
import com.ecommerce.exception.ResourceNotFoundException;
import com.ecommerce.model.Category;
import com.ecommerce.repository.CategoryRepository;
import lombok.RequiredArgsConstructor;
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;

import java.util.List;

@Service
@RequiredArgsConstructor
public class CategoryService {

    private final CategoryRepository categoryRepository;

    @Transactional
    public Category createCategory(String name, String description, Long parentId) {
        if (categoryRepository.existsByName(name)) {
            throw new BadRequestException("Category already exists: " + name);
        }

        Category category = Category.builder()
                .name(name)
                .description(description)
                .build();

        if (parentId != null) {
            Category parent = categoryRepository.findById(parentId)
                    .orElseThrow(() -> new ResourceNotFoundException(
                            "Parent category not found: " + parentId));
            category.setParent(parent);
        }

        return categoryRepository.save(category);
    }

    public Category getCategoryById(Long id) {
        return categoryRepository.findById(id)
                .orElseThrow(() -> new ResourceNotFoundException(
                        "Category not found: " + id));
    }

    public List getAllTopLevelCategories() {
        return categoryRepository.findByParentIsNull();
    }

    public List getSubCategories(Long parentId) {
        return categoryRepository.findByParentId(parentId);
    }

    @Transactional
    public Category updateCategory(Long id, String name, String description) {
        Category category = getCategoryById(id);

        if (name != null && !name.equals(category.getName())) {
            if (categoryRepository.existsByName(name)) {
                throw new BadRequestException("Category name already taken: " + name);
            }
            category.setName(name);
        }
        if (description != null) {
            category.setDescription(description);
        }

        return categoryRepository.save(category);
    }

    @Transactional
    public void deleteCategory(Long id) {
        Category category = getCategoryById(id);
        categoryRepository.delete(category);
    }
}

Annotations Explained

@Transactional On createCategory, updateCategory, deleteCategory. Notably, this class returns the entity Category directly rather than a DTO, because categories are simpler and the same entity is used in the admin API layer — the controller can further wrap it if needed. This is a pragmatic choice; for more complex domains, DTOs are always preferred.

Key Methods — Reasoning

Category hierarchy: The parentId parameter supports a self-referencing parent-child relationship. When parentId is null, the category is top-level. The code explicitly fetches the parent entity to establish the relationship; this ensures referential integrity.

Uniqueness check: existsByName() is called before save to provide a clear error message. Since the database also has a unique constraint on name, this is a pre-validation to avoid a DataIntegrityViolationException.

Exceptions Thrown

  • BadRequestException — duplicate category name.
  • ResourceNotFoundException — category not found by ID (get, update, delete) or parent category not found.

CartService

CartService.java
package com.ecommerce.service;

import com.ecommerce.dto.request.CartItemRequest;
import com.ecommerce.dto.response.CartItemResponse;
import com.ecommerce.dto.response.CartResponse;
import com.ecommerce.exception.BadRequestException;
import com.ecommerce.exception.ResourceNotFoundException;
import com.ecommerce.model.Cart;
import com.ecommerce.model.CartItem;
import com.ecommerce.model.Product;
import com.ecommerce.model.User;
import com.ecommerce.repository.CartRepository;
import com.ecommerce.repository.ProductRepository;
import com.ecommerce.repository.UserRepository;
import lombok.RequiredArgsConstructor;
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;

import java.math.BigDecimal;
import java.util.ArrayList;
import java.util.List;
import java.util.stream.Collectors;

@Service
@RequiredArgsConstructor
public class CartService {

    private final CartRepository cartRepository;
    private final ProductRepository productRepository;
    private final UserRepository userRepository;

    @Transactional
    public CartResponse addItemToCart(Long userId, CartItemRequest request) {
        Cart cart = getOrCreateCart(userId);
        final Cart finalCart = cart;

        Product product = productRepository.findById(request.getProductId())
                .orElseThrow(() -> new ResourceNotFoundException(
                        "Product not found: " + request.getProductId()));

        if (product.getStock() < request.getQuantity()) {
            throw new BadRequestException("Insufficient stock. Available: " + product.getStock());
        }

        finalCart.getItems().stream()
                .filter(item -> item.getProduct().getId().equals(request.getProductId()))
                .findFirst()
                .ifPresentOrElse(
                        existing -> existing.setQuantity(
                                existing.getQuantity() + request.getQuantity()),
                        () -> {
                            CartItem newItem = CartItem.builder()
                                    .cart(finalCart)
                                    .product(product)
                                    .quantity(request.getQuantity())
                                    .build();
                            finalCart.getItems().add(newItem);
                        });

        cartRepository.save(finalCart);
        return toCartResponse(finalCart);
    }

    @Transactional
    public CartResponse updateItemQuantity(Long userId, Long productId, int quantity) {
        Cart cart = getOrCreateCart(userId);

        CartItem item = cart.getItems().stream()
                .filter(ci -> ci.getProduct().getId().equals(productId))
                .findFirst()
                .orElseThrow(() -> new ResourceNotFoundException(
                        "Product not found in cart"));

        if (quantity <= 0) {
            cart.getItems().remove(item);
        } else {
            Product product = item.getProduct();
            if (product.getStock() < quantity) {
                throw new BadRequestException(
                        "Insufficient stock. Available: " + product.getStock());
            }
            item.setQuantity(quantity);
        }

        cart = cartRepository.save(cart);
        return toCartResponse(cart);
    }

    @Transactional
    public CartResponse removeItemFromCart(Long userId, Long productId) {
        Cart cart = getOrCreateCart(userId);

        cart.getItems().removeIf(item ->
                item.getProduct().getId().equals(productId));

        cart = cartRepository.save(cart);
        return toCartResponse(cart);
    }

    public CartResponse getCart(Long userId) {
        Cart cart = getOrCreateCart(userId);
        return toCartResponse(cart);
    }

    @Transactional
    public void clearCart(Long userId) {
        Cart cart = getOrCreateCart(userId);
        cart.getItems().clear();
        cartRepository.save(cart);
    }

    public Cart getOrCreateCart(Long userId) {
        return cartRepository.findWithItemsByUserId(userId)
                .orElseGet(() -> {
                    User user = userRepository.findById(userId)
                            .orElseThrow(() -> new ResourceNotFoundException(
                                    "User not found: " + userId));
                    Cart newCart = Cart.builder().user(user).items(new ArrayList<>()).build();
                    return cartRepository.save(newCart);
                });
    }

    private CartResponse toCartResponse(Cart cart) {
        List itemResponses = cart.getItems().stream()
                .map(item -> CartItemResponse.builder()
                        .cartItemId(item.getId())
                        .productId(item.getProduct().getId())
                        .productName(item.getProduct().getName())
                        .productImage(item.getProduct().getImageUrl())
                        .unitPrice(item.getProduct().getPrice())
                        .quantity(item.getQuantity())
                        .subtotal(item.getSubtotal())
                        .build())
                .collect(Collectors.toList());

        BigDecimal total = itemResponses.stream()
                .map(CartItemResponse::getSubtotal)
                .reduce(BigDecimal.ZERO, BigDecimal::add);

        return CartResponse.builder()
                .cartId(cart.getId())
                .items(itemResponses)
                .totalAmount(total)
                .totalItems(itemResponses.size())
                .build();
    }
}

Annotations Explained

@Transactional On all mutation methods (addItemToCart, updateItemQuantity, removeItemFromCart, clearCart). The getCart and getOrCreateCart methods are not transactional since they perform reads. Changing the cart always requires a transaction because it involves loading the cart, modifying its items collection, and saving — all within one atomic unit.

Key Methods — Reasoning

getOrCreateCart pattern: This is the most important pattern in CartService. It first attempts to findWithItemsByUserId() (which joins items eagerly to avoid lazy loading in a non-transactional context). If no cart exists, it creates a new one with orElseGet(). This is a single-query-then-conditional-save pattern that avoids race conditions better than a separate exists-then-create approach.

Stream + filter for item operations: The code uses cart.getItems().stream().filter(...).findFirst() to locate a specific product in the cart. ifPresentOrElse() handles both cases: if the item exists, increment the quantity; otherwise, create a new CartItem and add it to the collection. This avoids multiple database queries and works on the in-memory collection after loading the cart.

Stock validation on add: Before adding, the service checks that the product's current stock can satisfy the requested quantity. This is a best-effort validation; the definitive stock check happens at order placement time.

Quantity <= 0 removal: In updateItemQuantity, if the quantity is zero or negative, the item is removed from the cart entirely. This is a UX convenience — clients can set quantity to 0 instead of calling a separate remove endpoint.

Exceptions Thrown

  • ResourceNotFoundException — product not found, user not found, product not in cart during update.
  • BadRequestException — insufficient stock.

OrderService

OrderService.java
package com.ecommerce.service;

import com.ecommerce.dto.response.OrderItemResponse;
import com.ecommerce.dto.response.OrderResponse;
import com.ecommerce.enums.OrderStatus;
import com.ecommerce.enums.PaymentStatus;
import com.ecommerce.exception.BadRequestException;
import com.ecommerce.exception.ResourceNotFoundException;
import com.ecommerce.model.*;
import com.ecommerce.repository.CartRepository;
import com.ecommerce.repository.OrderRepository;
import com.ecommerce.repository.ProductRepository;
import lombok.RequiredArgsConstructor;
import org.springframework.data.domain.Page;
import org.springframework.data.domain.Pageable;
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;

import java.math.BigDecimal;
import java.util.List;
import java.util.stream.Collectors;

@Service
@RequiredArgsConstructor
public class OrderService {

    private final OrderRepository orderRepository;
    private final CartRepository cartRepository;
    private final ProductRepository productRepository;
    private final CartService cartService;

    @Transactional
    public OrderResponse placeOrder(Long userId, String shippingAddress) {
        Cart cart = cartService.getOrCreateCart(userId);

        if (cart.getItems().isEmpty()) {
            throw new BadRequestException("Cart is empty");
        }

        List orderItems = cart.getItems().stream()
                .map(cartItem -> {
                    Product product = cartItem.getProduct();

                    if (product.getStock() < cartItem.getQuantity()) {
                        throw new BadRequestException(
                                "Insufficient stock for: " + product.getName());
                    }

                    product.setStock(product.getStock() - cartItem.getQuantity());
                    productRepository.save(product);

                    BigDecimal unitPrice = product.getPrice();
                    BigDecimal subtotal = unitPrice.multiply(
                            BigDecimal.valueOf(cartItem.getQuantity()));

                    return OrderItem.builder()
                            .product(product)
                            .quantity(cartItem.getQuantity())
                            .unitPrice(unitPrice)
                            .subtotal(subtotal)
                            .build();
                })
                .collect(Collectors.toList());

        BigDecimal totalAmount = orderItems.stream()
                .map(OrderItem::getSubtotal)
                .reduce(BigDecimal.ZERO, BigDecimal::add);

        Order order = Order.builder()
                .user(cart.getUser())
                .orderStatus(OrderStatus.PENDING)
                .paymentStatus(PaymentStatus.PENDING)
                .totalAmount(totalAmount)
                .shippingAddress(shippingAddress)
                .build();

        orderItems.forEach(item -> item.setOrder(order));
        order.setOrderItems(orderItems);

        Order savedOrder = orderRepository.save(order);

        cart.getItems().clear();
        cartRepository.save(cart);

        return toOrderResponse(savedOrder);
    }

    public OrderResponse getOrderById(Long orderId, Long userId) {
        Order order = orderRepository.findById(orderId)
                .orElseThrow(() -> new ResourceNotFoundException(
                        "Order not found: " + orderId));

        if (!order.getUser().getId().equals(userId)) {
            throw new BadRequestException("Not authorized to view this order");
        }

        return toOrderResponse(order);
    }

    public Page getUserOrders(Long userId, Pageable pageable) {
        return orderRepository.findByUserId(userId, pageable)
                .map(this::toOrderResponse);
    }

    @Transactional
    public OrderResponse cancelOrder(Long orderId, Long userId) {
        Order order = orderRepository.findById(orderId)
                .orElseThrow(() -> new ResourceNotFoundException(
                        "Order not found: " + orderId));

        if (!order.getUser().getId().equals(userId)) {
            throw new BadRequestException("Not authorized to cancel this order");
        }

        if (order.getOrderStatus() == OrderStatus.SHIPPED ||
            order.getOrderStatus() == OrderStatus.DELIVERED) {
            throw new BadRequestException("Cannot cancel order in " +
                    order.getOrderStatus() + " status");
        }

        order.getOrderItems().forEach(item -> {
            Product product = item.getProduct();
            product.setStock(product.getStock() + item.getQuantity());
            productRepository.save(product);
        });

        order.setOrderStatus(OrderStatus.CANCELLED);
        order = orderRepository.save(order);

        return toOrderResponse(order);
    }

    public Page getAllOrders(Pageable pageable) {
        return orderRepository.findAll(pageable)
                .map(this::toOrderResponse);
    }

    @Transactional
    public OrderResponse updateOrderStatus(Long orderId, OrderStatus status) {
        Order order = orderRepository.findById(orderId)
                .orElseThrow(() -> new ResourceNotFoundException(
                        "Order not found: " + orderId));
        order.setOrderStatus(status);
        order = orderRepository.save(order);
        return toOrderResponse(order);
    }

    private OrderResponse toOrderResponse(Order order) {
        List itemResponses = order.getOrderItems().stream()
                .map(item -> OrderItemResponse.builder()
                        .productId(item.getProduct().getId())
                        .productName(item.getProduct().getName())
                        .quantity(item.getQuantity())
                        .unitPrice(item.getUnitPrice())
                        .subtotal(item.getSubtotal())
                        .build())
                .collect(Collectors.toList());

        return OrderResponse.builder()
                .id(order.getId())
                .orderStatus(order.getOrderStatus())
                .paymentStatus(order.getPaymentStatus())
                .totalAmount(order.getTotalAmount())
                .shippingAddress(order.getShippingAddress())
                .razorpayOrderId(order.getRazorpayOrderId())
                .items(itemResponses)
                .createdAt(order.getCreatedAt())
                .build();
    }
}

Annotations Explained

@Transactional On placeOrder, cancelOrder, updateOrderStatus. The placeOrder method is the most critical transactional boundary in the system. It coordinates inventory deduction, order creation, and cart clearing — all must succeed or fail together.

The @Transactional Proxy Internals

When Spring sees @Transactional on a bean, it creates a JDK dynamic proxy (if the class implements an interface) or a CGLIB proxy (if it does not — the case here, since no interface). The proxy wraps every call to the method:

  1. Before the method: the TransactionInterceptor checks the current transaction context. If none exists, it creates a new one using the configured PlatformTransactionManager (typically DataSourceTransactionManager or JpaTransactionManager).
  2. The transaction is bound to the current thread via TransactionSynchronizationManager.
  3. The method executes. If it returns normally, the interceptor commits the transaction.
  4. If the method throws a RuntimeException (or checked exception configured in rollbackFor), the interceptor rolls back.
  5. The underlying JDBC connection is thread-bound, meaning all repository calls inside the method participate in the same transaction.

If placeOrder throws at step 2 (stock check), step 3 (product save), step 4 (order creation), or step 5 (cart clear), the entire transaction rolls back. Inventory is restored because no save was ever committed. The cart remains populated because the clear was never persisted.

Key Methods — Reasoning

placeOrder atomicity: The flow is: (1) validate cart not empty, (2) for each cart item, check stock, deduct inventory, save product, build OrderItem, (3) calculate total, (4) create Order with PENDING status, (5) link order items to order, (6) save order, (7) clear cart items, (8) save cart. All inside one @Transactional method. If any step fails — e.g., a concurrent purchase exhausts stock between check and save — the JPA optimistic locking or database constraint will throw, causing a full rollback.

Cancel order inventory restoration: cancelOrder reverses the inventory deduction: it increments stock by the cancelled quantity for each item. It also validates that the order is in a cancellable state (PENDING or CONFIRMED — not SHIPPED or DELIVERED).

Authorization check: Both getOrderById and cancelOrder verify order.getUser().getId().equals(userId). This ensures a user can only view or cancel their own orders. The getAllOrders method (intended for admin) has no such check.

Exceptions Thrown

  • BadRequestException — cart empty, insufficient stock, unauthorized order view/cancel, order in non-cancellable state.
  • ResourceNotFoundException — order not found, product not found during stock check.

PaymentService

PaymentService.java
package com.ecommerce.service;

import com.ecommerce.enums.PaymentStatus;
import com.ecommerce.model.Order;
import com.ecommerce.repository.OrderRepository;
import com.razorpay.RazorpayClient;
import com.razorpay.RazorpayException;
import jakarta.annotation.PostConstruct;
import lombok.RequiredArgsConstructor;
import org.json.JSONObject;
import org.springframework.beans.factory.annotation.Value;
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;

import java.math.BigDecimal;

@Service
@RequiredArgsConstructor
public class PaymentService {

    @Value("${razorpay.key-id}")
    private String razorpayKeyId;

    @Value("${razorpay.key-secret}")
    private String razorpayKeySecret;

    private final OrderRepository orderRepository;
    private RazorpayClient razorpayClient;

    @PostConstruct
    public void init() throws RazorpayException {
        this.razorpayClient = new RazorpayClient(razorpayKeyId, razorpayKeySecret);
    }

    @Transactional
    public String createRazorpayOrder(Long orderId) {
        Order order = orderRepository.findById(orderId)
                .orElseThrow(() -> new RuntimeException("Order not found: " + orderId));

        try {
            JSONObject orderRequest = new JSONObject();
            orderRequest.put("amount", order.getTotalAmount()
                    .multiply(BigDecimal.valueOf(100)).intValue());
            orderRequest.put("currency", "INR");
            orderRequest.put("receipt", "order_" + orderId);
            orderRequest.put("payment_capture", 1);

            com.razorpay.Order razorpayOrder = razorpayClient.orders.create(orderRequest);

            order.setRazorpayOrderId(razorpayOrder.get("id"));
            orderRepository.save(order);

            return razorpayOrder.get("id");

        } catch (RazorpayException e) {
            throw new RuntimeException("Failed to create Razorpay order", e);
        }
    }

    @Transactional
    public void verifyPayment(String razorpayOrderId,
                              String razorpayPaymentId,
                              String signature) {
        String expectedSignature = HmacSHA256(
                razorpayOrderId + "|" + razorpayPaymentId,
                razorpayKeySecret);

        if (!expectedSignature.equals(signature)) {
            throw new RuntimeException("Invalid payment signature");
        }

        Order order = orderRepository.findAll().stream()
                .filter(o -> razorpayOrderId.equals(o.getRazorpayOrderId()))
                .findFirst()
                .orElseThrow(() -> new RuntimeException(
                        "Order not found for Razorpay order: " + razorpayOrderId));

        order.setRazorpayPaymentId(razorpayPaymentId);
        order.setPaymentStatus(PaymentStatus.PAID);
        orderRepository.save(order);
    }

    String HmacSHA256(String data, String secret) {
        try {
            javax.crypto.Mac mac = javax.crypto.Mac.getInstance("HmacSHA256");
            javax.crypto.spec.SecretKeySpec secretKeySpec =
                    new javax.crypto.spec.SecretKeySpec(
                            secret.getBytes(java.nio.charset.StandardCharsets.UTF_8),
                            "HmacSHA256");
            mac.init(secretKeySpec);
            byte[] hmacBytes = mac.doFinal(
                    data.getBytes(java.nio.charset.StandardCharsets.UTF_8));
            StringBuilder hexString = new StringBuilder();
            for (byte b : hmacBytes) {
                hexString.append(String.format("%02x", b));
            }
            return hexString.toString();
        } catch (Exception e) {
            throw new RuntimeException("Failed to generate HMAC-SHA256", e);
        }
    }
}

Annotations Explained

@Service @RequiredArgsConstructor Standard stereotype and constructor injection.

@PostConstruct A jakarta.annotation lifecycle callback. After Spring constructs the bean and injects dependencies, it calls init() to create the RazorpayClient instance. This is necessary because the Razorpay client cannot be injected — it must be constructed with runtime configuration properties (razorpayKeyId, razorpayKeySecret). The @Value fields are set before @PostConstruct runs.

@Transactional On createRazorpayOrder and verifyPayment. Both methods update the order entity and must be atomic.

Key Methods — Reasoning

Razorpay integration: The service creates an order on Razorpay's server via REST API. The amount is in paise (multiply by 100). The receipt field links Razorpay's order to the internal order ID. After creation, the Razorpay order ID is saved on the local order entity.

Signature verification: verifyPayment computes HMAC-SHA256 of razorpayOrderId + "|" + razorpayPaymentId using the secret key, then compares it with the provided signature. This cryptographic verification ensures the payment was genuinely processed by Razorpay and not tampered with by a client. The order lookup uses findAll().stream().filter() because Razorpay order ID is not indexed. In production, this should be a dedicated repository query.

Exception handling: RazorpayException (a checked exception) is caught and wrapped in a RuntimeException. Since the @Transactional interceptor rolls back on RuntimeException, the transaction is rolled back if the Razorpay API call fails (e.g., network timeout, invalid key). The caller receives a runtime exception that the GlobalExceptionHandler maps to 500 Internal Server Error.

Exceptions Thrown

  • RuntimeException — order not found, Razorpay API failure, invalid signature, order not found during verification. These are caught by GlobalExceptionHandler.handleGeneral() and return 500.

@Service @RequiredArgsConstructor @Slf4j Standard stereotype plus Lombok logger injection.

@Async Placed on each email method. When a caller invokes emailService.sendOrderConfirmation(...), the call is intercepted by Spring's AOP async proxy. The proxy submits the method execution to a ThreadPoolTaskExecutor and returns immediately (void return type means the caller gets CompletableFuture<Void> behavior in practice, but since the return is void, the caller receives nothing — fire-and-forget). If the email fails (e.g., SMTP server is down), the method catches the exception inside the async thread, logs the error, and continues. The original request is unaffected. This is a best-effort pattern — the system trades delivery guarantees for latency improvement.

Exceptions Thrown

  • None — all exceptions are caught and logged internally. This is intentional: email failures are non-fatal and should not propagate to the caller.

2. Business Logic Deep Dives

Registration Flow

Step by step:
  1. Uniqueness check — userRepository.existsByEmail(request.getEmail()). If true, throw BadRequestException. This is pre-validation; the database unique constraint is the definitive guard.
  2. Role determination — Default is Role.CUSTOMER. If a role is provided, validate it via Role.valueOf(). Admin role is explicitly blocked — only admin-creation endpoints (not shown here) can assign admin.
  3. Password encoding — passwordEncoder.encode(request.getPassword()). Never store raw passwords.
  4. User creation — Build a User entity via the Builder pattern, persist with userRepository.save().
  5. Token generation — Generate access token (short-lived, contains userId, email, role) and refresh token (longer-lived, contains userId only).
  6. Response assembly — buildAuthResponse() constructs the AuthResponse DTO with tokens, user metadata, and expiration.
Transactional boundary: The entire method is @Transactional. If the save fails (e.g., database constraint), the method throws, and no partial state persists.

Login Flow

Step by step:
  1. Authentication — authenticationManager.authenticate() delegates to Spring Security's DaoAuthenticationProvider, which loads UserDetails from the database and verifies the password.
  2. User fetch — After successful authentication, fetch the full User entity from the database by email. This is the "2 DB calls" pattern: auth internally loads UserDetails; the service then loads the entity again. This is acceptable because the second query is fast (indexed by email) and returns a managed entity with all fields.
  3. Token generation — Same as registration.
  4. Response assembly — Same buildAuthResponse().
Why not @Transactional? Login performs no writes. It reads the user, validates credentials via Spring Security, and returns tokens. Transactions are unnecessary for read-only operations.

Place Order Flow (Most Complex)

Step by step:
  1. Get or create cart — cartService.getOrCreateCart(userId) retrieves the cart with items eagerly loaded.
  2. Validate cart — If cart.getItems().isEmpty(), throw BadRequestException.
  3. For each cart item (stream):
    1. Stock check — Compare product.getStock() vs cartItem.getQuantity(). If insufficient, throw BadRequestException with the product name.
    2. Inventory deduction — product.setStock(product.getStock() - cartItem.getQuantity()) and persist.
    3. OrderItem creation — Build an OrderItem with product, quantity, unit price, and subtotal.
  4. Total calculation — Sum all subtotals via stream().map().reduce().
  5. Order creation — Build Order with user, status PENDING, payment PENDING, total, shipping address. Set order items on the order (bidirectional relationship).
  6. Save order — orderRepository.save(order) cascades to order items.
  7. Clear cart — cart.getItems().clear() and cartRepository.save(cart).
  8. Return DTO — toOrderResponse(savedOrder) maps to OrderResponse.
Transactional atomicity: All steps execute in a single transaction. If inventory deduction succeeds but order creation fails (e.g., constraint violation), the inventory changes roll back. If the order saves but cart clearing fails, everything rolls back. The @Transactional proxy uses the JPA transaction manager which issues rollback() on any RuntimeException.

Cart Operations

getOrCreateCart pattern: This is the cornerstone of CartService.
  • cartRepository.findWithItemsByUserId(userId) — tries to find an existing cart with its items eagerly loaded. This is important because without @Transactional on getCart(), lazy loading would fail with LazyInitializationException.
  • .orElseGet(() -> {...}) — if no cart exists, creates a new one: validates the user exists, builds a cart with an empty items list, saves it, and returns it.
addItem flow:
  1. Call getOrCreateCart(userId).
  2. Load product; validate stock.
  3. Stream through existing items, filter by product ID.
  4. ifPresentOrElse: if found, increment quantity. If not, create new CartItem.
  5. Save cart; return DTO.
Stock validation: Two-tier — quick check on add, definitive check on order placement. This prevents customers from adding more items than available, but the real atomic check happens during placeOrder to handle concurrent purchases.

The project uses @RequiredArgsConstructor from Lombok, which generates a constructor accepting one parameter per final field. Spring auto-discovers this constructor and uses it for injection. This is the idiomatic Spring approach since Spring 4.3+ (single-constructor classes are auto-wired without any annotation).

Bean Lifecycle

  1. Instantiation — Spring creates the bean instance via constructor injection. Dependencies are resolved and passed.
  2. Field injection — @Autowired fields (if any) and @Value fields are set via reflection.
  3. Post-initialization — Methods annotated with @PostConstruct are invoked (e.g., PaymentService.init() creates the Razorpay client).
  4. Ready — The bean is fully initialized and registered in the application context, ready to handle calls.
  5. Pre-destroy — @PreDestroy methods (if any) are called when the context shuts down.

Circular Dependencies

Circular dependencies occur when Bean A depends on Bean B, and Bean B depends on Bean A. With constructor injection, Spring throws BeanCurrentlyInCreationException at startup, clearly identifying the cycle. Fixes:

  • Refactor — Extract the conflicting dependency into a third bean.
  • @Lazy — Use @Lazy on one of the constructor parameters to create a proxy that is resolved on first use.
  • Setter/field injection — As a last resort, switch one side to field injection (breaks immutability).

In this project, OrderService depends on CartService. If CartService depended on OrderService, a cycle would exist. Currently, no cycle exists.

Proxy Mechanism for @Transactional and @Async

Spring creates a proxy wrapping the target bean. For @Transactional:

  • A CGLIB proxy is created (subclass of the service class) because no interface is implemented.
  • The proxy intercepts every method call. If the method has @Transactional, the TransactionInterceptor wraps the call in a transaction.
  • Self-invocation problem: If a method in the same class calls another @Transactional method, the proxy is bypassed (the call goes to this, not the proxy), so the inner transaction is not applied. To fix, inject a self-reference or extract the transactional method into a separate service.

For @Async:

  • Same proxying mechanism. The AsyncInterceptor submits the method to the TaskExecutor.
  • Self-invocation problem applies identically: calling this.sendOrderConfirmation(...) from the same class will not execute asynchronously.

AsyncConfig.java
package com.ecommerce.config;

@EnableAsync on a @Configuration class activates Spring's async method execution capability. It registers an AsyncAnnotationBeanPostProcessor that scans for @Async annotations on beans and creates proxies to intercept those methods.

How ThreadPoolTaskExecutor is Configured

When no explicit Executor bean is defined (as in this config), Spring auto-configures a SimpleAsyncTaskExecutor by default. However, the actual runtime behavior depends on the boot autoconfiguration. A production setup would define:

@Bean(name = "taskExecutor")
public Executor taskExecutor() {
    ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
    executor.setCorePoolSize(5);
    executor.setMaxPoolSize(10);
    executor.setQueueCapacity(100);
    executor.setThreadNamePrefix("async-");
    executor.setRejectedExecutionHandler(
            new ThreadPoolExecutor.CallerRunsPolicy());
    executor.initialize();
    return executor;
}
ParameterDescription
corePoolSizeNumber of threads kept alive even when idle (5).
maxPoolSizeMaximum number of threads (10). New threads are created only when the queue is full.
queueCapacityNumber of tasks that can be queued before new threads are spawned (100).
RejectedExecutionHandlerWhat happens when the pool and queue are full. CallerRunsPolicy runs the task in the caller's thread (backpressure). Other options: AbortPolicy (throws), DiscardPolicy (silent discard), DiscardOldestPolicy.

What Happens When Thread Pool is Full

  1. Active threads = corePoolSize (5). New tasks go to the queue.
  2. Queue full (100 tasks). New threads are created up to maxPoolSize (10).
  3. All 10 threads busy and queue full. The RejectedExecutionHandler decides: with CallerRunsPolicy, the thread that called the async method runs the task synchronously, applying backpressure to the caller.

Error Handling for Async Methods

The @Async methods in EmailService catch exceptions internally (try-catch with log). If an unhandled exception escapes an @Async method with a Future return type, it is captured in the Future. For void methods, the exception propagates to Spring's AsyncUncaughtExceptionHandler. To handle these globally:

@Configuration
@EnableAsync
public class AsyncConfig implements AsyncConfigurer {

    @Override
    public Executor getAsyncExecutor() { ... }

    @Override
    public AsyncUncaughtExceptionHandler getAsyncUncaughtExceptionHandler() {
        return (ex, method, params) -> {
            log.error("Async method [{}] threw: {}", method.getName(), ex.getMessage());
            // Notify monitoring, trigger alerts, etc.
        };
    }
}

Since EmailService catches all exceptions internally, this global handler is only a safety net for unhandled exceptions in other async methods.

GlobalExceptionHandler.java
package com.ecommerce.exception;

import org.springframework.http.HttpStatus;
import org.springframework.http.ResponseEntity;
import org.springframework.validation.FieldError;
import org.springframework.web.bind.MethodArgumentNotValidException;
import org.springframework.web.bind.annotation.ExceptionHandler;
import org.springframework.web.bind.annotation.RestControllerAdvice;

import java.time.LocalDateTime;
import java.util.HashMap;
import java.util.Map;

@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(ResourceNotFoundException.class)
    public ResponseEntity<Map<String, Object>> handleNotFound(ResourceNotFoundException ex) {
        return buildResponse(HttpStatus.NOT_FOUND, ex.getMessage());
    }

    @ExceptionHandler(BadRequestException.class)
    public ResponseEntity<Map<String, Object>> handleBadRequest(BadRequestException ex) {
        return buildResponse(HttpStatus.BAD_REQUEST, ex.getMessage());
    }

    @ExceptionHandler(UnauthorizedException.class)
    public ResponseEntity<Map<String, Object>> handleUnauthorized(UnauthorizedException ex) {
        return buildResponse(HttpStatus.UNAUTHORIZED, ex.getMessage());
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<Map<String, Object>> handleValidation(
            MethodArgumentNotValidException ex) {
        Map<String, String> errors = new HashMap<>();
        ex.getBindingResult().getAllErrors().forEach(error -> {
            String field = ((FieldError) error).getField();
            String message = error.getDefaultMessage();
            errors.put(field, message);
        });
        Map<String, Object> response = new HashMap<>();
        response.put("timestamp", LocalDateTime.now());
        response.put("status", HttpStatus.BAD_REQUEST.value());
        response.put("errors", errors);
        return ResponseEntity.badRequest().body(response);
    }

    @ExceptionHandler(Exception.class)
    public ResponseEntity<Map<String, Object>> handleGeneral(Exception ex) {
        return buildResponse(HttpStatus.INTERNAL_SERVER_ERROR,
                "An unexpected error occurred");
    }

    private ResponseEntity<Map<String, Object>> buildResponse(HttpStatus status, String message) {
        Map<String, Object> response = new HashMap<>();
        response.put("timestamp", LocalDateTime.now());
        response.put("status", status.value());
        response.put("error", status.getReasonPhrase());
        response.put("message", message);
        return new ResponseEntity<>(response, status);
    }
}

Exception Propagation

  1. A service method throws e.g., ResourceNotFoundException.
  2. The exception propagates up through the controller layer (the controller does not catch it).
  3. Spring's DispatcherServlet sees the unhandled exception and delegates to the HandlerExceptionResolver chain.
  4. @RestControllerAdvice-annotated GlobalExceptionHandler matches via @ExceptionHandler type.
  5. The matching handler method builds a consistent JSON response map with timestamp, status, and message.
  6. Spring returns the ResponseEntity with the appropriate HTTP status.

6. DTO Mapping Pattern

Why Entities Are Not Exposed Directly

ReasonExplanation
SecurityEntities may contain sensitive fields (e.g., User.password). DTOs select only what the API should expose.
DecouplingEntity changes (field renames, new relationships) do not break API consumers. The DTO is the contract.
PerformanceDTOs fetch only needed fields. Avoids N+1 queries from lazy-loaded relationships during serialization.

Computed fieldsDTO can include fields not present in the entity (e.g., subtotal, totalAmount, categoryName).

The Private toXxxResponse() Pattern

Each service defines a private mapping method (e.g., toProductResponse(), toCartResponse(), toOrderResponse()). This is a simple, explicit mapping approach:

  • No framework dependency — no MapStruct annotation processors, no ModelMapper runtime reflection.
  • Explicit — every field mapping is visible in the source code. No magic.
  • Type-safe — compile-time checked.
  • Private — prevents external callers from bypassing the pattern to get raw entity data.

For large projects, MapStruct is preferred because it generates mapping code at compile time via annotation processing, avoiding runtime reflection while being far less verbose. For this project's scale, manual private methods are clean and sufficient.

7. Interview Q&A

Q1: How does @Transactional work internally in Spring?
Spring creates a CGLIB proxy around the bean. When a @Transactional method is called, the proxy's TransactionInterceptor begins a transaction using the PlatformTransactionManager, binds the connection to the current thread via TransactionSynchronizationManager, invokes the target method, then commits or rolls back based on whether the method threw a RuntimeException. The key internal mechanism: AOP around advice using a proxy pattern, with thread-bound transaction state.
Q2: How does @Async work, and what happens if the thread pool is saturated?
@Async causes Spring to create an AOP proxy that intercepts the method call and submits it to a TaskExecutor. If no custom executor is defined, Spring uses SimpleAsyncTaskExecutor (creates a new thread per invocation — not suitable for production). When the pool is saturated (all core threads busy, queue full, all max threads busy), the RejectedExecutionHandler fires. With CallerRunsPolicy, the calling thread runs the task synchronously, applying backpressure. With AbortPolicy (default for ThreadPoolExecutor), a RejectedExecutionException is thrown.
Q3: Why constructor injection over @Autowired field injection?
Constructor injection provides immutability (fields are final), straightforward testability (no reflection needed to inject mocks), null safety (constructor enforces all dependencies at construction time), and early circular dependency detection (BeanCurrentlyInCreationException at startup). @Autowired field injection requires the class to be instantiated by a container, uses slower reflection, cannot be used with final fields, and can hide null dependencies until runtime.
Q4: What does @RequiredArgsConstructor actually generate?
Lombok generates a public constructor with one parameter for every final field (or non-initialized final field) in the class. For AuthService with four final fields (userRepository, passwordEncoder, jwtTokenProvider, authenticationManager), it generates exactly: public AuthService(UserRepository userRepository, PasswordEncoder passwordEncoder, JwtTokenProvider jwtTokenProvider, AuthenticationManager authenticationManager). Spring auto-discovers this constructor and performs constructor injection without any @Autowired annotation.
Q5: How does Spring create proxies for @Transactional and @Async, and what is the self-invocation problem?
Spring creates a CGLIB proxy (subclass) of the service class. The proxy intercepts method calls and applies transaction/async logic. The self-invocation problem: when method A in the same class calls method B that has @Transactional, the call goes to this.methodB() — the proxy is bypassed. The transaction interceptor never fires, so method B runs without a transaction (or in the caller's existing transaction without propagation isolation). Solutions: extract the transactional method to a separate @Service, inject a self-reference via @Autowired on a field, or use AspectJ compile-time weaving.
Q6: How does AuthenticationManager work in the login flow?
AuthenticationManager.authenticate() is implemented by ProviderManager, which iterates over registered AuthenticationProvider instances. For username/password auth, DaoAuthenticationProvider is used. It calls UserDetailsService.loadUserByUsername() (custom implementation that queries UserRepository by email), then uses PasswordEncoder.matches() to compare the raw password with the stored hash. If they match, it returns a fully populated UsernamePasswordAuthenticationToken with granted authorities. If not, it throws BadCredentialsException, which AuthService catches and wraps as UnauthorizedException.
Q7: What does readOnly=true on @Transactional actually optimize?
readOnly=true is a hint to the JPA provider. Hibernate sets FlushMode to MANUAL (no dirty checking), which avoids the need to snapshot entity state and skip flush at commit. It also allows some databases to optimize the connection (e.g., PostgreSQL might use a read-only transaction snapshot). However, it does NOT prevent writes — if a method annotated with @Transactional(readOnly = true) calls save(), the data will still be persisted. The flag is primarily a performance optimization that tells the ORM to skip the flush cycle after each query.
Q8: How do custom exceptions propagate from the service layer to the HTTP response?
A service method throws a RuntimeException subclass (e.g., ResourceNotFoundException). It propagates through the controller (which does not catch it). Spring's DispatcherServlet catches the exception and invokes HandlerExceptionResolvers. The @RestControllerAdvice-annotated GlobalExceptionHandler matches the exception type via @ExceptionHandler annotations. The matching handler builds a ResponseEntity with a consistent JSON body (timestamp, status, error, message) and the appropriate HTTP status code. The response is serialized and sent to the client.
Q9: Why convert entities to DTOs instead of exposing entities directly?
Three primary reasons: (1) Security — entities may contain sensitive fields like passwords, internal IDs, or audit trails that should never be serialized. (2) Decoupling — entity schema changes (field renames, relationship additions) become breaking API changes if entities are exposed directly; DTOs define the API contract independently. (3) Performance — entities often have lazy-loaded relationships; serializing them triggers N+1 queries or LazyInitializationException; DTOs eagerly map only the required fields.
Q10: How do you handle errors in @Async methods, and what happens if an exception escapes an @Async void method?
For @Async methods returning void, if an unhandled exception escapes, Spring's AsyncUncaughtExceptionHandler is invoked (if implemented via AsyncConfigurer), otherwise the exception is silently lost (the default behavior logs a warning at best). The recommended approach: catch exceptions inside the async method and log them (as EmailService does with try-catch + log.error). For @Async methods returning Future, the exception is captured in the Future object and surfaced when .get() is called. In production, implement AsyncConfigurer.getAsyncUncaughtExceptionHandler() to log errors to monitoring systems and trigger alerts. Note that exceptions in async methods cannot propagate to the original caller because the method runs in a different thread.
