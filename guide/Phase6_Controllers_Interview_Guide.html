1. REST API Design Principles

What is REST?

Representational State Transfer (REST) is an architectural style defined by Roy Fielding. It leverages the HTTP protocol as an application-layer transport, treating every resource as a URL-addressable entity. REST APIs are stateless: each request from a client to a server must contain all information needed to understand and process the request. The server does not store any client context between requests.

Resource-Based URL Structure

Our API base path is /api/, followed by the plural resource name:

/api/auth       -- Authentication (Register, Login, Refresh)
/api/products   -- Product CRUD & search
/api/categories -- Category management
/api/cart       -- Shopping cart operations
/api/orders     -- Order placement & management
/api/payments   -- Razorpay payment integration
/api/health     -- Health check

Key Design Decisions

  • Plural nouns: /api/products not /api/product. Consistent, predictable.
  • No verbs in URLs: actions are expressed via HTTP methods (POST = create, GET = read, PUT = update, DELETE = delete, PATCH = partial update). Exception: /api/auth/register and /api/auth/login are RPC-style because authentication is a verb (action), not a resource.
  • Sub-resource paths: /api/cart/items/{productId} for nested resources.
  • Stateless with JWT: every request carries a Bearer token in the Authorization header. No HttpSession.
  • Versioning: not present in URLs for this project (v1 is implied). Future versions could use a header or /api/v2/.
Statelessness is enforced by SessionCreationPolicy.STATELESS in SecurityConfig — Spring Security never creates an HttpSession. Every request must be authenticated via JWT.

2. Controller-by-Controller Deep Dive

AuthController

@RestController
@RequestMapping("/api/auth")
@RequiredArgsConstructor
@Tag(name = "Authentication", description = "Register, login, and token refresh endpoints")
public class AuthController {

    private final AuthService authService;

    @PostMapping("/register")
    @Operation(summary = "Register a new user",
               description = "Creates a new user account. Default role is CUSTOMER.")
    public ResponseEntity<AuthResponse> register(
            @Valid @RequestBody RegisterRequest request) {
        return ResponseEntity.status(HttpStatus.CREATED)
                .body(authService.register(request));
    }

    @PostMapping("/login")
    @Operation(summary = "Login",
               description = "Authenticates user and returns JWT tokens")
    public ResponseEntity<AuthResponse> login(
            @Valid @RequestBody LoginRequest request) {
        return ResponseEntity.ok(authService.login(request));
    }

    @PostMapping("/refresh")
    @Operation(summary = "Refresh token",
               description = "Generates new access and refresh tokens")
    public ResponseEntity<AuthResponse> refresh(
            @RequestBody Map<String, String> request) {
        String refreshToken = request.get("refreshToken");
        return ResponseEntity.ok(authService.refreshToken(refreshToken));
    }
}

Annotation Breakdown

AnnotationExplanation
@RestController A meta-annotation combining @Controller + @ResponseBody. Every handler method's return value is automatically serialized (HttpMessageConverter) into JSON and written to the HTTP response body. With plain @Controller, methods must be annotated with @ResponseBody individually, otherwise Spring MVC treats the return value as a View name (Thymeleaf/JSP) and throws TemplateResolutionException.
@RequestMapping("/api/auth") Class-level URL prefix. All endpoints in this controller are relative to /api/auth. Keeps mappings DRY and makes URL structure explicit.
@RequiredArgsConstructor Lombok constructor injection. Generates a constructor for all final fields. Spring injects the AuthService bean via this constructor (no need for @Autowired).
@Tag Swagger/OpenAPI grouping annotation. Groups endpoints under "Authentication" in Swagger UI.
@PostMapping Maps HTTP POST to the method. POST is used for create operations and non-idempotent actions. POST to /register creates a user; POST to /login is an action (not idempotent — each call issues a new token).
@Valid Triggers validation on the @RequestBody parameter. If validation fails, Spring throws MethodArgumentNotValidException (handled by GlobalExceptionHandler). Without @Valid, constraint annotations on DTO fields are ignored.
@RequestBody Deserializes the HTTP request body (JSON) into the annotated Java object using HttpMessageConverter (typically MappingJackson2HttpMessageConverter). Cannot be used with GET requests (GET has no body).
@Operation Swagger method-level annotation. Provides endpoint summary and description for generated OpenAPI docs.
ResponseEntity Allows full control over the HTTP response: status code, headers, and body. ResponseEntity.status(HttpStatus.CREATED) sets 201; ResponseEntity.ok() sets 200.

ProductController

@RestController
@RequestMapping("/api/products")
@RequiredArgsConstructor
@Tag(name = "Products", description = "Product CRUD and search endpoints")
public class ProductController {

    private final ProductService productService;

    @GetMapping
    @Operation(summary = "Search products",
               description = "Search products by keyword with pagination")
    public ResponseEntity<Page<ProductResponse>> searchProducts(
            @RequestParam(required = false) String q,
            @PageableDefault(size = 20, sort = "createdAt",
                             direction = Sort.Direction.DESC) Pageable pageable) {
        if (q != null && !q.isBlank()) {
            return ResponseEntity.ok(productService.searchProducts(q, pageable));
        }
        return ResponseEntity.ok(Page.empty());
    }

    @GetMapping("/{id}")
    @Operation(summary = "Get product by ID")
    public ResponseEntity<ProductResponse> getProduct(@PathVariable Long id) {
        return ResponseEntity.ok(productService.getProductById(id));
    }

    @GetMapping("/category/{categoryId}")
    @Operation(summary = "Get products by category")
    public ResponseEntity<Page<ProductResponse>> getByCategory(
            @PathVariable Long categoryId,
            @PageableDefault(size = 20) Pageable pageable) {
        return ResponseEntity.ok(
                productService.getProductsByCategory(categoryId, pageable));
    }

    @PostMapping
    @PreAuthorize("hasRole('SELLER')")
    @Operation(summary = "Create product",
               description = "Only SELLER can create products")
    public ResponseEntity<ProductResponse> createProduct(
            @Valid @RequestBody ProductRequest request,
            @AuthenticationPrincipal UserPrincipal principal) {
        return ResponseEntity.status(HttpStatus.CREATED)
                .body(productService.createProduct(request, principal.getUserId()));
    }

    @PutMapping("/{id}")
    @PreAuthorize("hasAnyRole('SELLER', 'ADMIN')")
    @Operation(summary = "Update product")
    public ResponseEntity<ProductResponse> updateProduct(
            @PathVariable Long id,
            @Valid @RequestBody ProductRequest request,
            @AuthenticationPrincipal UserPrincipal principal) {
        return ResponseEntity.ok(
                productService.updateProduct(id, request, principal.getUserId()));
    }

    @DeleteMapping("/{id}")
    @PreAuthorize("hasAnyRole('SELLER', 'ADMIN')")
    @Operation(summary = "Delete product")
    public ResponseEntity<Void> deleteProduct(
            @PathVariable Long id,
            @AuthenticationPrincipal UserPrincipal principal) {
        productService.deleteProduct(id, principal.getUserId());
        return ResponseEntity.noContent().build();
    }
}

Annotation Breakdown

AnnotationExplanation
@GetMapping Maps HTTP GET. Used for read-only, idempotent, cacheable operations. Should not have side effects. Multiple overloads demonstrated: no path (/api/products), with /{id}, with /category/{categoryId}.
@RequestParam(required = false) Binds a query parameter (?q=searchterm). required = false makes it optional — if absent, the value is null. defaultValue can also be specified. Compare to @PathVariable which extracts from the URI path template.
@PathVariable Long id Extracts a value from the URL template /{id}. By default required = true — if the path variable is missing, Spring returns 404. The value is automatically converted to Long via Spring's type conversion system (Converter<String, Long>).
@PageableDefault Spring Data web support. Injects a Pageable object from query parameters (?page=0&size=20&sort=createdAt,desc). Provides sensible defaults: size=20, sort by createdAt DESC.
@PreAuthorize("hasRole('SELLER')") Spring Security method-level authorization. Only users with the SELLER role can execute this method. Powered by Spring AOP — MethodSecurityInterceptor wraps the bean, checks the SecurityContextHolder before invocation. If authorization fails, AccessDeniedException is thrown (handled by GlobalExceptionHandler).
@PreAuthorize("hasAnyRole('SELLER', 'ADMIN')") Allows both SELLER and ADMIN roles. hasAnyRole() accepts a varargs of role strings. The ROLE_ prefix is automatically prepended by the expression handler.
@AuthenticationPrincipal UserPrincipal Extracts the currently authenticated user's principal object from SecurityContextHolder.getContext().getAuthentication().getPrincipal(). The UserPrincipal class extends Spring Security's User (which implements UserDetails). This annotation uses the AuthenticationPrincipalArgumentResolver to resolve the parameter. Provides access to getUserId() and getUserEmail().
@PutMapping("/{id}") Maps HTTP PUT. PUT is idempotent — calling it multiple times with the same body produces the same state. Used for full replacement of a resource. Contrast with PATCH (partial update, not idempotent).
@DeleteMapping("/{id}") Maps HTTP DELETE. Idempotent — deleting a resource that doesn't exist still returns 204 (or 404 depending on design, we return 204 after service call succeeds).
ResponseEntity<Void> + noContent() 204 No Content. No response body. Used for successful DELETE operations where there is nothing to return.

CategoryController

@RestController
@RequestMapping("/api/categories")
@RequiredArgsConstructor
@Tag(name = "Categories", description = "Category management endpoints")
public class CategoryController {

    private final CategoryService categoryService;

    @GetMapping
    @Operation(summary = "Get all top-level categories")
    public ResponseEntity<List<Category>> getAllCategories() {
        return ResponseEntity.ok(categoryService.getAllTopLevelCategories());
    }

    @GetMapping("/{id}")
    @Operation(summary = "Get category by ID")
    public ResponseEntity<Category> getCategory(@PathVariable Long id) {
        return ResponseEntity.ok(categoryService.getCategoryById(id));
    }

    @GetMapping("/{id}/subcategories")
    @Operation(summary = "Get subcategories of a category")
    public ResponseEntity<List<Category>> getSubCategories(@PathVariable Long id) {
        return ResponseEntity.ok(categoryService.getSubCategories(id));
    }

    @PostMapping
    @PreAuthorize("hasRole('ADMIN')")
    @Operation(summary = "Create category", description = "Only ADMIN can create categories")
    public ResponseEntity<Category> createCategory(
            @RequestBody Map<String, Object> request) {
        String name = (String) request.get("name");
        String description = (String) request.get("description");
        Long parentId = request.get("parentId") != null
                ? ((Number) request.get("parentId")).longValue() : null;
        return ResponseEntity.status(HttpStatus.CREATED)
                .body(categoryService.createCategory(name, description, parentId));
    }

    @PutMapping("/{id}")
    @PreAuthorize("hasRole('ADMIN')")
    @Operation(summary = "Update category")
    public ResponseEntity<Category> updateCategory(
            @PathVariable Long id,
            @RequestBody Map<String, String> request) {
        return ResponseEntity.ok(categoryService.updateCategory(
                id, request.get("name"), request.get("description")));
    }

    @DeleteMapping("/{id}")
    @PreAuthorize("hasRole('ADMIN')")
    @Operation(summary = "Delete category")
    public ResponseEntity<Void> deleteCategory(@PathVariable Long id) {
        categoryService.deleteCategory(id);
        return ResponseEntity.noContent().build();
    }
}

Key Design Notes

  • No DTO classes for categories — the controller directly uses the Category entity in responses. This is acceptable for simple read-only projections, but generally DTOs are preferred to avoid exposing internal entity structure.
  • Map-based request bodies — category creation uses Map<String, Object> instead of a typed DTO. This means @Valid cannot be applied (no constraint annotations). Validation is delegated to the service layer.
  • Hierarchical structure — /api/categories/{id}/subcategories demonstrates nested resource paths. The subcategory is a dependent resource of a category.

CartController

@RestController
@RequestMapping("/api/cart")
@RequiredArgsConstructor
@Tag(name = "Cart", description = "Shopping cart management endpoints")
public class CartController {

    private final CartService cartService;

    @GetMapping
    @Operation(summary = "Get current user's cart")
    public ResponseEntity<CartResponse> getCart(
            @AuthenticationPrincipal UserPrincipal principal) {
        return ResponseEntity.ok(cartService.getCart(principal.getUserId()));
    }

    @PostMapping("/items")
    @Operation(summary = "Add item to cart")
    public ResponseEntity<CartResponse> addItem(
            @Valid @RequestBody CartItemRequest request,
            @AuthenticationPrincipal UserPrincipal principal) {
        return ResponseEntity.ok(
                cartService.addItemToCart(principal.getUserId(), request));
    }

    @PutMapping("/items/{productId}")
    @Operation(summary = "Update item quantity in cart")
    public ResponseEntity<CartResponse> updateItem(
            @PathVariable Long productId,
            @RequestParam int quantity,
            @AuthenticationPrincipal UserPrincipal principal) {
        return ResponseEntity.ok(
                cartService.updateItemQuantity(
                    principal.getUserId(), productId, quantity));
    }

    @DeleteMapping("/items/{productId}")
    @Operation(summary = "Remove item from cart")
    public ResponseEntity<CartResponse> removeItem(
            @PathVariable Long productId,
            @AuthenticationPrincipal UserPrincipal principal) {
        return ResponseEntity.ok(
                cartService.removeItemFromCart(principal.getUserId(), productId));
    }

    @DeleteMapping
    @Operation(summary = "Clear entire cart")
    public ResponseEntity<Void> clearCart(
            @AuthenticationPrincipal UserPrincipal principal) {
        cartService.clearCart(principal.getUserId());
        return ResponseEntity.noContent().build();
    }
}

Key Observations

  • No @PreAuthorize — cart endpoints rely on isAuthenticated() from SecurityConfig (.anyRequest().authenticated()). Since all cart URLs require authentication at the filter-chain level, method-level annotations are unnecessary.
  • Implicit user binding — every method extracts the authenticated user via @AuthenticationPrincipal. No user ID is ever passed as a request parameter, preventing horizontal privilege escalation.
  • Sub-resource structure — cart items are at /api/cart/items/{productId}. The product ID in the URL identifies which item to update/remove.

@RestController
@RequestMapping("/api/orders")
@RequiredArgsConstructor
@Tag(name = "Orders", description = "Order management endpoints")
public class OrderController {

    private final OrderService orderService;

    }

    @GetMapping
    @Operation(summary = "Get current user's orders")
    public ResponseEntity<Page<OrderResponse>> getMyOrders(
            @AuthenticationPrincipal UserPrincipal principal,
            @PageableDefault(size = 10, sort = "createdAt",
                             direction = Sort.Direction.DESC) Pageable pageable) {
        return ResponseEntity.ok(
                orderService.getUserOrders(principal.getUserId(), pageable));
    }

    @GetMapping("/{id}")
    @Operation(summary = "Get order by ID")
    public ResponseEntity<OrderResponse> getOrder(
            @PathVariable Long id,
            @AuthenticationPrincipal UserPrincipal principal) {
        return ResponseEntity.ok(
                orderService.getOrderById(id, principal.getUserId()));
    }

    @PostMapping("/{id}/cancel")
    @Operation(summary = "Cancel an order")
    public ResponseEntity<OrderResponse> cancelOrder(
            @PathVariable Long id,
            @AuthenticationPrincipal UserPrincipal principal) {
        return ResponseEntity.ok(
                orderService.cancelOrder(id, principal.getUserId()));
    }

    @GetMapping("/admin")
    @PreAuthorize("hasRole('ADMIN')")
    @Operation(summary = "Get all orders (Admin only)")
    public ResponseEntity<Page<OrderResponse>> getAllOrders(
            @PageableDefault(size = 20, sort = "createdAt",
                             direction = Sort.Direction.DESC) Pageable pageable) {
        return ResponseEntity.ok(orderService.getAllOrders(pageable));
    }

    @PatchMapping("/{id}/status")
    @PreAuthorize("hasRole('ADMIN')")
    @Operation(summary = "Update order status (Admin only)")
    public ResponseEntity<OrderResponse> updateStatus(
            @PathVariable Long id,
            @RequestBody Map<String, String> request) {
        OrderStatus status = OrderStatus.valueOf(
                request.get("status").toUpperCase());
        return ResponseEntity.ok(
                orderService.updateOrderStatus(id, status));
    }
}

Key Observations

  • Owner check — getOrder() passes principal.getUserId() to the service, which verifies the order belongs to the requesting user. This is authorization at the service layer, not the controller layer.
  • Action endpoints as POST — /api/orders/{id}/cancel uses POST because cancellation is an action (not a resource). REST purists call these "controller endpoints" or "RPC-style" endpoints.
  • PATCH for partial update — /api/orders/{id}/status uses @PatchMapping. PATCH is semantically correct for partial updates (we're only changing the order status, not replacing the entire order).
  • Admin segregation — admin endpoints are prefixed with /admin to avoid path collisions with /{id}. Spring matches the most specific path first, so /admin won't conflict with /{id}.
  • OrderStatus.valueOf() — converts a string from the request body into an enum, throwing IllegalArgumentException (caught by GlobalExceptionHandler's catch-all @ExceptionHandler(Exception.class)) if invalid.

PaymentController

@RestController
@RequestMapping("/api/payments")
@RequiredArgsConstructor
@Tag(name = "Payments", description = "Razorpay payment integration endpoints")
public class PaymentController {

    private final PaymentService paymentService;

    @PostMapping("/create-order/{orderId}")
    @Operation(summary = "Create Razorpay order for checkout")
    public ResponseEntity<Map<String, String>> createRazorpayOrder(
            @PathVariable Long orderId) {
        String razorpayOrderId = paymentService.createRazorpayOrder(orderId);
        return ResponseEntity.ok(Map.of(
                "razorpayOrderId", razorpayOrderId,
                "status", "created"
        ));
    }

    @PostMapping("/verify")
    @Operation(summary = "Verify Razorpay payment signature")
    public ResponseEntity<Map<String, String>> verifyPayment(
            @RequestBody Map<String, String> paymentData) {
        paymentService.verifyPayment(
                paymentData.get("razorpayOrderId"),
                paymentData.get("razorpayPaymentId"),
                paymentData.get("signature")
        );
        return ResponseEntity.ok(Map.of("status", "verified"));
    }
}

Key Observations

  • RPC-style endpoints — payment actions (create order, verify) are verbs, not resources. Using POST for these RPC-style operations is the RESTful convention ("POST is the catch-all for non-CRUD operations").
  • Map responses — returns Map<String, String> instead of typed DTOs. Simpler for small, ad-hoc response structures. Not recommended for complex responses (loses type safety, documentation).
  • No authentication annotations — relies on SecurityConfig's .anyRequest().authenticated().

@RestController

    @GetMapping("/api/health")
    @Operation(summary = "Health check endpoint")
    public ResponseEntity<Map<String, Object>> health() {

                "status", "UP",
                "timestamp", LocalDateTime.now().toString(),
                "version", "1.0.0"
        ));
    }
}

Key Observations

  • No @RequestMapping — the base path is specified directly on the @GetMapping("/api/health") method. Since there is only one endpoint, class-level mapping was deemed unnecessary.
  • No authentication — /api/health is in the permitAll() list in SecurityConfig. Health checks must be accessible without authentication for load balancers and orchestrators (Kubernetes liveness/readiness probes).
  • No DB dependency — returns a simple map reflecting the app is running. For production, consider DataSourceHealthIndicator or Spring Boot Actuator.

How @Valid Triggers Validation

Validation Flow
  1. Request arrives — JSON body is deserialized by MappingJackson2HttpMessageConverter into the DTO object.
  2. MethodArgumentResolver — Spring MVC's RequestResponseBodyMethodProcessor detects the @RequestBody parameter.
  3. Validation trigger — If the parameter has @Valid (or @Validated), the processor calls the Validator bean (which is a LocalValidatorFactoryBean — a Spring-wrapped Jakarta Bean Validation engine).
  4. Constraint evaluation — The validator iterates over all @NotBlank, @Email, etc. annotations and evaluates them against the field values.
  5. ConstraintViolation collection — Each failed constraint generates a ConstraintViolation with: invalid value, violation message, property path.
  6. Exception thrown — Spring wraps the violations in MethodArgumentNotValidException (a subclass of Exception), which extends BindingResult-aware exceptions.
  7. ExceptionHandler catches — The GlobalExceptionHandler's handleValidation() method extracts field-level errors and returns 400 Bad Request.
What happens without @Valid? The DTO is still deserialized from JSON, but constraint annotations are never evaluated. Invalid data (e.g., null password) would pass through to the service layer, potentially causing cryptic NullPointerException or database constraint violations instead of clean validation errors.

Validation Error Response Format

{
  "timestamp": "2026-06-16T14:30:00.123",
  "status": 400,
  "errors": {
    "email": "Invalid email format",
    "password": "Password must be at least 6 characters"
  }
}

Each field that fails validation appears as a key in the "errors" map with its violation message. This gives the frontend precise, field-level error information.

How Roles Work: ROLE_ Prefix

ROLE_ Prefix Mechanics

In UserPrincipal, the authority is constructed as:

new SimpleGrantedAuthority("ROLE_" + user.getRole().name())

For a user with role CUSTOMER, the granted authority is "ROLE_CUSTOMER".

hasRole('ADMIN') — Spring Security's SecurityExpressionRoot.hasRole(String role) internally calls hasAuthority("ROLE_" + role). So hasRole('ADMIN') checks for "ROLE_ADMIN".

hasAnyRole('SELLER', 'ADMIN') — Similarly, hasAnyRole() prepends ROLE_ to each argument: hasAnyAuthority("ROLE_SELLER", "ROLE_ADMIN").

hasAuthority('ROLE_ADMIN') — If you used hasAuthority() instead, you must include the ROLE_ prefix yourself. hasAuthority() performs an exact match against the granted authority string.

Internal Mechanism of @PreAuthorize

  • AOP Proxy — Spring creates a CGLIB or JDK dynamic proxy around beans containing @PreAuthorize methods.
  • MethodSecurityInterceptor — When an annotated method is called, the interceptor invokes the SecurityExpressionHandler to evaluate the expression.
  • SecurityContextHolder — The expression evaluator reads the current Authentication from SecurityContextHolder.getContext().getAuthentication().
  • Evaluation — hasRole('ADMIN') checks if the authentication's granted authorities contain ROLE_ADMIN.
  • AccessDeniedException — If the expression evaluates to false, AccessDeniedException is thrown. This is caught by GlobalExceptionHandler.handleAccessDenied() which returns 403 Forbidden.

5. HTTP Status Codes

CodeStatusWhen It OccursExample
200 OK Successful GET, PUT, POST (non-create), PATCH responses. The response body contains the resource. GET /api/products/{id} returns product JSON. PUT /api/products/{id} returns updated product.
201 Created Successful resource creation via POST. Response includes Location header conceptually (though not explicitly set here). Body contains the created resource. POST /api/auth/register returns auth tokens. POST /api/products returns created product.
204 No Content Successful DELETE where no response body is needed. Also used for DELETE /api/cart (clear cart). DELETE /api/products/{id} returns no body. DELETE /api/categories/{id}.
400 Bad Request Validation failure (MethodArgumentNotValidException), malformed JSON, missing required body, invalid enum values (OrderStatus.valueOf). POST /api/products with missing name → { "errors": { "name": "Product name is required" } }
401 Unauthorized Missing/invalid JWT token, token expired, wrong credentials at login. The request has not been authenticated. POST /api/auth/login with wrong password. Request with missing Authorization header.
403 Forbidden Valid JWT token but insufficient role. AccessDeniedException is thrown when @PreAuthorize evaluates to false. POST /api/products by a CUSTOMER role user. GET /api/orders/admin by a SELLER.
404 Not Found Resource doesn't exist. ResourceNotFoundException is thrown when a product, category, or order with the given ID is not found. GET /api/products/999 when product 999 doesn't exist. GET /api/orders/999
409 Conflict Duplicate resource creation attempt. Duplicate email during registration, duplicate category name. POST /api/auth/register with existing email. POST /api/categories with duplicate name.
500 Internal Server Error Unhandled exceptions, null pointer exceptions, database connection failures, unexpected errors. Service layer throws RuntimeException. Caught by GlobalExceptionHandler.handleGeneral().
401 vs 403 distinction: 401 means "you are not authenticated" (missing or invalid credentials). 403 means "you are authenticated but not authorized" (insufficient permissions). Spring Security distinguishes these: authentication failures (bad JWT) produce 401; authorization failures (AccessDeniedException) produce 403.

6. Exception Handling & GlobalExceptionHandler

The GlobalExceptionHandler is the centralized error-handling component. All exceptions thrown from controllers propagate to this handler, which maps them to consistent JSON error responses.

Full Code

@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(ResourceNotFoundException.class)
    public ResponseEntity<Map<String, Object>> handleNotFound(
            ResourceNotFoundException ex) {
        return buildResponse(HttpStatus.NOT_FOUND, ex.getMessage());
    }

    @ExceptionHandler(BadRequestException.class)
    public ResponseEntity<Map<String, Object>> handleBadRequest(
            BadRequestException ex) {
        return buildResponse(HttpStatus.BAD_REQUEST, ex.getMessage());
    }

    @ExceptionHandler(UnauthorizedException.class)
    public ResponseEntity<Map<String, Object>> handleUnauthorized(
            UnauthorizedException ex) {
        return buildResponse(HttpStatus.UNAUTHORIZED, ex.getMessage());
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<Map<String, Object>> handleValidation(
            MethodArgumentNotValidException ex) {
        Map<String, String> errors = new HashMap<>();
        ex.getBindingResult().getAllErrors().forEach(error -> {
            String field = ((FieldError) error).getField();
            String message = error.getDefaultMessage();
            errors.put(field, message);
        });
        Map<String, Object> response = new HashMap<>();
        response.put("timestamp", LocalDateTime.now());
        response.put("status", HttpStatus.BAD_REQUEST.value());
        response.put("errors", errors);
        return ResponseEntity.badRequest().body(response);
    }

    @ExceptionHandler(Exception.class)
    public ResponseEntity<Map<String, Object>> handleGeneral(Exception ex) {
        return buildResponse(HttpStatus.INTERNAL_SERVER_ERROR,
                "An unexpected error occurred");
    }

    private ResponseEntity<Map<String, Object>> buildResponse(
            HttpStatus status, String message) {
        Map<String, Object> response = new HashMap<>();
        response.put("timestamp", LocalDateTime.now());
        response.put("status", status.value());
        response.put("error", status.getReasonPhrase());
        response.put("message", message);
        return new ResponseEntity<>(response, status);
    }
}

Annotation Breakdown

AnnotationExplanation
@RestControllerAdvice A combination of @ControllerAdvice + @ResponseBody. @ControllerAdvice is a Spring stereotype that allows a class to intercept exceptions across all @Controller classes. Without @ResponseBody, the handler methods would need method-level @ResponseBody. Using @RestControllerAdvice applies @ResponseBody globally — all return values from handler methods are serialized directly to JSON.
@ExceptionHandler(ResourceNotFoundException.class) Marks a method as an exception handler for the specified exception class (and its subclasses). Spring's DispatcherServlet propagates exceptions to HandlerExceptionResolver beans; ExceptionHandlerExceptionResolver (internal to Spring MVC) dispatches to methods annotated with @ExceptionHandler. The most specific handler wins (based on the exception type hierarchy).

Exception Handler Priority Order

Spring matches the most specific (most derived) exception class first:

  1. ResourceNotFoundException — 404 (most specific)
  2. BadRequestException — 400
  3. UnauthorizedException — 401
  4. MethodArgumentNotValidException — 400 with field errors
  5. Exception.class — 500 catch-all (least specific)

If a ResourceNotFoundException is thrown, Spring checks handler #1 first (match), then stops. The generic Exception.class handler is only reached for exception types not caught by any more specific handler.

Error Response Structure

Standard error response (for 400, 401, 404, 500):

{
  "timestamp": "2026-06-16T14:30:00.123",
  "status": 404,
  "error": "Not Found",
  "message": "Product not found with id: 999"
}

Validation error response (400):

{
  "timestamp": "2026-06-16T14:30:00.123",
  "status": 400,
  "errors": {
    "email": "Invalid email format",
    "password": "Password must be at least 6 characters"
  }
}
Why buildResponse is private: It's a helper method reused by all simple exception handlers to avoid repeating the response structure. The validation handler is separate because its response structure is different (map of field errors instead of a single message).

8. Interview Q&A

1. What is the difference between @RestController and @Controller?
@RestController = @Controller + @ResponseBody. With @Controller, each handler method's return value is interpreted as a view name (for Thymeleaf/JSP resolution) unless the method is also annotated with @ResponseBody. This means a @Controller returning a User object without @ResponseBody would fail with a TemplateResolutionException because Spring would look for a User.ftlh or User.jsp template. @RestController avoids this by applying @ResponseBody on every method — the return value is directly serialized via HttpMessageConverter (JSON by default). For REST APIs, always use @RestController.
2. What is the difference between @RequestBody and @ModelAttribute?
@RequestBody deserializes the entire HTTP request body (usually JSON) into a Java object using HttpMessageConverter. It expects a single deserializable payload. @ModelAttribute binds query parameters, path variables, and form fields to an object using property setters — it's designed for form submissions (application/x-www-form-urlencoded) or query parameter binding. In this codebase, all POST/PUT endpoints use @RequestBody + JSON because we're building a JSON API. @ModelAttribute is more common in Spring MVC applications with form rendering.
3. How does @Valid trigger validation? What happens on failure?
When a handler parameter is annotated with @Valid, Spring MVC's RequestResponseBodyMethodProcessor invokes the configured Validator (a LocalValidatorFactoryBean wrapping the Jakarta Bean Validation engine). The validator evaluates all constraint annotations on the DTO fields. If any constraint fails, a ConstraintViolation is created. These violations are wrapped in MethodArgumentNotValidException, which is thrown before the handler method executes. The GlobalExceptionHandler.handleValidation() method catches this exception and returns a 400 Bad Request with a map of field-level errors. Without @Valid, no validation occurs — constraints are silently ignored.
4. How does @PreAuthorize work internally?
@PreAuthorize relies on Spring AOP. When @EnableMethodSecurity is active, Spring creates a CGLIB proxy (or JDK dynamic proxy for interfaces) around beans containing @PreAuthorize methods. When the method is called, the proxy's MethodSecurityInterceptor intercepts the invocation and uses SecurityExpressionHandler to evaluate the SpEL expression. The expression evaluator reads the current Authentication from SecurityContextHolder. For hasRole('ADMIN'), it checks Authentication.getAuthorities() for "ROLE_ADMIN". If the check fails, AccessDeniedException is thrown. If it passes, the real method is invoked. This all happens transparently — the controller code has no knowledge of the proxy.
5. What is the difference between 401 Unauthorized and 403 Forbidden?
401 Unauthorized: The request lacks valid authentication credentials. The client is not logged in, the JWT is missing, expired, or malformed. The server does not know who the client is. Spring Security's AuthenticationEntryPoint returns this when AuthenticationException occurs (e.g., bad JWT). 403 Forbidden: The request is authenticated (the JWT is valid, the server knows who the user is), but the authenticated user does not have the required role or permission. AccessDeniedException triggers this, handled by AccessDeniedHandler. In this project: a CUSTOMER trying to POST to /api/products gets 403 (valid token, but missing SELLER role). A request with no token at all to the same endpoint gets 401.
6. How does @AuthenticationPrincipal extract the current user?
@AuthenticationPrincipal is resolved by AuthenticationPrincipalArgumentResolver, a Spring MVC argument resolver. It reads from SecurityContextHolder.getContext().getAuthentication().getPrincipal(). The Authentication.getPrincipal() returns an Object (typically UserDetails). The resolver casts it to the parameter type — UserPrincipal. Our UserPrincipal extends Spring Security's User (which implements UserDetails) and adds userId and userEmail fields. The resolver returns this as the method argument. This is cleaner than calling SecurityContextHolder directly in every controller method.
7. What is the difference between @ControllerAdvice and @RestControllerAdvice?
Same relationship as @Controller vs @RestController. @ControllerAdvice is a global interceptor for controllers — it can define @ExceptionHandler, @InitBinder, and @ModelAttribute methods that apply across all controllers. However, handler methods in @ControllerAdvice need @ResponseBody to return JSON. @RestControllerAdvice = @ControllerAdvice + @ResponseBody, meaning all exception handler methods automatically serialize their return values to JSON. For REST APIs, always use @RestControllerAdvice.
8. What determines the priority of @ExceptionHandler methods?
Spring matches exception handlers based on the specificity of the exception type. It uses ExceptionDepthComparator to find the closest matching handler. For example, if ResourceNotFoundException extends RuntimeException extends Exception:
- Throwing ResourceNotFoundException matches handleNotFound() (closest)
- Throwing IllegalArgumentException (a RuntimeException) does NOT match handleNotFound() and falls through to handleGeneral() (the catch-all for Exception.class).
Order of declaration in the class does NOT matter — Spring always picks the most specific match. If there are multiple handlers at the same level (e.g., two handlers for RuntimeException), Spring reports an ambiguity error.
9. How would you design a global exception handler for a microservice?
A well-designed global exception handler should: (1) Use @RestControllerAdvice for automatic JSON serialization. (2) Cover all application-specific exceptions (e.g., ResourceNotFoundException → 404, BadRequestException → 400, DuplicateResourceException → 409, UnauthorizedException → 401). (3) Handle Spring exceptions (MethodArgumentNotValidException → 400 with field errors, HttpMessageNotReadableException → 400 for malformed JSON, AccessDeniedException → 403). (4) Have a catch-all for Exception.class returning 500 without leaking stack traces. (5) Use a consistent response envelope with timestamp, status, error, message, and optionally a path field. (6) Log exceptions appropriately (WARN for 4xx, ERROR for 5xx). (7) Consider a unique error code for each exception type for easier client-side handling.
10. Why use DTOs instead of entities in controller responses?
Returning JPA entities directly from controllers couples the API contract to the persistence model. Problems include: (1) Lazy loading — accessing lazily-loaded fields outside a transaction throws LazyInitializationException. (2) Over-fetching — entities may contain sensitive fields (password hash, internal IDs) or unnecessary data. (3) Circular references — bidirectional relationships cause infinite JSON recursion without @JsonIgnore / @JsonManagedReference. (4) Tight coupling — changing the database schema changes the API contract. DTOs provide a clean separation layer, allowing each endpoint to return exactly the data the client needs, in the shape the client expects. They also enable versioning: you can change the entity structure without breaking the API.
11. How are validation errors structured in the response?
The handleValidation() method in GlobalExceptionHandler extracts field-level errors from MethodArgumentNotValidException.getBindingResult().getAllErrors(). Each ObjectError is cast to FieldError to access .getField() (the field name) and .getDefaultMessage() (the validation message from the annotation). These are collected into a Map<String, String> (errors) and wrapped in the response with timestamp and status 400. The frontend can display field-level error messages directly under each input field. Example response:
{ "timestamp": "...", "status": 400, "errors": { "email": "Invalid email format", "password": "Password must be at least 6 characters" } }
12. What happens when @RequestBody is missing in a POST request?
If the @RequestBody parameter is not annotated with @Valid (or @Validated), no validation occurs. For example, CategoryController.createCategory() uses @RequestBody Map<String, Object> request without @Valid. Any JSON body will be accepted — missing fields will simply be null in the map. This means validation of required fields must happen in the service layer. If the @RequestBody annotation is present but the request body is empty or not valid JSON, Spring throws HttpMessageNotReadableException (400 Bad Request). If the request body is missing entirely (no Content-Type header or empty body), the same exception occurs. If there were no @RequestBody at all (hypothetically), Spring would bind query parameters instead, and the parameter would be null.
